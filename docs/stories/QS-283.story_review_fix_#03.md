## Findings to fix

### [should-fix] Self-contradictory "run the full quality gate" intro line in all three `qs-implement-task.md` mirrors
- File: `.claude/agents/qs-implement-task.md:14`, `.cursor/agents/qs-implement-task.md:15`, `.opencode/agents/qs-implement-task.md:53`
- Severity: should-fix
- Source: qs-review-coderabbit
- Description: This is the exact bug class QS-283 exists to eliminate. Each
  `qs-implement-task.md` mirror still opens with the summary line:
  *"You implement the story under `custom_components/quiet_solar/` and
  `tests/`, run the full quality gate, and open a PR."* That instruction
  directly contradicts the B1/B3 mandate this same PR adds later in the file
  ("**ALWAYS** run the impacted gate before commit/PR. Do **not** run, or
  substitute, the full gate locally"). An agent reading the intro can follow
  the stale full-gate command path — precisely the regression the story is
  meant to close. The contradiction is present in all three harness mirrors
  and must be fixed identically (byte-for-byte parity is asserted by the
  existing `test_b1_*` / `test_b2_b3_*` parametrized tests).
- Proposed fix: Reword the intro summary line in all three mirrors so it names
  the impacted gate as the inner-loop command (and, if the full gate is
  mentioned at all, frame it as CI's job). Suggested wording:
  *"…run the **impacted** quality gate (`--impacted`) before commit/PR, and
  open a PR. The whole-repo gate is CI's job."* Keep the three mirrors
  byte-identical. Consider extending the enforcement test so the
  contradictory phrase ("run the full quality gate, and open a PR") can never
  reappear in an implement-task agent file — e.g. assert the intro references
  `--impacted` and does not instruct running the full gate locally. Also
  re-scan the `.opencode` copy's frontmatter/`description` (line ~5 region)
  and the other mirrors' `description` for the same stale phrasing and align
  them.

### [should-fix] No test asserts the retry-path diagnostic for a non-coverage retry failure
- File: `scripts/qs/quality_gate.py:1248` (behavior) / `tests/test_quality_gate.py` (missing test)
- Severity: should-fix
- Source: qs-review-acceptance-auditor
- Description: The self-heal retry collapses to `return 0 if retry_verdict ==
  _IMPACTED_PASS else 1`, discarding the specific retry verdict. The fix-#01
  resolution was that `_run_impacted_pass` already `_emit`s the per-verdict
  diagnostic before returning, so a non-coverage retry failure
  (`_IMPACTED_TESTS_FAILED` / `_IMPACTED_NO_COVERAGE_XML` /
  `_IMPACTED_DIFF_COVER_TIMEOUT`) still surfaces its message on stderr even
  though the exit code is collapsed. That claim currently rests on inspection
  only — no test exercises it.
- Proposed fix: Add a unit test in the `TestCheckImpactedSelfHeal` group that
  drives a self-heal retry whose second pass returns a non-coverage failure
  verdict (e.g. `_IMPACTED_TESTS_FAILED`) and asserts BOTH that `check_impacted`
  returns exit code 1 AND that the per-verdict diagnostic message is emitted
  (capture stdout/stderr or assert the `_emit` call). No production behavior
  change required unless the diagnostic is found to be missing on the retry
  path — in which case route the retry verdict through the same `_emit` the
  first pass uses before collapsing to the exit code.

### [nice-to-have] Redundant exception tuple `except (FileNotFoundError, OSError)`
- File: `scripts/qs/quality_gate.py` (the `was_incremental` probe, ~line 1437)
- Severity: nice-to-have
- Source: qs-review-blind-hunter
- Description: `FileNotFoundError` is a subclass of `OSError`, so
  `except (FileNotFoundError, OSError)` is redundant; behavior is correct but
  ruff/style may flag the redundant member.
- Proposed fix: Reduce to `except OSError`. Ensure the existing
  `test_testmondata_vanishing_before_stat_is_non_incremental` still passes
  (it should, unchanged).

### [nice-to-have] "No production-coverage delta" claim in setup-task agent docs
- File: `.opencode/agents/qs-implement-setup-task.md:150` (+ `.claude` / `.cursor` mirrors)
- Severity: nice-to-have
- Source: qs-review-coderabbit
- Description: The docs state "Dev-only changes (`scripts/`, `docs/`, agent
  files) carry no production-coverage delta, so `--impacted` just runs the
  impacted tests fast." CodeRabbit notes that `scripts/qs/**` (and `.github/**`)
  edits can alter the quality-gate / CI wiring itself, so framing all dev-only
  edits as coverage-neutral could imply tooling regressions are impossible.
  The claim is technically accurate about *production* (`custom_components/`)
  coverage — and the gate's own tests are testmon-selected under `--impacted` —
  so this is a wording-precision nit, not a behavioral gap.
- Proposed fix: Tighten the wording to make explicit that "no
  production-coverage delta" means no delta to `custom_components/quiet_solar/`
  coverage specifically, while the gate/tooling's own correctness is still
  guarded by its testmon-selected tests under `--impacted` (and the whole-repo
  gate in CI). Apply identically across the mirrors that carry the phrase.
