## Already fixed in the working tree (verify + land)

### [should-fix → resolved] Far-out person forecast no longer leaks a raw newline
- Files: `home_model/constraints.py`, `ha_model/person.py`, `ha_model/car.py`
- The maintainer threaded an `allow_cr` flag through
  `get_readable_date_string(..., allow_cr=True)` and
  `LoadConstraint.get_readable_next_target_date_string(..., allow_cr=True)`.
  When `allow_cr=False`, the far-out date renders as `"%m-%d %H:%M"` (space)
  instead of `"%m-%d\n%H:%M"` (newline).
  - `person.py:get_forecast_readable_string()` now passes `allow_cr=False`, so the
    person-forecast string can never contain a newline (fixes SF2 at the source —
    covers the person branch the previous `_origin_target_single_line` fix missed).
  - `car.py:_origin_target_single_line` now calls `allow_cr=False` instead of the
    fragile `.replace("\n", " ")` (resolves NTH2 — the shared-formatter suggestion).
- Implement-phase tasks to land this:
  1. Commit the three working-tree edits.
  2. **Add the missing test**: a person-branch case with a `predicted_leave_time`
     >~24h out, asserting the `qs_car_charge_origin` string contains no raw `\n`
     (the existing far-out tests cover only Calendar/Manual). Put it in
     `tests/test_car_charge_origin.py`.
  3. Minor cleanups: remove the double space in
     `car.py` `return  ct.get_readable_next_target_date_string(...)`; give
     `get_readable_date_string`'s new param an explicit annotation
     (`allow_cr: bool = True`) to match `get_readable_next_target_date_string`;
     refresh the `_origin_target_single_line` docstring (it still says
     "newline-collapse" — now it delegates via `allow_cr=False`).
  4. Confirm the existing far-out Calendar/Manual tests still pass (the rendered
     string is unchanged: a space either way).
