# Story QS-177: Re-implement OpenCode handover for the new static-agent pipeline

issue: 177
branch: "QS_177"
Status: ready-for-dev

## Story

**As** an OpenCode user of the new static-agent pipeline,
**I want** the handoff between phases to spawn a fresh interactive session
in the same OpenCode instance with the next phase's agent already activated
and a kickoff prompt sent automatically,
**so that** I don't have to manually pick the agent in the UI or type the
initial command.

## Context

The new static-agent pipeline (`scripts/qs/`) replaced the per-task
rendered agents of the legacy pipeline (`scripts/qs_opencode/`). The
OpenCode launcher in the new pipeline (`scripts/qs/launchers/opencode.py`)
is currently a stub that delegates to the legacy pipeline for any real
handoff.

The legacy `scripts/qs_opencode/spawn_session.py` is the canonical
OpenCode handoff: it calls `POST /session` to create a session, then fires
a delayed `POST /instance/reload` to re-scan agent files. Because reload
races against active session runners, it cannot activate the agent on the
new session — the user has to manually pick the agent in the UI.

With static agents (no live re-scan needed), we can drop the reload and
use `POST /session/<id>/prompt_async` with body
`{agent: "<name>", parts: [{type: "text", text: "<kickoff>"}]}` to
activate the agent and send the kickoff prompt in one synchronous flow.

**API endpoints verified against the local OpenCode source at
`/Users/tmenguy/Developer/WildMoka/opencode/`:**

- `POST /session` (`session.create`) — body `{title?: string, ...}`,
  returns `Session.Info` with `id`. Source: `packages/opencode/src/server/routes/instance/session.ts:223` and `packages/opencode/src/session/session.ts:200-208`.
- `POST /session/<id>/prompt_async` (`session.prompt_async`) — body matches
  `PromptInput` minus `sessionID`; accepts `agent: string?` (the field
  that binds the session to a primary agent) and `parts: Array<TextPartInput | …>`.
  Returns 204. Source: `packages/opencode/src/server/routes/instance/session.ts:908-947`
  and `packages/opencode/src/session/prompt.ts:1776-1798`.
- `DELETE /session/<id>` — returns `true` on success, 404 if the session
  doesn't exist. Used for best-effort orphan-session cleanup. Source:
  `packages/opencode/src/server/routes/instance/session.ts:247-278`.

**URL prefix**: all three routes are mounted under `/session/...`, NOT
`/instance/session/...`. Only `/instance/reload` and `/instance/dispose`
live under the `/instance/` prefix.

## Acceptance Criteria

1. **Given** a running OpenCode server with the static `qs-<phase>` agents
   discoverable in `.opencode/agents/`,
   **When** `python scripts/qs/spawn_session.py --agent qs-create-plan
   --directory /path/to/worktree --prompt "Begin your phase protocol."` is invoked,
   **Then** the script calls `POST /session` with body `{"title": "qs-create-plan"}`
   (or the value of `--title`), captures the returned session `id`, then
   calls `POST /session/<id>/prompt_async` with body exactly
   `{"agent": "qs-create-plan", "parts": [{"type": "text", "text": "Begin your phase protocol."}]}`
   (no `model`, no `system`, no `messageID` fields — server defaults apply),
   and exits 0 with a JSON payload on stdout containing keys:
   `session_id` (str), `agent` (str), `title` (str), `directory` (str),
   `prompt` (str), `status: "session_created"`. Both API calls send the
   `x-opencode-directory: <directory>` header.

2. **Given** the new `spawn_session.py`,
   **When** invoked at all,
   **Then** it makes NO HTTP call to any URL containing `/instance/reload`,
   spawns NO `subprocess.Popen` (`urllib.request.urlopen` only), and
   accepts NO `--delay` CLI flag (argparse exits 2 with "unrecognized
   arguments" if `--delay` is passed). A negative regression test
   `test_no_instance_reload_call` pins this contract.

3. **Given** `python scripts/qs/next_step.py --next-cmd create-plan
   --harness opencode --work-dir /worktree --issue 42 --title "Foo"`,
   **When** invoked,
   **Then** the JSON payload returned has `tool: "opencode"`,
   `agent: "qs-create-plan"`, `same_context: "create-plan"`, and
   `new_context` equal to a single-line shell command of the form
   `python scripts/qs/spawn_session.py --agent qs-create-plan --directory
   <shlex.quote(worktree)> --title <shlex.quote('QS_42: Foo')> --prompt
   <shlex.quote(kickoff)>`. All interpolated values go through
   `shlex.quote` (matching `scripts/qs/launchers/claude.py`'s convention).

4. **Given** an unknown phase passed to the new OpenCode launcher,
   **When** `next_step.py --next-cmd bogus --harness opencode --work-dir /tmp
   --issue 1 --title T` is invoked,
   **Then** the launcher raises `UnknownPhaseError` (subclass of
   `ValueError`), `next_step.py` catches it and emits a JSON error payload
   `{"error": "unknown phase", "value": "bogus", "known": [...]}` to
   stdout, exit code 1. (Same contract as `claude-code` and `cursor`
   today.)

5. **Given** the OpenCode server is unreachable (any of: connection
   refused, DNS failure, timeout exceeded, `URLError`, `HTTPError` with
   non-2xx status, `OSError`, or malformed JSON response from
   `POST /session`),
   **When** `spawn_session.py --agent qs-create-plan --directory /tmp/x
   --prompt P` is invoked,
   **Then** the script catches the failure, probes
   `shutil.which("opencode")` for a CLI fallback binary, and emits a JSON
   payload on stdout with one of two shapes:
   - **If `opencode` is on PATH**: `{"status": "fallback_cli", "agent":
     "qs-create-plan", "directory": "/tmp/x", "new_context_cli": "opencode
     /tmp/x --agent qs-create-plan --prompt 'P'", "detail": "<error
     message>"}` — exit code 0.
   - **If `opencode` is NOT on PATH**: `{"status": "fallback_unavailable",
     "agent": "qs-create-plan", "directory": "/tmp/x", "detail": "<error
     message>; opencode CLI not on PATH — install or activate the server
     manually"}` — exit code 2.
   In both cases the script also writes a single human-readable line to
   stderr describing what happened. The pipeline never hangs.

6. **Given** `POST /session` succeeds (the server returns 200 with an
   `id`) but the subsequent `POST /session/<id>/prompt_async` fails with
   any error,
   **When** the failure occurs,
   **Then** the script makes a best-effort `DELETE /session/<id>` to clean
   up the orphan session. If DELETE succeeds, the script emits the
   fallback-CLI payload from AC #5 (because the user still needs to land
   in the next phase). If DELETE also fails, the script emits
   `{"status": "session_orphaned", "session_id": "<id>", "agent": "...",
   "directory": "...", "new_context_cli": "...", "detail": "..."}` —
   exit code 2 — so the user knows to clean up manually in the OpenCode
   UI.

7. **Given** an HTTP timeout (default 10 seconds, matching the legacy
   `_api()` timeout) is exceeded on either `POST /session` or
   `POST /session/<id>/prompt_async`,
   **When** the timeout fires,
   **Then** the same fallback-CLI / orphan-session paths from AC #5 / #6
   apply. No infinite hang.

8. **Given** `setup_task.py` builds the Phase 1 → create-plan handoff
   payload under harness=opencode,
   **When** it invokes `LAUNCHERS["opencode"].build_payload(...,
   caller="setup_task")`,
   **Then** the opencode launcher returns `new_context` set to a
   CLI-form command (`opencode <worktree> --agent qs-create-plan --prompt
   <kickoff>` via a generated `sh /tmp/qs_oc_launch_<N>.sh` one-liner,
   parallel to `scripts/qs_opencode/utils.py:opencode_launch_command`),
   NOT a `spawn_session.py` invocation. Rationale: setup-task runs on the
   main checkout; the new worktree is a different OpenCode workspace so
   HTTP-API session creation against the main-checkout server can't open
   the worktree as a project. Cross-workspace requires a fresh CLI
   launch. The legacy Phase 1 behavior is preserved.

9. **Given** `next_step.py` builds an intermediate-phase handoff payload
   under harness=opencode,
   **When** it invokes `LAUNCHERS["opencode"].build_payload(...,
   caller="next_step")` (the default if `caller=` is omitted),
   **Then** the launcher returns `new_context` set to the
   `python scripts/qs/spawn_session.py …` invocation per AC #3 (HTTP API
   path).

10. **Given** the legacy OpenCode pipeline files
    (`scripts/qs_opencode/`, `_qsprocess_opencode/`,
    `.opencode/agents/qs-setup-task.md`, `.opencode/agents/qs-release.md`,
    `.opencode/commands/`),
    **When** the new files are added,
    **Then** none of those legacy paths are modified (verified by
    `git status` at quality-gate time — no diff lines outside the
    new pipeline's edit scope).

11. **Given** `docs/workflow/harness.md`,
    **When** the new launcher is in place,
    **Then** two surgical edits are applied:
    - Line 9 (table row for OpenCode legacy) gains a sibling row for the
      "OpenCode (new pipeline)" entry whose "Session spawn" column reads
      `HTTP API: POST /session + POST /session/<id>/prompt_async (no reload)`.
    - Lines 55-56 (Launcher dispatch bullet for `launchers/opencode.py`)
      are replaced with a description of the new behavior — POSTs to the
      OpenCode HTTP API via `scripts/qs/spawn_session.py` with CLI
      fallback when the server is unreachable; setup-task Phase 1 keeps
      the CLI-form (cross-workspace).
    No other docs are touched in this PR.

12. **Given** static `.opencode/agents/qs-<phase>.md` files are NOT
    created by this PR (per user clarifying-question 4 — out of scope),
    **When** a user runs the new opencode handoff before adding those
    files,
    **Then** the launcher's docstring AND the new `harness.md` paragraph
    BOTH explicitly note: "the API call succeeds, but the session lands
    on the default OpenCode agent if `.opencode/agents/qs-<phase>.md` is
    missing. Mirror `.claude/agents/*.md` into `.opencode/agents/` (with
    frontmatter conversion) to enable agent activation." This is a known
    limitation documented up-front, not a silent bug.

### OpenCode HTTP API contract (verified against `/Users/tmenguy/Developer/WildMoka/opencode`)
- **Base URL**: `http://127.0.0.1:{port}` (port from `OPENCODE_SERVER_PORT`
  env var, default 4096).
- **Mandatory header on every call**: `x-opencode-directory: <worktree-abs-path>`
  — without it the session is created under the global project and won't
  show in the worktree's sidebar.
- **Session create**:
  - `POST /session`
  - Body: `{"title": "<str>"}` (`parentID`, `permission`, `workspaceID`
    accepted but not used by this script).
  - Returns 200 with `Session.Info` JSON including `id`.
  - Source: `packages/opencode/src/server/routes/instance/session.ts:223`.
- **Prompt + agent activation**:
  - `POST /session/<id>/prompt_async`
  - Body (exactly): `{"agent": "<qs-phase>", "parts": [{"type": "text",
    "text": "<prompt>"}]}` — no `model`, no `system`, no `messageID`.
    Server defaults apply.
  - Returns 204 (no content).
  - The `agent` field sets the session's bound primary agent.
  - Source: `packages/opencode/src/session/prompt.ts:1776-1798` (PromptInput
    schema) and `packages/opencode/src/session/message-v2.ts:482-498`
    (TextPartInput schema).
- **Session delete (best-effort cleanup)**:
  - `DELETE /session/<id>`
  - Returns 200 with `true` on success, 404 if not found.
  - Source: `packages/opencode/src/server/routes/instance/session.ts:247-278`.
- **Reload — NOT used** in the new pipeline. The legacy
  `POST /instance/reload` call was needed to make the server re-scan
  per-task rendered agent files. With static agents (committed under
  `.opencode/agents/`), the server discovers them at startup; no live
  re-scan is needed. Skipping reload eliminates the `SessionRunState`
  cancellation race that blocked the legacy pipeline from activating the
  agent in the same request.

### Bifurcation mechanism (`caller=` kwarg)
- Both `scripts/qs/setup_task.py` and `scripts/qs/next_step.py` dispatch
  to the same `LAUNCHERS[harness].build_payload(...)`. With no caller
  signal, the opencode launcher cannot tell "Phase 1 cross-workspace"
  apart from "intermediate in-worktree handoff".
- Solution: add a `caller: Literal["setup_task", "next_step"] =
  "next_step"` kwarg to every launcher's `build_payload`. Only the
  opencode launcher branches on it; other launchers accept and ignore
  (documented as "reserved for harness-specific bifurcation").
- `setup_task.py` passes `caller="setup_task"` explicitly. `next_step.py`
  omits the kwarg (default applies). This keeps the call sites
  self-documenting and grep-able.

### CLI fallback semantics
- When the OpenCode server is unreachable (any error class — connection
  refused, DNS failure, timeout, HTTPError, OSError, JSONDecodeError,
  missing `id`), `spawn_session.py` catches the failure and probes
  `shutil.which("opencode")`. If the CLI binary is on PATH, emit
  `status: "fallback_cli"` + `new_context_cli` shell command (exit 0).
  Otherwise emit `status: "fallback_unavailable"` (exit 2 — automation
  must distinguish "user can recover" from "user cannot recover").
- Why exit 0 for `fallback_cli`: the user has an actionable next step
  (the printed shell command). Why exit 2 for `fallback_unavailable`:
  the user has no recovery path until they install opencode. Three
  distinct exit codes (`0` = session live, `0` = printed-CLI fallback,
  `2` = no fallback / orphan) let downstream tooling react correctly.

### Orphan-session handling
- If `POST /session` succeeds (returns an `id`) but the subsequent
  `POST /session/<id>/prompt_async` fails, the partial session is
  detected and a best-effort `DELETE /session/<id>` is attempted. The
  user is then routed to the fallback-CLI path (so they still land in
  the next phase).
- If DELETE also fails, the orphan session id is surfaced in the
  output JSON (`status: "session_orphaned"`) so the user knows to clean
  up via the OpenCode UI. Exit code 2 (matches the
  `fallback_unavailable` "user must act" signal).

### Decisions made
- **Bifurcation** = explicit `caller=` kwarg (Option Q1.a). Cleaner than
  phase-name heuristic (fragile if create-plan is re-launched) or
  dual-payload (forces every caller to choose).
- **CLI fallback** = inside `spawn_session.py` (Option Q2.a). Single
  decision point, contained test surface, automation-friendly exit codes.
- **Kickoff prompt** = `--prompt` flag with default (Option Q3.a). Cheap
  flexibility, matches `claude --agent` pattern.
- **Orphan session** = best-effort DELETE + `session_orphaned` status
  (Option Q4.a). Safe, small, observable.
- **Static `.opencode/agents/`** = ship launcher only, document
  limitation (Option Q5.a). Keeps PR focused per user direction.

### Known risks accepted
- Until `.opencode/agents/qs-<phase>.md` static files are added (follow-up
  PR), the new launcher emits commands that succeed at the API level but
  leave the session on the default OpenCode agent. The launcher
  docstring and `harness.md` both flag this; users are expected to
  either mirror agents manually or wait for the follow-up PR.
- The `caller=` kwarg is added to all 4 launchers (claude, cursor,
  codex, opencode) but only opencode uses it. The other launchers gain
  a no-op parameter — minor signature noise, but keeps the dispatch
  table uniform and prevents `unexpected keyword argument` errors.
- HTTP timeout default is 10 seconds (matches the legacy `_api()`
  timeout). A slow-but-eventually-responsive server within 10s gets the
  happy path; beyond 10s it's treated as unreachable. Tunable via
  future flag if needed (not in scope).
