# QS-173 Review Fix #04

Phase 4 re-review of PR #174 (after fix #03) surfaced 2 must-fix and 5
should-fix items, all clustered around the cleanliness-gate / recovery
logic introduced by fix #03. User selected option (c): land all
must-fix + all should-fix. Nice-to-haves are explicitly out of scope.

## Findings to address

### must-fix

**MF-1 (cleanliness gate blocks happy-path no-op):** at
`scripts/worktree-setup.sh:104-113`, the cleanliness check fires
**before** the "already set up" early-return at L130-132. A developer
actively working in `QS_<N>` (uncommitted edits, untracked files) who
re-runs setup for any reason (re-symlink, re-verify upstream) now gets
"Refusing to destructively recover" and exit 1 — even though no
recovery is needed. The fix #03 hoist was structurally correct per
fix-#03 AC-2 wording but logically wrong: cleanliness gating must only
apply on the destructive arms, not before the happy-path detection.

**MF-2 (macOS `.DS_Store` blocks untracked-directory recovery):** at
`scripts/worktree-setup.sh:100`, the `find` exclusion is only
`! -name '.git' ! -name '.git*'`. Finder browsing the worktree creates
`.DS_Store`; common tool dotfiles (`.idea/`, `.vscode/`, `.envrc`,
`.python-version`, etc.) also trigger `HAS_NONGIT_CONTENT="yes"`, so
after any partial-state failure the operator must manually `rm` these
before re-running.

### should-fix

**SF-1 (`.git*` glob overshoot):** the same exclusion line treats
`.gitignore`, `.gitattributes`, `.gitmodules` as "no content" → `rm -rf`
silently deletes them. Tighten to exclude only `.git` exactly.

**SF-2 (push-failure hint misleads operator):** at
`scripts/worktree-setup.sh:290-297`, the wrapper says *"Worktree IS
fully set up locally — only the publish step failed."* But re-running
the script rebuilds the worktree from scratch via the partial-state
recovery arm. Either redirect the operator to `git push -u origin
<branch>` manually, or clarify that re-running rebuilds.

**SF-3 (`HAS_NONGIT_CONTENT` diagnostic uninformative):** at L109, the
message prints `"Other: yes"` instead of the offending filenames.

**SF-4 (silent `ls-remote` failure skips divergence guard):** at L244,
`2>/dev/null` suppresses network/auth errors → empty `REMOTE_TIP` →
divergence check becomes a no-op without warning. If origin/QS_<N> has
actually diverged, the subsequent push attempt becomes a
non-fast-forward that the divergence check would have blocked.

**SF-5 (`find ... 2>/dev/null` swallows permission errors):** at L100,
if `$WORKTREE_DIR` is unreadable (sudo-owned from a prior run), find
emits nothing → `HAS_NONGIT_CONTENT` stays empty → recovery falls
through to `rm -rf` which then fails messily.

## Out of scope (explicitly NOT in this fix)

- `coderabbit MD038/MD040` markdownlint nits on fix-plan #03 — committed
  artifact, not worth re-touching.
- Fetch `2>/dev/null` at L256 hiding error detail (separate concern).
- Repo-wide stash false-positive on cleanliness check (acknowledged OOS
  multiple times).
- `MAIN_DIR` containing `..` segments breaking the membership check

- Credential-prompt hangs (`GIT_TERMINAL_PROMPT=0`) — separate concern.
- Untracked-`??`-only relaxation in cleanliness gate (subsumed by MF-1

  gate).
- Empty-string rev-parse fallback at L115 (mitigated by MF-1 reorder).

## Plan

### Task A — Reorder cleanliness gate (`scripts/worktree-setup.sh`)

**A.1 (MF-1)** — Move the cleanliness-gate block (currently L76-113) to
run **only inside the destructive recovery arms**, not before them.
Target structure inside `if [ -d "$WORKTREE_DIR" ]; then`:

```bash
if [ -d "$WORKTREE_DIR" ]; then
    # 1. Detect IS_TRACKED_WORKTREE (lines 57-67 unchanged)
    IS_TRACKED_WORKTREE="no"
    if git -C "$MAIN_DIR" worktree list --porcelain | ... ; then
        IS_TRACKED_WORKTREE="yes"
    fi

    if [ "$IS_TRACKED_WORKTREE" = "yes" ]; then
        # 2. Detached-HEAD short-circuit (unchanged, currently L115-128)
        ACTUAL_HEAD="$(...)"
        if [ "$ACTUAL_HEAD" = "HEAD" ]; then
            # …existing detached-HEAD error path…
            exit 1
        fi
        # 3. Happy-path early return (unchanged, currently L129-133)
        ACTUAL_UPSTREAM="$(...)"
        if [ "$ACTUAL_HEAD" = "$BRANCH" ] && [ "$ACTUAL_UPSTREAM" = "origin/${BRANCH}" ]; then
            echo "Worktree ${WORKTREE_DIR} already set up on ${BRANCH} tracking origin/${BRANCH}."
            exit 0
        fi
        # 4. Partial state → CLEANLINESS GATE (moved here)
        # …moved cleanliness check (DIRTY/STASHES/HAS_NONGIT_CONTENT)…
        echo "Worktree ${WORKTREE_DIR} exists but is in a partial state"
        echo "  (HEAD='${ACTUAL_HEAD}', upstream='${ACTUAL_UPSTREAM}')."
        # 5. worktree remove --force (unchanged, currently L136-147)
        echo "Removing tracked worktree to recover..."
        if ! git -C "$MAIN_DIR" worktree remove --force "$WORKTREE_DIR"; then
            # …existing unlock-hint error path…
            exit 1
        fi
    else
        # 6. Untracked stale directory → CLEANLINESS GATE (moved here, run BEFORE rm -rf)
        # …moved cleanliness check (HAS_NONGIT_CONTENT only — no .git, no DIRTY/STASHES)…
        echo "Directory ${WORKTREE_DIR} exists but is not a tracked git worktree."
        echo "Removing stale directory to recover..."
        rm -rf "$WORKTREE_DIR"
    fi
fi
```

The cleanliness-gate logic is **duplicated** into both arms (DIRTY/
STASHES/non-git-content for the tracked-recovery arm; just non-git-content
for the untracked arm) rather than hoisted. Acceptable code duplication
to keep the happy-path no-op fast and unblocked.

Alternative (preferred for less duplication): keep the gate as a single
block but invoke it **after** the happy-path early-return and detached-HEAD
short-circuit, still gating both destructive paths.

### Task B — Tighten and improve content detection (`scripts/worktree-setup.sh`)

**B.1 (MF-2 + SF-1)** — Change the `find` exclusion at line 100 to:

```bash
if find "$WORKTREE_DIR" -mindepth 1 -maxdepth 1 \
    ! -name '.git' \
    ! -name '.DS_Store' \
    ! -name '.idea' \
    ! -name '.vscode' \
    ! -name '.envrc' \
    ! -name '.python-version' \
    -print 2>/dev/null | head -1 | grep -q .; then
    HAS_NONGIT_CONTENT="yes"
    ...
fi
```

Key changes:
- Drop `! -name '.git*'` (was overshooting to `.gitignore` etc.).
- Add explicit exclusion of common macOS/IDE/tool dotfiles that aren't
  user-meaningful.
- Keep `.gitignore`, `.gitattributes`, `.gitmodules` as **content** —
  if they're present in a stale directory, the operator probably wants
  to know.
- Use `-print | head -1` rather than `-print -quit` to also capture the
  *name* of the first offending file (see B.2).

**B.2 (SF-3)** — Capture the names of offending files into the diagnostic:

```bash
OFFENDING="$(find "$WORKTREE_DIR" -mindepth 1 -maxdepth 1 \
    ! -name '.git' \
    ! -name '.DS_Store' \
    ! -name '.idea' \
    ! -name '.vscode' \
    ! -name '.envrc' \
    ! -name '.python-version' \
    -print 2>/dev/null | head -5)"
if [ -n "$OFFENDING" ]; then
    HAS_NONGIT_CONTENT="$OFFENDING"
fi
```

Then in the diagnostic message (currently L106-111):
```bash
echo "  Other:   ${HAS_NONGIT_CONTENT:-(none)}"  # now prints filenames
```

The first 5 names cap the output length while still being informative.

**B.3 (SF-5)** — Check readability before the find call:

```bash
if ! { [ -r "$WORKTREE_DIR" ] && [ -x "$WORKTREE_DIR" ]; }; then
    echo "Error: ${WORKTREE_DIR} exists but is not readable/executable."
    echo "Permission issue (possibly sudo-owned from a prior run). Resolve manually:"
    echo "  ls -la ${WORKTREE_DIR}"
    echo "  sudo chown -R \$(whoami) ${WORKTREE_DIR}   # if appropriate"
    exit 1
fi
```

Place this before the find call.

### Task C — Push wrapper wording (`scripts/worktree-setup.sh`)

**C.1 (SF-2)** — At the push wrapper (currently L290-297), reword to
direct the operator at the manual fix that actually works:

```bash
echo "Publishing ${BRANCH} to origin and setting upstream..."
if ! git -C "$WORKTREE_DIR" push -u origin "${BRANCH}"; then
    echo "Error: 'git push -u origin ${BRANCH}' failed."
    echo "Common causes: auth expired, network drop, pre-push hook reject."
    echo "The worktree is built locally and the failure is just in the"
    echo "publish step. The simplest fix is to retry the push directly:"
    echo "  cd ${WORKTREE_DIR} && git push -u origin ${BRANCH}"
    echo "Re-running this script would rebuild the worktree from scratch"
    echo "(slower, but harmless if you prefer that)."
    exit 1
fi
```

### Task D — `ls-remote` failure detection (`scripts/worktree-setup.sh`)

**D.1 (SF-4)** — Distinguish "remote branch doesn't exist yet" from
"`ls-remote` failed". Capture exit code:

```bash
REMOTE_TIP_OUTPUT="$(git -C "$WORKTREE_DIR" ls-remote --heads origin "$BRANCH" 2>&1)"
LS_REMOTE_RC=$?
if [ $LS_REMOTE_RC -ne 0 ]; then
    echo "Error: 'git ls-remote --heads origin ${BRANCH}' failed."
    echo "  Output: ${REMOTE_TIP_OUTPUT}"
    echo "Cannot verify divergence before push. Resolve manually:"
    echo "  cd ${WORKTREE_DIR} && git ls-remote --heads origin ${BRANCH}"
    exit 1
fi
REMOTE_TIP="$(echo "$REMOTE_TIP_OUTPUT" | awk '{print $1}')"
# Empty REMOTE_TIP now genuinely means "branch doesn't exist on origin",
# not "ls-remote failed".
if [ -n "$REMOTE_TIP" ]; then
    # …existing divergence-check body…
fi
```

### Task E — Manual verification

1. Working dev session: edit a tracked file in an existing healthy
   `QS_<N>` worktree, re-run setup → exits 0 with "already set up"
   message (MF-1 regression test).
2. Add `.DS_Store` to a stale untracked `$WORKTREE_DIR` → recovery
   proceeds (MF-2 regression test).
3. Add `.gitignore` (alone) to a stale untracked `$WORKTREE_DIR` →
   recovery refuses, message names `.gitignore` (SF-1 + SF-3 regression).
4. `chmod 000 $WORKTREE_DIR` → script exits with permission message
   (SF-5).
5. Set `origin` to a non-existent host or kill the network, run setup →
   D.1 catches `ls-remote` failure with the actual error message
   (SF-4).
6. Push fails (e.g. revoke gh token, run script) → C.1 push-wrapper
   message points at `git push -u origin <branch>` retry, not at
   re-running the script (SF-2).
7. Re-run full happy path on fresh issue → no regression.

No automated tests (parent story decision).

## Acceptance criteria

- **AC-1:** in `scripts/worktree-setup.sh`, an existing tracked worktree
  with HEAD on `$BRANCH` and upstream `origin/$BRANCH` exits 0 with the
  "already set up" message **regardless of working-tree cleanliness**.
- **AC-2:** the destructive recovery arms (`worktree remove --force`
  and `rm -rf`) each refuse to run when uncommitted edits, stashes,
  or non-git content are present.
- **AC-3:** `find`-based content detection excludes `.git`, `.DS_Store`,
  `.idea`, `.vscode`, `.envrc`, `.python-version` but **does not** treat
  `.gitignore`/`.gitattributes`/`.gitmodules` as ignorable.
- **AC-4:** the cleanliness-gate diagnostic enumerates the first few
  offending filenames (not just `yes`).
- **AC-5:** the script checks `$WORKTREE_DIR` is readable and executable
  before attempting any `find` against it, with a clear error message
  on failure.
- **AC-6:** the push-failure error message points the operator at
  `git push -u origin <branch>` as the primary remediation and notes
  that re-running rebuilds the worktree.
- **AC-7:** `ls-remote` failures (network, auth) are detected and
  surfaced with the actual git error output; the divergence check
  becomes a no-op only when origin/`$BRANCH` genuinely doesn't exist.
- **AC-8:** all ACs from the parent story, fix #01, fix #02, and fix
  #03 still hold (no regressions).

## Implementer notes

- `/implement-setup-task` is correct: only file touched is
  `scripts/worktree-setup.sh`.
- For Task A, prefer the "single block invoked after happy-path
  short-circuit" form (less duplication) over duplicating into both
  arms. The single block runs after the IS_TRACKED_WORKTREE=yes →
  detached-HEAD → already-set-up sequence has had a chance to exit
  cleanly, and also runs once before the `rm -rf` for the untracked
  arm.
- For B.2, `head -5` is a tradeoff: too few names hide useful detail,
  too many spam the terminal. 5 is enough to show the operator the
  pattern (e.g. several `.cache/` files) without flooding.
- For C.1, the rewording is the only deliverable — don't redesign the
  push flow.
- For D.1, capture `2>&1` so stderr appears in `$REMOTE_TIP_OUTPUT`
  for the error-path diagnostic.
- After implementing, re-grep `scripts/worktree-setup.sh` for any
  remaining `2>/dev/null || true` patterns — they're the SF-4 / SF-5
  class of swallowed-error bug; flag any to the next reviewer if you
  encounter them but consider them out of scope unless they affect
  one of this round's ACs.
