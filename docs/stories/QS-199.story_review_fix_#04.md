## Findings to fix

### [should-fix] ES1 — drag snap range desyncs from the rendered gauge when `max_default_hours > 168`
- File: `custom_components/quiet_solar/ui/resources/shared/qs-ring-duration-base.js:126` (`_allowedHalfHours`), and the gauge math in each duration card (`qs-radiator-card.js:209/269-270`, `qs-climate-card.js:695/888-889`, `qs-water-boiler-card.js:734/792-793`, `qs-on-off-duration-card.js:72/102-103`)
- Severity: should-fix
- Source: qs-review-edge-case-hunter
- Scope: NEW regression from fix #03 (the `Math.min(n, 168)` clamp added for M1/N1)
- Description: The clamp bounds only the draggable snap range. The gauge
  rendering (`hoursToPct`/`pctToHours`, background/progress arcs, handle
  initial position) uses the **unclamped** `maxHours`. With
  `max_default_hours: 200` the ring renders the full 0..200h scale, but
  `allowedHours = _allowedHalfHours(maxHours)` spans only 0..168, so the
  snap-reduce in `_wireTargetHandle` caps the dragged value at 168 — the user
  sees the top ~16% of the gauge as a dead zone and can never select a target
  between 168h and the configured max. The `Number.isFinite` guard from M1 is
  what actually prevents the Infinity hang; the additional `Math.min(n, 168)`
  is what creates this mismatch (a finite 200h snap list is only 401 elements
  — not a performance problem).
- Proposed fix (root-cause): Clamp `maxHours` ONCE at the single source that
  feeds BOTH the gauge and the snap list (i.e. where each card derives
  `maxHours` from `cfg.max_default_hours`), then let `_allowedHalfHours`
  consume that already-clamped value WITHOUT its own `Math.min`. Keep the
  `Number.isFinite(n) && n > 0 ? … : 12` guard in `_allowedHalfHours` (defends
  against Infinity regardless of caller). This guarantees gauge and snap can
  never diverge. If a shared clamp helper is cleaner, hoist it into the ring
  base. Add a test asserting that for `max_default_hours = 200` the gauge max
  and the max snap value are equal (and that `Infinity`/`1e999` still produce
  a bounded list).

### [should-fix] ES2 — orphan `.qstmp` sweep deletes concurrently-active temps (race re-introduced)
- File: `custom_components/quiet_solar/ui/dashboard.py:586` (`_async_remove_orphan_temps`, called from `async_update_resources`)
- Severity: should-fix
- Source: qs-review-edge-case-hunter
- Scope: NEW regression from fix #03 (N2 orphan sweep)
- Description: The startup sweep deletes EVERY `*.qstmp` under the destination
  tree. If a second invocation of `async_update_resources` is running
  concurrently (the code is explicitly unlocked — startup-restore at
  dashboard.py:192 can race the "Generate Dashboard" button at dashboard.py:708,
  or two rapid button presses), the sweep can delete a temp that the other
  invocation is holding between its `await f_out.write(data)` and its
  `await os.replace(tmp, dst)`. The `os.replace` then raises `FileNotFoundError`
  (an `OSError`), which `_async_atomic_write_bytes` re-raises and which bubbles
  to the broad `except Exception` — silently aborting the rest of that
  invocation's resource copy and logging only a warning. This is the same
  "silently skips the remaining files" failure mode the round-2 S1 unique-name
  scheme was meant to eliminate; the unique name no longer protects the temp
  once a blanket suffix-glob delete runs against it.
- Proposed fix (root-cause): Serialize the whole resource update with a
  module-level `asyncio.Lock` (e.g. `_RESOURCE_UPDATE_LOCK`) acquired at the top
  of `async_update_resources` for the duration of sweep + copy + register. This
  eliminates the ENTIRE concurrency-race class (both the round-2 S1 temp race
  and this ES2 sweep race) rather than patching the sweep in isolation. (Keep
  the unique temp name as defense-in-depth.) Add a test that two overlapping
  `async_update_resources` calls both complete without spurious
  `FileNotFoundError` and without skipping files (e.g. assert all expected
  resources are registered after concurrent invocations).

### [nice-to-have] NH1 — local-target clear timer scheduled before the (dead) `await onCommit`
- File: `custom_components/quiet_solar/ui/resources/shared/qs-card-base.js:438-447` (`_wireTargetHandle.onUp`)
- Severity: nice-to-have
- Source: qs-review-blind-hunter
- Description: The 5s `_pendingClearLocalTarget` timer is scheduled BEFORE
  `await onCommit(dragValue)`. If a future caller passes a slow `onCommit`, the
  local-target could clear mid-commit. Latent today — no card passes `onCommit`
  (only `onBeforeCommit` is used), so the branch is effectively dead.
- Proposed fix: Schedule the timer AFTER `onCommit`, or drop the unused
  `onCommit` param entirely (keep only `onBeforeCommit` if that's all any card
  needs).

### [nice-to-have] NH2 — pool commit guard semantics shift
- File: `custom_components/quiet_solar/ui/resources/qs-pool-card.js:1263-1284` vs `shared/qs-card-base.js:436`
- Severity: nice-to-have
- Source: qs-review-blind-hunter
- Description: Pre-migration the pool committed only when
  `dragValue != null && e.default_on_duration && e.pool_mode` (all three). Now
  the base guard is `dragValue != null && entityId(=default_on_duration)` and
  `onBeforeCommit` self-guards `if (e.pool_mode)`. So if `pool_mode` is absent
  but `default_on_duration` is present, the duration is now written without the
  mode-select (old code skipped entirely). Arguably more correct, but a
  diff-visible semantics shift.
- Proposed fix: Confirm the intended behavior. If the old "skip entirely when
  pool_mode absent" was deliberate, restore that combined guard; otherwise add
  a brief comment documenting the intentional change.

### [nice-to-have] CR1 — disconnected pseudo-buttons remain in the tab order

### [nice-to-have] CR3 — merge palette defaults per key, not only on falsy input
- File: `custom_components/quiet_solar/ui/resources/shared/qs-card-styles.js:36`
- Severity: nice-to-have
- Source: qs-review-coderabbit (Minor; raised vs 45f25a62 — verify vs head)
- Description: `const colors = palette || { … }` only substitutes defaults when
  `palette` is entirely falsy. Passing `{}` or a partial palette emits
  `undefined` into the generated CSS and breaks gradients / `color-mix` rules.
  Latent today (all 6 callers pass a full palette) but fragile.

  Add a test passing `{}` / a partial palette and asserting no `undefined` in
  the emitted CSS.
