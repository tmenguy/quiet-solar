## Findings to fix

### [must-fix] `load_score` possible `UnboundLocalError` in `LoadConstraint.score()`
- File: `custom_components/quiet_solar/home_model/constraints.py:335-337` (referenced at `:375`)
- Severity: must-fix
- Source: qs-review-coderabbit (verified still present at head `05fa062c`)
- Description: `load_score` is only assigned inside the `if self.load is not None:`
  branch but referenced unconditionally in the return expression
  (`+ energy_score_span * load_score`). QS-256 makes `score()` participate
  directly in override dedup/ordering, so a load-less constraint would crash
  with `UnboundLocalError` instead of falling back to a zero-load baseline.
- Proposed fix: initialize `load_score = 0.0` before the
  `if self.load is not None:` branch. Pin with a test that calls `score()` on
  a constraint whose `load` is None and asserts it returns the zero-load
  baseline (no exception), and that the override dominance invariant still
  holds with `load_score == 0.0`.

### [nice-to-have] Hardcoded persisted-payload key literals in tests
- File: `tests/test_qs256_override_lifecycle.py:141-147, 346-356, 390-396, 643-650, 731-739, 753-761`
- Severity: nice-to-have
- Source: qs-review-coderabbit (coding guideline: config/storage keys live in `const.py`, never hardcoded)
- Description: the restore payloads in the QS-256 lifecycle tests hardcode
  storage key strings (`"num_on_off"`, `"current_command"`,
  `"last_state_change_time"`, `"external_user_initiated_state_time"`,
  `"asked_for_reset_user_initiated_state_time"`, etc.). A future rename of a
  persisted key would silently desynchronize tests from production.
- Proposed fix: centralize the persisted-payload key names as constants in
  `custom_components/quiet_solar/const.py` (or reuse them if they already
  exist where the payload is built/read), use them at the production
  read/write sites, and import them in the tests. Keep the stored payload
  format byte-identical (constants must equal the current literals) so no
  migration is needed.

### [nice-to-have] Test-author trap: static conftest default `last_changed`
- File: `tests/conftest.py:55`
- Severity: nice-to-have
- Source: qs-review-blind-hunter
- Description: `FAKE_STATE_DEFAULT_LAST_CHANGED` is a hardcoded past date
  (2024-01-01). Combined with the QS-256 causality guard
  (`state_last_changed <= last_command_execution_time` → not a user action),
  any future test that exercises override detection and forgets to pass an
  explicitly fresher `last_changed` will silently get "no override detected".
- Proposed fix: documentation-level — add a comment at the constant
  explaining the interaction with the causality guard and instructing test
  authors to set an explicit `last_changed` newer than the load's
  `last_command_execution_time` when a test must classify a user override.
  No behavior change; no test value change.
