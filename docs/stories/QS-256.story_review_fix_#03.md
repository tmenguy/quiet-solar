## Findings to fix

### [must-fix] `TypeError` when override state is set but its timestamp is `None`
- File: `custom_components/quiet_solar/ha_model/bistate_duration.py:967`
- Severity: must-fix
- Source: qs-review-edge-case-hunter
- Description: The idle-override timer fallback computes
  `do_push_constraint_after = self.external_user_initiated_state_time +
  timedelta(...)` and raises `TypeError: unsupported operand type(s)
  for +: 'NoneType' and 'datetime.timedelta'` when
  `external_user_initiated_state` is set while
  `external_user_initiated_state_time` is `None`. Reproduces from
  legacy/hand-edited storage: the two fields are serialized
  independently (`home_model/load.py:889-892`; `_restored_utc_datetime`
  returns `None` for an absent key), and when the entity state already
  equals the override state the "still same override, all good" branch
  (`bistate_duration.py:736-740`) skips the path that would re-stamp
  the time. Every existing guard (AC2 restore drop at `:185`, the
  expiry/coercion guards at `:677-687`) short-circuits on
  `external_user_initiated_state_time is not None`, so nothing
  reconciles this combination before the dereference.
- Proposed fix: In `use_saved_extra_device_info`, if
  `external_user_initiated_state is not None` while
  `external_user_initiated_state_time is None`, drop the whole override
  (mirror the AC2 reset: clear all four override fields, log). Belt and
  braces: also guard the timer-fallback branch with
  `and self.external_user_initiated_state_time is not None`. Add a test
  restoring `external_user_initiated_state="off"` with no timestamp and
  the entity already in the override state.

### [should-fix] Future-dated `asked_for_reset_user_initiated_state_time` makes the cooldown permanently active
- File: `custom_components/quiet_solar/ha_model/bistate_duration.py:813`
- Severity: should-fix
- Source: qs-review-edge-case-hunter
- Description: A clearly future-dated restored
  `asked_for_reset_user_initiated_state_time` makes
  `(time - ask_time).total_seconds()` negative, which is always
  `< min(180, override_h*3600/2)`, so the post-override cooldown is
  treated as permanently active and override detection is suppressed
  until wall-clock catches up to the poison timestamp. The round-2
  future-dated drop (`age_s < -60.0`) was applied only to
  `external_user_initiated_state_time`, not to the restored ask-time.
- Proposed fix: Apply the same `< -60s` future-dated drop to
  `asked_for_reset_user_initiated_state_time` at restore (preferred,
  same site as the AC2 drop), or clamp negative diffs to "cooldown
  elapsed" at the check site. Add tests: future-dated ask-time (now+1h)
  → cooldown not active; small skew (now+30s) → tolerated.

### [nice-to-have] `push_live_constraint` same-end-time path contradicts the documented recency tie-break
- File: `custom_components/quiet_solar/home_model/load.py:1504-1513`
- Severity: nice-to-have
- Source: qs-review-edge-case-hunter
- Description: The `score()` docs (constraints.py:354-362) promise "the
  NEWER override wins an otherwise exact score tie". The cluster-dedup
  path (`load.py:1379`) honors this via score comparison, but the
  same-end-time branch in `push_live_constraint` unconditionally
  replaces the existing constraint with the newly-pushed one ("last
  pushed wins") regardless of which `start_of_constraint` is newer.
  Normally unreachable (detection wipes constraints before pushing), so
  impact is doc-drift only.
- Proposed fix: In the same-end-time branch, keep the higher-scoring
  constraint instead of always replacing — or scope the `score()` doc
  claim to the cluster-dedup/solver-ordering paths only. Either
  resolution accepted; if code is changed, add a test for
  push-older-after-newer.

### [nice-to-have] Sub-Wh energy delta can be reordered by the recency term
- File: `custom_components/quiet_solar/home_model/constraints.py:389-403`
- Severity: nice-to-have
- Source: qs-review-blind-hunter
- Description: The recency tie-break term is bounded (0, 0.5] and is
  added unconditionally to every USER_OVERRIDE score, not only on exact
  ties. Two overrides whose energy scores differ by less than the
  recency magnitude (<~0.5 Wh) would be ordered by recency rather than
  energy. The single-override-per-load invariant makes coexistence
  theoretical; the existing
  `test_qs256_newer_override_wins..._bounded_by_energy_granularity`
  asserts the bound but not the sub-Wh case.
- Proposed fix: Documentation-level resolution acceptable: state at the
  recency-term site that sub-Wh energy ties between two overrides (an
  impossible state by construction) are intentionally recency-broken.
  Optionally add a test pinning that behavior.

### [nice-to-have] Comment: restore-anchor blind spot at the causality-guard comparison
- File: `custom_components/quiet_solar/ha_model/bistate_duration.py:~775`
- Severity: nice-to-have
- Source: qs-review-blind-hunter

- Proposed fix: One-line comment at the
  `state_last_changed <= self.last_command_execution_time` comparison
  referencing the documented blind-spot tradeoff.
