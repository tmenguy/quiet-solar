### [should-fix] `was_incremental` captured pre-hygiene â†’ wasted self-heal retry on corrupt/schema-mismatched baseline
- File: `scripts/qs/quality_gate.py:1031`
- Severity: should-fix
- Source: qs-review-edge-case-hunter
- Description: `check_impacted` captures `was_incremental` from the *pre-hygiene*
  DB state (`.testmondata` present and `st_size > 0`). The actual
  select-all-vs-incremental decision happens later inside
  `_run_impacted_pass` â†’ `_ensure_testmon_db_safe()`, which can purge the DB.
  When `.testmondata` is present and non-empty but **corrupt or
  schema-mismatched**, `_ensure_testmon_db_safe()` purges it inside the first
  pass, so that pass already runs as a clean select-all. On a genuine
  changed-line gap the first pass returns `_IMPACTED_CHANGED_LINES_UNCOVERED`;
  `check_impacted` then sees `was_incremental=True` and fires a self-heal
  rebuild + a second full select-all retry, emitting a misleading "rebuilding
  testmon baseline" notice for what was never a desync. No false PASS/FAIL
  (the genuine gap retry still exits 1), but it doubles a full-suite run and
  the notice is incorrect. The existing self-heal tests never exercise this
  path because they mock `_run_impacted_pass` whole.
- Proposed fix: Make the "was this an incremental run?" signal reflect the
  state the first pass actually selected against, not the pre-hygiene state.
  Have `_run_impacted_pass` (or `_ensure_testmon_db_safe`) report whether it
  purged/reset the baseline (e.g. return a flag or set an out-param), and gate
  the self-heal retry on `was_incremental AND not purged_during_first_pass`.
  Equivalent alternative: re-check `.testmondata` existence/size immediately
  after the first pass and treat a baseline that was purged mid-pass as
  non-incremental so no retry fires. Add a unit test that drives the real
  purge-between-capture-and-selection path (corrupt/schema-mismatched
  `.testmondata`, genuine changed-line gap) and asserts exactly one pass runs,
  no self-heal notice is emitted, and exit code is 1.

### [nice-to-have] `check_impacted` docstring references a non-existent `allow_self_heal=False` inner-pass param
- File: `scripts/qs/quality_gate.py` (`check_impacted` docstring)
- Severity: nice-to-have
- Source: qs-review-blind-hunter
- Description: The docstring says the retry "runs with `allow_self_heal=False`
  via the inner pass," but the retry calls `_run_impacted_pass(base)` directly,
  which takes no `allow_self_heal` argument. No-recursion is correct
  structurally, but the wording implies a parameter that does not exist.
- Proposed fix: Reword to "the retry calls `_run_impacted_pass` directly, so it
  cannot recurse."

### [nice-to-have] Self-heal retry collapses every non-PASS verdict to exit 1, losing per-verdict diagnostics
- File: `scripts/qs/quality_gate.py:1047`
- Severity: nice-to-have
- Source: qs-review-edge-case-hunter
- Description: The retry maps `0 if _run_impacted_pass(base) == _IMPACTED_PASS
  else 1`, so a retry that fails for a non-coverage reason
  (`_IMPACTED_DIFF_COVER_TIMEOUT`, `_IMPACTED_NO_COVERAGE_XML`) is reported
  identically to a genuine coverage gap, with no distinguishing diagnostic.
  This matches the first-pass all-failures-are-exit-1 behavior, so it is not a
  regression, but the retry path loses the per-verdict messaging the first pass
  still emits.
- Proposed fix: Route the retry verdict through the same per-verdict
  diagnostic/messaging path the first pass uses before collapsing to the exit
  code, so a non-coverage retry failure is distinguishable in the output.
