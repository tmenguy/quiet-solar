## Findings to fix

### [must-fix] Falsy / non-string server `id` slips past the None-guard
- File: `scripts/qs/spawn_session.py` (around line 225, the `session_id = session.get("id")` + `if session_id is None: raise ...` block)
- Severity: must-fix
- Source: qs-review-edge-case-hunter
- Description: The guard `if session_id is None: raise SpawnSessionError(...)` only catches the missing-key and explicit-null cases. A server returning `{"id": ""}` (empty string), `{"id": 0}` (zero), `{"id": False}`, or `{"id": ["x"]}` (list) all survive the check. The value is then stringified via `str(session_id)` and URL-escaped via `urllib.parse.quote(..., safe="")`, producing nonsense paths like `/session//prompt_async` (double slash) or `/session/False/prompt_async`. The server responds with 404 and the script falls back gracefully, but the orphan-cleanup `DELETE` targets the same nonsense URL and the error logs are confusing.
- Proposed fix: Tighten the guard:
  ```python
  if not isinstance(session_id, str) or not session_id.strip():
      raise SpawnSessionError(
          f"server response missing or invalid 'id' field: {session!r}",
          ...
      )
  ```
  After this, `session_id` is guaranteed to be a non-empty string. Add a regression test `test_falsy_or_non_string_id_falls_back` parametrized over `[{"id": ""}, {"id": 0}, {"id": False}, {"id": ["x"]}, {"id": None}]`, each asserting the fallback path is taken (exit 0 with `fallback_cli` when `opencode` on PATH).

### [must-fix] `--agent ""` argparse validation gap
- File: `scripts/qs/spawn_session.py` (around line 413 in `main`, where the `--directory`, `--prompt`, `--title` empty/whitespace guards live)
- Severity: must-fix
- Source: qs-review-edge-case-hunter
- Description: `--agent` is declared `required=True` but accepts present-but-empty values (`""`, `"   "`). The parallel `parser.error` guards for `--directory`, `--prompt`, and `--title` (added in fix plan #01) all exist; `--agent` is the missing one. An empty agent name reaches `spawn_session()` and is sent verbatim as `{"agent": ""}` in the `prompt_async` body — the server's error is opaque, and the fallback CLI becomes `opencode <dir> --agent '' --prompt 'P'` which is shell-safe but useless.
- Proposed fix: Add a guard immediately after `parser.parse_args(...)`, mirroring the existing three:
  ```python
  if not args.agent.strip():
      parser.error("--agent must be a non-empty string")
  ```
  Add `test_empty_agent_rejected` parametrized over `["", "   ", "\t"]`, each asserting argparse exit code 2.

### [should-fix] `_spawn_session_command` does not coerce `issue` to int (parity gap with `_opencode_cli_command`)
- File: `scripts/qs/launchers/opencode.py` (`_spawn_session_command`, the function that builds the HTTP-API form of `new_context`)
- Severity: should-fix
- Source: qs-review-blind-hunter
- Description: `_opencode_cli_command` coerces `issue_int = int(issue)` at entry (fix plan #01 must-fix #2), but the sibling `_spawn_session_command` accepts the same `issue: int | str` signature without the same coercion. Today the title `f"QS_{issue}: {title}"` is wrapped in `shlex.quote` before reaching the shell, so a malicious `issue="42 && evil"` cannot break out — but the asymmetric input-validation contract is fragile. If a future refactor drops the `shlex.quote` wrapping, the vector reopens.
- Proposed fix: Add `issue_int = int(issue)` at the top of `_spawn_session_command` (or coerce to `int` and reject `bool` per the must-fix #2 pattern). Use `issue_int` consistently in the title interpolation. Tests: parametrize the existing `test_build_payload_returns_spawn_session_invocation` with `issue=str("42")` and `issue=42` and assert both produce the same `new_context`; add `test_spawn_session_command_rejects_non_integer_issue` mirroring the existing `test_opencode_cli_command_rejects_non_integer_issue`.

### [should-fix] Weak OR-disjunction assertion in shell-quote test
- File: `tests/qs/launchers/test_opencode_launcher.py:316-318` (`test_opencode_cli_command_shell_quotes_script_path`)
- Severity: should-fix
- Source: qs-review-blind-hunter
- Description: The assertion `assert shlex.quote(...) in new_context or "'" in new_context` permits any payload containing a stray single quote to satisfy the second clause — including unrelated content like the banner string or an agent name containing a quote. A regression that drops the path quoting would still pass.
- Proposed fix: Replace the OR with a structural check using `shlex.split`:
  ```python
  parts = shlex.split(new_context)
  assert parts[0] == "sh"
  assert parts[1].startswith(str(weird_tmp)), parts
  ```
  This pins that token 1 is exactly the (correctly-unquoted-after-split) script path, not just "some single quote appears somewhere".

### [should-fix] `_emit_fallback` triplicate stderr/json/exit pattern (refactor)
- File: `scripts/qs/spawn_session.py:262-302` (`_emit_fallback`)
- Severity: should-fix
- Source: qs-review-blind-hunter
- Description: Three code paths (`fallback_cli` / `fallback_unavailable` / `session_orphaned`) each duplicate the `print(..., file=sys.stderr) + output_json(payload) + sys.exit(N)` triplet with subtly different `detail` shaping. The asymmetry is documented but the structural duplication makes drift easy. A single-letter typo in the stderr message in one branch wouldn't be caught.
- Proposed fix: Introduce a small inner helper:
  ```python
  def _emit(payload: dict, *, stderr_msg: str, exit_code: int) -> None:
      print(stderr_msg, file=sys.stderr)
      output_json(payload)
      sys.exit(exit_code)
  ```
  Call it from each branch with the branch-specific payload. No behavior change; the existing tests should pass unchanged. Add a `test_emit_fallback_uses_shared_helper` if helpful (or skip — covered transitively).

### [should-fix] `build_existing_session_prompt` `os.path.relpath` ValueError on Windows / mixed-relative
- File: `scripts/qs/launchers/phases.py` (`build_existing_session_prompt`, around line 161)
- Severity: should-fix
- Source: qs-review-blind-hunter
- Description: `os.path.relpath(fix_plan_path, work_dir)` raises `ValueError` in two cases not currently covered: (a) Windows cross-drive paths (`C:\foo` vs `D:\bar`) and (b) one path absolute + other relative. The current `try/except` (if any) is too narrow or absent; if absent, the function crashes through `build_payload` → `next_step.py` as an uncaught exception. The docstring claims paths outside the worktree go through `relpath` returning `../...` form, but that doesn't apply in either of these cases.
- Proposed fix: Wrap the `relpath` call in `try/except ValueError`; on catch, fall back to using the absolute `fix_plan_path` directly in the prompt (the prompt is still functional — the user can copy the absolute path). Add `test_build_existing_session_prompt_handles_cross_drive_paths` parametrized over (a) a `Path("C:\\foo")` vs `Path("D:\\bar")` (mocked on POSIX via `monkeypatch.setattr(os.path, "relpath", ...)` raising `ValueError`) and (b) absolute fix_plan_path with relative work_dir.

### [should-fix] Inline fallback in `_opencode_cli_command` drops banner UX silently
- File: `scripts/qs/launchers/opencode.py:1376-1379` (the `except OSError: return opencode_cmd` branch in `_opencode_cli_command`)
- Severity: should-fix
- Source: qs-review-blind-hunter
- Description: When `tempfile.NamedTemporaryFile` write or `chmod` fails, the function returns the bare `opencode_cmd` — discarding the tab-title banner (`printf '\033]0;%s\007'`), the activate banner, the preload banner, and the kickoff echo. The user gets a degraded UX with no visual confirmation that the new session was set up correctly, and the user can't tell from the output that the script-file path was unavailable.
- Proposed fix: Either (a) update the docstring of `_opencode_cli_command` to explicitly document "Inline fallback drops banner UX; this is a degraded path triggered only when the temp filesystem is unwritable", OR (b) rebuild a minimal banner sequence into the inline form (e.g. `printf '...' && opencode ...`). Prefer (a) — the failure path is rare and the documentation closes the loop without adding more shell to debug. Add a test `test_opencode_cli_command_inline_fallback_documented` that reads the docstring and asserts the substring "degraded path" or similar.

### [should-fix] `--directory` whitespace surrounding non-empty value not stripped
- File: `scripts/qs/spawn_session.py:446` (`main`)
- Severity: should-fix
- Source: qs-review-edge-case-hunter
- Description: The validation `if not args.directory.strip(): parser.error(...)` rejects empty/whitespace-only directories but does not strip surrounding whitespace from otherwise-valid values. `--directory "  /tmp/wt  "` (e.g. shell-history copy-paste with a trailing space) survives the check and is sent verbatim as the `x-opencode-directory` header — the OpenCode server then fails to map the header to the worktree.
- Proposed fix: After the validation, assign back:
  ```python
  args.directory = args.directory.strip()
  ```
  Same treatment for `args.agent` (after the must-fix #2 guard lands) and `args.prompt`. `args.title` is optional; if provided, strip too. Add `test_directory_whitespace_stripped` asserting that `--directory "  /tmp/wt  "` results in the header value `"/tmp/wt"` (no surrounding whitespace).

### [should-fix] `chmod` failure leaves temp file orphaned on disk
- File: `scripts/qs/launchers/opencode.py:174` (`_opencode_cli_command`, the `script_path.chmod(0o700)` call)
- Severity: should-fix
- Source: qs-review-edge-case-hunter
- Description: `tempfile.NamedTemporaryFile(... delete=False)` creates the file; if the subsequent `chmod(0o700)` raises `OSError` (e.g. FAT32 / some NFS mounts that don't support POSIX permissions), the `except OSError` branch returns the inline fallback BUT the temp file is left behind unchmod'd. Over time these accumulate.
- Proposed fix: In the except branch, attempt a best-effort cleanup:
  ```python
  try:
      script_path.unlink(missing_ok=True)
  except OSError:
      pass
  return opencode_cmd
  ```
  Add `test_opencode_cli_command_cleans_temp_file_on_chmod_failure` — monkeypatch `Path.chmod` to raise `OSError` and assert the temp file does not exist after `_opencode_cli_command` returns.

### [should-fix] `int(None)` raises `TypeError` not `ValueError` — test gap
- File: `scripts/qs/launchers/opencode.py:130` (`_opencode_cli_command`, the `issue_int = int(issue)` line) AND `tests/qs/launchers/test_opencode_launcher.py::test_opencode_cli_command_rejects_non_integer_issue`
- Severity: should-fix
- Source: qs-review-edge-case-hunter
- Description: The existing test pins `ValueError` for shell-injection-style strings (`"42; rm -rf"`). But `int(None)` raises `TypeError: int() argument must be a string, a bytes-like object or a real number, not 'NoneType'`. Programmer error via the in-process API (`build_payload("/tmp", None, ...)`) hits this unguarded path.
- Proposed fix: Widen the exception handling. Two options:
  (a) Coerce more permissively:
  ```python
  try:
      issue_int = int(issue)
  except (TypeError, ValueError) as exc:
      raise ValueError(f"issue must be an integer, got {issue!r}") from exc
  ```
  (b) Add an upfront type check + the same `int()` call.
  Prefer (a). Then update the test to parametrize over `[None, "42; rm -rf", "abc"]` and assert all raise `ValueError` (the wrapped one). Add an additional parametrized case `True` to confirm the bool-subclass concern (`int(True) == 1`) is either accepted (current behavior) or rejected — pick one and pin it. Recommendation: accept `True`/`False` as `1`/`0` is silently wrong; add `isinstance(issue, bool)` early-reject.

### [should-fix] `--pr-number` accepts 0 and negative values

- Severity: should-fix

  if args.pr_number is not None and args.pr_number <= 0:

### [should-fix] `build_existing_session_prompt` doesn't reject whitespace-only `fix_plan_path`
- File: `scripts/qs/launchers/phases.py` (around line 161, the `if not fix_plan_path or pr_number is None: return None` guard in `build_existing_session_prompt`)
- Severity: should-fix
- Source: qs-review-edge-case-hunter

  if not fix_plan_path or not fix_plan_path.strip() or pr_number is None:

### [should-fix] `next_step.py` docstring drift on opencode strict resolution
- File: `scripts/qs/next_step.py:30` (module/function docstring describing the harness contract)
- Severity: should-fix
- Source: qs-review-edge-case-hunter
- Description: The docstring claims "Trailing/leading whitespace inside an otherwise-non-empty `--next-cmd` … IS preserved verbatim under codex/opencode; those launchers treat `--next-cmd` as free-form, so an intentional space is the user's call." This was true pre-QS-177 but the new contract has opencode resolving agents strictly (like claude/cursor) — `--next-cmd 'create-plan '` (trailing space) raises `UnknownPhaseError`. Only codex remains free-form.
- Proposed fix: Update the docstring to:
  > "Trailing/leading whitespace inside an otherwise-non-empty `--next-cmd` IS preserved verbatim under codex (the only remaining free-form harness). Claude, cursor, and opencode resolve `--next-cmd` strictly via `PHASE_TO_AGENT` and reject unknown values (including those with stray whitespace) with exit code 1."

  No test required — it's a docstring. But if convenient, add `test_opencode_rejects_trailing_whitespace_next_cmd` asserting `--harness opencode --next-cmd 'create-plan '` exits 1 with the standard unknown-phase JSON.
