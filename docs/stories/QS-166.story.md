# QS-166: Fix wrong domain in entity IDs

## Story

**As a** Quiet Solar user
**I want** my entities to use the correct platform domain prefix (e.g., `button.*`, `sensor.*`, `binary_sensor.*`)
**So that** deprecation warnings stop appearing and the integration remains compatible with HA 2027.5.0+

## Root Cause

HA logs 667 occurrences of:
> Detected that custom integration 'quiet_solar' sets an entity ID with wrong domain: 'quiet_solar.qs_pac_heat_pump_qs_device_clean_and_reset'. Expected domain is 'button'.

The cause is in `custom_components/quiet_solar/entity.py`, in `QSDeviceEntity.__init__`:
```python
self.entity_id = async_generate_entity_id(ENTITY_ID_FORMAT, name=self._attr_unique_id, hass=data_handler.hass)
```
where `ENTITY_ID_FORMAT = DOMAIN + ".{}"` (i.e., `quiet_solar.{}`). This forces all entities — regardless of their platform (button, sensor, binary_sensor) — to use the `quiet_solar` domain prefix.

HA expects entities registered on the `button` platform to have `button.*` entity IDs, etc.


## Dev Notes

- **Architecture**: Purely HA integration layer change — no `home_model/` modifications
- **Entity registry migration**: HA entity registry keys on `(platform, unique_id)`. Existing installations already have entity IDs stored in `.storage/core.entity_registry`. Removing the explicit assignment does NOT change entity IDs for existing entities — the registry preserves them. Only new installations will get the correct platform-prefixed IDs.
- **Breaking change note for release**: Mention in release notes that entity ID domain prefixes are now correct. Existing users are unaffected (entity registry preserves old IDs). New installations will see `button.*`, `sensor.*`, etc. instead of `quiet_solar.*`.
- **No other consumers**: Grep confirmed `ENTITY_ID_FORMAT` is only used in `entity.py` and defined in `const.py`. No tests reference it.
