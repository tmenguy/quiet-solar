# QS-256 — Fix bistate user-override lifecycle bugs (pool pump stuck off 18h) + make solver override-aware

- **Issue:** #256
- **Branch:** `QS_256`
- **Next phase:** `implement-task` (product code under `custom_components/quiet_solar/`)

## Problem

A false "user override → off" on a pool pump (`Piscine`,
`switch.lhk_filtration_facade`) self-perpetuated for ~18h
(2026-06-04 15:57 → 2026-06-05 10:08) despite daily filtration
constraints being pushed and the QS UI showing RUNNING. Diagnosed from
the HA log `/Users/tmenguy/Downloads/home-assistant-2.log` (local,
69 MB).

Root-cause chain (all verified in code):

1. **Phantom ack** — `probe_if_command_set`
   (`ha_model/bistate_duration.py:437-444`) compared the entity state
   against `expected_state_from_command_or_user()` — the *override*
   state, not the command's expected state. A solver `on` during an
   `off` override returned "Command already set" and was acked without
   any transport call (`home_model/load.py:609-630`), corrupting
   `current_command`, `num_on_off`, `last_state_change_time`
   (`load.py:545-557`) and — because
   `TimeBasedSimplePowerLoadConstraint.compute_value` accrues runtime
   from `current_command` (`constraints.py:2662-2668`) — the
   constraint progress shown as RUNNING in the UI.
2. **No re-execution after override end** — `launch_command`'s
   early-return when `current_command == command` (`load.py:598-601`)
   prevented the physical `on` forever once the phantom ack landed.
3. **Re-arm loop** — the 60 s post-override cooldown
   (`USER_OVERRIDE_STATE_BACK_DURATION_S`, `bistate_duration.py:32`,
   checked at `:674-683`) expired, then the persistent
   state-vs-phantom-command mismatch was classified as a NEW 8h user
   override. Repeat forever.
4. **Reset button ineffective** — `user_clean_and_reset`
   (`load.py:1644`) does not clear `external_user_initiated_state`,
   which is also persisted to storage (`load.py:826-829`) and survives
   restarts.
5. **OFF-overrides invisible to the solver** — ON-overrides push a
   real override constraint (and end cleanly via
   `ack_completed_constraint` → `reset_override_state_and_set_reset_ask_time`,
   `load.py:1154-1167`); OFF-overrides push nothing and rely purely on
   the fragile timer path.

The trigger (stale template-switch state for ~6 min after an HA
restart) is environmental, but the lifecycle must recover from any
false override.

## Design decisions (agreed interactively with the product owner)

- **D1 — Drop, never phantom-ack.** Command swallowing during an
  active override is BY DESIGN (don't fight the user). A suppressed
  command is DROPPED in `launch_command` before `running_command` is
  set: no ack, no `num_on_off`/`last_state_change_time` mutation, no
  relaunch spam (nothing for `check_commands` /
  `force_relaunch_command` to act on).
- **D2 — Truthful probe.** `probe_if_command_set` answers only "did MY
  command land": it compares against
  `expected_state_from_command(command)`. The `_or_user` variant is
  deleted.
- **D3 — OFF-overrides become a real constraint.** New
  `TimeBasedHoldOffConstraint` (zero power, CMD_IDLE window,
  wall-clock progress, `always_end_at_end_of_constraint` completion)
  so the override ends through the SAME proven path as ON overrides:
  constraint completes → `ack_completed_constraint` USER_OVERRIDE
  branch → `reset_override_state_and_set_reset_ask_time`. The solver
  sees the pinned-off window natively and the daily constraint chains
  after it. The legacy timer reset (`bistate_duration.py:590-604`)
  stays as fallback (override without constraint, e.g. restored from
  storage).
- **D4 — Override constraints always win.** Any constraint whose
  `load_info[CONSTRAINT_ORIGINATOR_KEY] ==
  CONSTRAINT_ORIGINATOR_USER_OVERRIDE` (ON and OFF) gets a new
  highest-order additive term in `LoadConstraint.score()` so it wins
  allocation ordering and same-end-time cluster dedup. Minimal
  additive term, not a new banding abstraction.
- **D5 — Causality, not timers.** A state mismatch classifies as a
  user override only if the entity state's `last_changed` is NEWER
  than the load's last real command execution
  (`last_command_execution_time`, new in-memory field; set on real
  service-call executions AND initialized at storage restore when a
  `current_command` is restored — closing the post-restart blind
  spot). Plus cooldown 60 s → 180 s, still bounded by the existing
  `min(float(USER_OVERRIDE_STATE_BACK_DURATION_S),
  (3600.0 * self.override_duration) / 2.0)` (seconds, float) at the
  check site.
- **D6 — No solver change for the boundary slot.** The partial-slot
  grab (`constraints.py:2061-2065`, `first_slot -= 1`) may stamp a
  command up to one slot before `current_start_of_constraint`; the
  execution-layer drop guard (D1) makes it harmless and a re-solve
  after override end re-issues it. Test-only verification (AC12).
- **D7 — Reset paths (Fix-1 investigation outcome).**
  `user_clean_and_reset` clears ALL override fields (the issue named
  `external_user_initiated_state`; the full set is cleared because the
  acceptance is "the reset button breaks ANY override loop" and a
  half-cleared ask-time would leave the UI in ASKED_FOR_RESET).
  Storage restore drops an already-expired override. `reset()` and
  `qs_enable_device` are deliberately UNTOUCHED: `reset()` fires
  during car-plug cycles
  (`clean_constraints_for_load_param_and_if_same_key_same_value_info`,
  `load.py:1136`) and the enable toggle — paths where a legitimate
  user override must survive. `async_load_constraints_from_storage`
  untouched: constraints and override fields restore independently;
  the staleness guard (T10) covers the poison case.
- **T5 is additive only** — `execute_command`'s override interception
  block (`bistate_duration.py:446-468`) stays byte-for-byte identical
  (it is the last line of defense on the `force_relaunch_command`
  path, which bypasses `launch_command`'s drop point); it only gains a
  comment. The suppression hook must replicate its nuance: when the
  override constraint is degraded to non-mandatory and the command is
  off/idle, the command is ALLOWED through (`:449-454`).

## Acceptance criteria

- **AC1 (user reset breaks any loop)** — Given an active user override
  on a bistate load, When `user_clean_and_reset` runs, Then
  `external_user_initiated_state`, `external_user_initiated_state_time`,
  `asked_for_reset_user_initiated_state_time`,
  `asked_for_reset_user_initiated_state_time_first_cmd_reset_done` and
  `last_command_execution_time` are all None, constraints are cleared,
  CMD_IDLE is launched, and no override is re-detected on the next
  check cycle.
- **AC2 (expired override not restored)** — Given stored device info
  whose `external_user_initiated_state_time` (tz-aware UTC, written
  via isoformat and parsed via `fromisoformat`) is older than
  `override_duration` hours at restore time
  (`datetime.now(pytz.UTC)`), When the load restores from storage,
  Then the override fields are dropped with an info log; a still-valid
  stored override is kept.
- **AC3 (no phantom ack)** — Given an active OFF override (state
  'off'), When `launch_command(CMD_ON)` is called, Then no service
  call occurs, `current_command` is unchanged, `num_on_off` unchanged,
  `last_state_change_time` unchanged, `running_command` is None and a
  suppression info log is emitted.
- **AC4 (truthful probe)** — Given override 'off', command CMD_ON,
  entity state 'off', When `probe_if_command_set` runs, Then it
  returns False.
- **AC5 (post-override re-execution)** — Given an OFF override whose
  hold-off constraint completes, When the next solve + dispatch runs,
  Then the daily constraint's CMD_ON physically executes (transport
  call observed), is acked, and `num_on_off` increments by exactly 1.
- **AC6 (causality guard)** — Given a real command execution at T and
  a mismatching entity state with `last_changed <= T`, Then NO
  override is classified (stale state). Given `last_changed > T`, Then
  the override IS classified. Given `last_command_execution_time is
  None` (no execution AND nothing restored), Then classification
  proceeds purely on state mismatch + cooldown as before this story.
- **AC7 (cooldown)** — `USER_OVERRIDE_STATE_BACK_DURATION_S == 180`;
  effective cooldown is `min(180.0, override_h*3600/2)` seconds;
  boundary test for a small `override_duration` (e.g. 0.05 h → 90 s).
- **AC8 (hold-off pushed)** — Given a user-override-to-idle detection,
  Then a `TimeBasedHoldOffConstraint` is pushed with
  `load_param=override_state`, originator
  `CONSTRAINT_ORIGINATOR_USER_OVERRIDE`, `from_user=True`,
  `type=CONSTRAINT_TYPE_MANDATORY_END_TIME`, `start=now`,
  `end=now+override_duration`, `target=override_duration_s`,
  `power=0.0`. Given the computed `end <= now` (already expired at
  push time), Then no constraint is pushed and the override is reset
  directly.
- **AC9 (solver integration)** — Given a load with an active hold-off
  constraint and a daily constraint chained after it, When
  `PeriodSolver.solve(with_self_test=True)` runs, Then the hold-off
  window allocates zero power and CMD_IDLE commands, the daily
  constraint is allocated after the override end (boundary partial
  slot excepted, see AC12), and the self-test passes.
- **AC10 (constraint-driven override end)** — Given a hold-off
  constraint reaching its end, When `update_live_constraints` acks it,
  Then the existing USER_OVERRIDE branch of `ack_completed_constraint`
  resets the override, arms the cooldown, and a re-solve is forced.
- **AC11 (score dominance)** — Given any USER_OVERRIDE-originated
  constraint, `score()` strictly exceeds the MAXIMUM possible
  non-override score (test compares a worst-case override constraint
  against a max-energy, type-9, `from_user=True` non-override
  constraint); same-end-time cluster dedup in `set_live_constraints`
  keeps the override constraint.
- **AC12 (fix-5 verification, test-only)** — Given an override ending
  mid-slot and a daily constraint starting then, When the solver emits
  a boundary-slot command stamped before the constraint start, Then
  dispatch-time `launch_command` drops it (no execution, no ack,
  assertions on `running_command`/counters at the drop site) and
  after the override ends a re-solve re-issues the command, which
  executes.
- **AC13 (flagship June 4→5 replay)** — A freezegun-driven simulation
  of the normative timeline above: (a) restart + restored
  `current_command` + stale facade (`last_changed` ≤ restore anchor) →
  NO false override pair; (b) then a genuine OFF override (fresh
  `last_changed`) → hold-off constraint pushed → midnight daily
  constraint → no phantom ack, no re-arm loop, the morning `on`
  physically executes; (c) NET `num_on_off` across the whole replay
  equals the exact number of real physical transitions asserted as an
  integer; (d) UI metrics never show RUNNING progress while the pump
  is physically off.
- **AC14 (timer fallback)** — An override WITHOUT a constraint (e.g.
  restored from storage without its constraint) still expires via the
  legacy timer path; a "both mechanisms armed" scenario converges with
  whichever fires first making the other a no-op.
- **AC15 (gate)** — 100% coverage; full quality gate green
  (`python scripts/qs/quality_gate.py`); agent docs updated (T13).

## Tasks

### T1 — Score dominance for override constraints (`home_model/constraints.py`)

In `LoadConstraint.score()` (`:316-338`) add a minimal additive term.
Exact spec (dominance proven: max non-override total ≈
`1e6 + 1e12 + 9e12 + 1e13 < 2.1e13`, type values capped at 9 per
`const.py:340-344`):

```python
override_score = 0.0
if self.load_info is not None and self.load_info.get(CONSTRAINT_ORIGINATOR_KEY, None) == CONSTRAINT_ORIGINATOR_USER_OVERRIDE:
    override_score = 1.0
# new highest-order term, 5x margin above the max possible sum of all others:
+ energy_score_span * reserved_load_score_span * type_score_span * type_score_span * override_score   # == 1e14
```

Import `CONSTRAINT_ORIGINATOR_KEY` / `CONSTRAINT_ORIGINATOR_USER_OVERRIDE`
from `..const` (pure-Python module, boundary-safe).

### T2 — `TimeBasedHoldOffConstraint` (`home_model/constraints.py`)

New class `TimeBasedHoldOffConstraint(TimeBasedSimplePowerLoadConstraint)`
(must live in `constraints.py` so the name-based
`new_from_saved_dict` factory resolves it; serialization round-trips
via the inherited `to_dict`/`from_dict_to_kwargs` — `power_steps` with
a zero-consign command, no new fields):

- `__init__`: default `power=0.0` when not given.
- `compute_value(time)`: wall-clock accrual
  `(time - self.last_value_update).total_seconds() + self.current_value`,
  UNCONDITIONAL — override needed because the base gates on
  `current_command not off_or_idle` (`:2662-2668`) and the hold-off
  window runs on CMD_IDLE.
- `compute_best_period_repartition(...)`: trivial fixed output
  matching the base 9-tuple exactly:
  `(out_constraint, final_ret, out_commands, out_power, first_slot,
  last_slot, min_idx_with_energy_impact, max_idx_with_energy_impact,
  remaining_additional_available_energy_to_deplete)` where
  `out_constraint = self.shallow_copy_for_budget_delta_quantity(0.0)`,
  `final_ret=True`, `out_commands` = `copy_command(CMD_IDLE)` for
  every slot in `[first_slot..last_slot]` (None outside), `out_power`
  = zeros, `min_idx = max_idx = -1` (the base's no-energy-impact
  sentinels, `:2519-2524`), and the deplete budget passed through
  unchanged. `first_slot`/`last_slot` bisected from
  `current_start_of_constraint`/`end_of_constraint` like the base.
- `adapt_repartition`: no-op returning "no change" (defensive — must
  never be shaved by `_constraints_delta`; it is mandatory anyway).
- Inherits `always_end_at_end_of_constraint=True` from the TimeBased
  parent → met at window end → acked → override reset (D3).
