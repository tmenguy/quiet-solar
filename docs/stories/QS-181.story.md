# Story QS-181: Speed up `quality_gate.py` runtime

issue: 181
branch: "QS_181"
Status: ready-for-dev

## Story

**As** a Quiet Solar developer running `python scripts/qs/quality_gate.py`
during the inner dev loop (and as the CI `Tests (100% Coverage)` job),
**I want** the quality gate to complete in roughly a third (or less) of
its current wall-clock time,
**so that** running the gate stays cheap enough to invoke on every
implement-phase iteration instead of being deferred.

## Context

### Measured baseline

On the dev machine, a clean `python scripts/qs/quality_gate.py --full`
run takes **~5 min** wall-clock (7 m 36 s observed under CPU contention
with another pytest stealing half the cores). CI's `Tests (100% Coverage)`
job is similarly long.

Where the time goes:

| Gate           | Wall time           | Share          |
| -------------- | ------------------- | -------------- |
| `ruff format`  | ~0.05 s             | negligible     |
| `ruff lint`    | ~0.05 s             | negligible     |
| `mypy`         | ~0.28 s             | negligible     |
| `translations` | small (sub-second)  | negligible     |
| `pytest --cov` | **multi-minute**    | **>95%**       |

The pytest+coverage step is essentially the entire cost. 5066 tests run
single-process, sequentially, with `coverage.py`'s default `settrace`
core (which adds 1.3–2× overhead per test).

### What's available but unused

- `pytest-xdist 3.8.0` is installed in the local venv but **not declared
  in `requirements_test.txt`** and **never invoked** by `check_pytest`.
- `coverage 7.13.5` on Python 3.14 — `COVERAGE_CORE=sysmon` (the new
  `sys.monitoring`-based core, faster than `settrace`) is supported but
  not enabled.
- `pytest.ini`'s default `addopts` includes `--cov-report=html` — every
  run writes a full HTML report to `htmlcov/` that nothing in the QG or
  CI path consumes.
- `main()` runs the 5 gates **sequentially**; the 4 cheap gates
  (ruff_format, ruff_lint, mypy, translations) could run concurrently
  with each other while pytest runs serially after them.
- `_stream_pytest`'s progress reporter is **broken under `--cov`**: it
  logged `pytest: done (0/5066) | passed=0 failed=0 errors=0` for an
  all-pass run because its `.FEsxX` regex doesn't match what pytest
  emits under coverage.

### What is in scope (trimmed after adversarial review — see notes)

- **T1** — invoke pytest with `-n auto` (configurable via
  `QS_QG_PYTEST_WORKERS`), set `COVERAGE_CORE=sysmon` in subprocess env,
  append `--cov-report=` (empty) to override `pytest.ini`'s html.
- **T2** — run the 4 cheap gates concurrently in a `ThreadPoolExecutor`;
  run pytest **after** them (serially) so xdist owns the full CPU budget
  without oversubscription.
- **T3** — rewrite `_stream_pytest`'s progress parser to be
  output-mode-agnostic (rely on dotted output + final summary line, not
  cmd-flag sniffing).
- **T4** — remove the single line `--cov-report=html` from
  `pytest.ini`'s `addopts`.
- **T9** — in `.github/workflows/pr-quality.yml`'s `test` job, swap the
  pytest invocation for `pytest -n auto --cov=… --cov-report=term-missing
  --cov-fail-under=100` with `env: COVERAGE_CORE: sysmon`. Keep `lint`,
  `typecheck`, `hacs-validate` as separate jobs (no double-running).
- **T10** — capture before/after wall-clock numbers during implement;
  paste into the PR body. Add a smoke test asserting `--full` exits 0
  and coverage ≥100% on merged data. No timing test (would be flaky).
- **B1** — add `pytest-xdist>=3.5.0` to `requirements_test.txt`.

### What is explicitly **out** of scope (cut during plan review)

These were considered and trimmed by reviewer consensus:

- **Splitting `test_home_extended_coverage.py` and
  `test_charger_coverage_deep.py`** into smaller files (~16 new files,
  ~12.5 K lines of churn). Reasons: xdist 3.x's `--dist loadscope`
  balances across test classes for free; concrete-planner found 3
  separate blockers (existing `test_home_misc.py` would collide; a
  duplicate `TestIntermediateTargetGreaterThanTarget` class lives at
  `:5269` and `:5866`; module-level helpers `_inject_sensor_value` (49
  uses) and `_make_lazy_states` (6 uses) cross class boundaries that
  would land in different output files).
- **A new "focused" scope mode** in `_detect_scope` that maps changed
  production modules to importing test files. Reasons: scope-guardian
  flagged it as a feature rather than a speedup; dev-proxy flagged that
  it conflicts with the 100% coverage mandate unless gated very
  explicitly; concrete-planner found that
  `home_model/constraints.py:319,325` uses `importlib.import_module`
  which defeats static import-grep.
- **Content-hash cache** replacing the existing commit-hash cache.
  Reasons: marginal value (commit-hash already invalidates correctly in
  the common case); hashing tens of MB of test files on every
  cache-check could exceed any reasonable "cache hit ≤2 s" budget.
- **Session-scoping `current_time` and `mock_solcast_data`** in
  `tests/conftest.py`. Reasons: `current_time` returns an immutable
  `datetime` — session vs. function scope is indistinguishable;
  `mock_solcast_data` has **zero consumers in the repo today** so
  changing its scope is a no-op.
- **Invoking `quality_gate.py --full` from CI's `test` job**. Reasons:
  the CI workflow has separate `lint` and `typecheck` jobs that already
  run ruff and mypy, so invoking the full gate from `test` would
  duplicate that work. The replacement (raw `pytest -n auto --cov`) is
  simpler and gets the same speedups.

## Acceptance Criteria

1. **Given** a developer runs `python scripts/qs/quality_gate.py` (with
   or without `--full`),
   **When** the pytest gate executes,
   **Then** pytest is invoked with `-n <workers>` where `<workers>`
   defaults to `"auto"` and is overridable by `QS_QG_PYTEST_WORKERS`,
   **And** if `pytest-xdist` is not importable from the venv, the gate
   falls back to a single-process invocation with a one-line
   `[pytest] xdist not available, running single-process` warning to
   stderr,
   **And** the `--collect-only` count subprocess at the top of
   `_stream_pytest` is invoked **without** the `-n` flag (collection is
   inherently single-process).

2. **Given** the pytest gate is invoked,
   **When** the subprocess is launched,
   **Then** its environment includes `COVERAGE_CORE=sysmon` in addition
   to the inherited parent environment,
   **And** the `--collect-only` count subprocess does **not** set
   `COVERAGE_CORE` (coverage is not active during collection),
   **And** one-off implementer validation (see T10) confirms that the
   set of covered lines under `sysmon` is identical to the set under
   the legacy `settrace` core (diff of `--cov-report=term-missing`
   output is empty).

3. **Given** `main()` runs without `--fix`,
   **When** the gates execute,
   **Then** `check_ruff_format`, `check_ruff_lint`, `check_mypy`, and
   `check_translations` run **concurrently** via
   `concurrent.futures.ThreadPoolExecutor(max_workers=4)`,
   **And** `check_pytest` runs **after** the cheap gates complete (not
   in parallel with them, to avoid CPU oversubscription with xdist),
   **And** each gate's stderr output is line-prefixed with `[<gate>]`
   via a `_emit(name, line)` helper using `print(..., file=sys.stderr)`
   (not `_LOGGER`),
   **And** the existing per-gate self-prefixes (e.g.
   `"  ruff lint: running...\n"`) are removed so prefixing happens
   exactly once,
   **And** if any gate fails, the script exits with code 1 and all
   failed gate names are listed.

4. **Given** the pytest gate runs,
   **When** the pytest command is built,
   **Then** the cmd includes `--cov-report=` (empty value) to override
   `pytest.ini`'s remaining `--cov-report=html` default,
   **And** the line `--cov-report=html` is removed from
   `pytest.ini`'s `addopts` (the existing `pytest.ini` line 18 is
   deleted; `--cov-report=term-missing` on the prior line stays),
   **And** no `htmlcov/` directory is written by a QG run.

5. **Given** `_stream_pytest` parses pytest's stdout,
   **When** any of the three current output shapes appears (sequential
   `-q`, sequential `-q --cov`, xdist `-q`),
   **Then** the parser correctly counts passed / failed / error tests
   by counting `.` / `F` / `E` characters in dotted progress lines AND
   by parsing the final summary line `<N> passed, <M> failed, …` as
   the authoritative count,
   **And** the final reported counts equal the pytest exit-code-derived
   pass/fail outcome (no more `0/5066` regression),
   **And** unit tests in `tests/test_quality_gate.py` exercise the
   parser with three captured-output fixtures (one per mode) committed
   under `tests/fixtures/quality_gate/`.

6. **Given** `requirements_test.txt`,
   **When** the change lands,
   **Then** it contains a line `pytest-xdist>=3.5.0` (so CI's
   `pip install -r requirements_test.txt` step installs xdist),
   **And** local venvs without `pytest-xdist` can still run the gate
   serially via the AC1 fallback.

7. **Given** the CI workflow `.github/workflows/pr-quality.yml`,
   **When** a PR opens or updates,
   **Then** the `test` job runs `pytest tests/ -n auto
   --cov=custom_components/quiet_solar --cov-report=term-missing
   --cov-fail-under=100` (or the equivalent with explicit
   `--numprocesses auto`),
   **And** the job env includes `COVERAGE_CORE: sysmon`,
   **And** `lint`, `typecheck`, and `hacs-validate` jobs are unchanged
   (no duplication of ruff/mypy work).

8. **Given** the existing `tests/test_quality_gate.py` suite,
   **When** it runs after this change,
   **Then** every existing test still passes,
   **And** existing CLI flags (`--cache`, `--no-cache`, `--full`,
   `--json`, `--fix`) behave identically to before.

9. **Given** the implement phase records before/after wall-clock for
   `python scripts/qs/quality_gate.py --full` on the same machine in
   the same shell session,
   **When** the implementer pastes those numbers into the PR
   description,
   **Then** the "after" number demonstrates a measurable wall-clock
   reduction vs. the "before" baseline. The target is ≥3× on a quiet
   shell session (per Dev Notes → "Performance expectation"), softened
   to ≥2× on a CPU-contended dev box where the machine cannot be
   quiesced (multiple parallel agents, IDE indexing, etc.); the
   ≥3× target must then be re-verified on the CI runner as a
   follow-up. This is the success criterion in lieu of a brittle
   absolute SLA,
   **And** the implementer also confirms `--full` exits 0 with coverage
   ≥100% on the merged xdist coverage data, and pastes that summary
   line into the PR description.

   **Amendment (review-fix #01, M3):** the original "≥3× unconditional"
   bar was not satisfied on this dev box (2.8× in the PR baseline →

10. **Given** a smoke test added to `tests/test_quality_gate.py`,

    **Then** it invokes `_stream_pytest` (or `check_pytest`) with all

- Add helper `_emit(name: str, line: str) -> None` that writes

  ```python
      futures = {pool.submit(g, fix=args.fix) if g in (check_ruff_format, check_ruff_lint) else pool.submit(g): g for g in cheap}
      cheap_results = [f.result() for f in as_completed(futures)]
  results.extend(sorted(cheap_results, key=lambda r: ["ruff_format","ruff_lint","mypy","translations"].index(r["name"])))
  results.append(check_pytest())
  ```
  (Exact ordering preserved for human-readable output.)

  runs there; no benefit from a thread pool).

Changes:
- In `_stream_pytest`, the line-classification block becomes:
  ```python
  PROGRESS_CHARS = ".FEsxX"
  SUMMARY_RE = re.compile(r"^=+ (.*?) =+$")
  COUNT_RE = re.compile(r"(\d+) (passed|failed|error|errors|skipped)")
  ```
  The parser:
  1. For each line, strip trailing `[ NN%]` percentage markers and any
     leading `[gwN]` xdist-worker prefix.
  2. If the cleaned line is non-empty and consists entirely of
     `PROGRESS_CHARS`, count `.` / `F` / `E` / `s` / `x` / `X`.
  3. Track the most recent summary line (matching `SUMMARY_RE`).
  4. After `proc.wait()`, re-parse the summary line via `COUNT_RE` —
     these counts are authoritative; the per-character tally is for
     mid-run progress only.
- The final stderr line emits the authoritative counts from the
  summary regex, not the running tally.
- Detection by "is `-n` in cmd?" is **removed** — the parser handles
  all modes uniformly.

### Architecture constraints

  `project-context.md` ("no blocking calls in async code") don't apply.

### Performance expectation (no hard SLA)

- xdist `-n auto` on an 8-core dev machine typically delivers a 3–5×
  speedup on test suites this size (Amdahl's law caps it; fixture
  setup is sequentialish per worker).
- `COVERAGE_CORE=sysmon` typically removes another 25–40% of the
  remaining coverage tax.
- Combined expected wall-clock reduction: **3–6× vs. baseline**. AC9
  encodes "at least 3×" as the success bar.
- T2's parallelization of the cheap gates saves ~0.5 s — small but
  free (no risk, near-zero diff).

## Adversarial Review Notes

Four parallel reviewers examined the original (broader) plan draft:
`qs-plan-critic`, `qs-plan-concrete-planner`, `qs-plan-dev-proxy`,
`qs-plan-scope-guardian`. Their findings drove substantial trimming.

### Findings incorporated (decisions taken)

**Cuts (entirely removed from scope):**

- **Splitting `test_home_extended_coverage.py` and
  `test_charger_coverage_deep.py` into ~16 new test files**
  (original T7). All four reviewers flagged it. Concrete-planner
  identified three hard blockers: `tests/ha_tests/test_home_misc.py`
  already exists, `TestIntermediateTargetGreaterThanTarget` is
  duplicated at `test_charger_coverage_deep.py:5269` and `:5866`, and
  module-level helpers `_inject_sensor_value` (49 uses) and
  `_make_lazy_states` (6 uses) cross the planned class-to-file
  boundaries. xdist's `--dist loadscope` (default for class-organized
  suites) handles load balancing without the churn.
- **"Focused" scope mode that maps changed production modules to test
  files** (original T6). Scope-guardian: feature creep, not a speedup.
  Dev-proxy: conflicts with the 100% coverage mandate without explicit
  gating. Concrete-planner: dynamic imports in
  `home_model/constraints.py:319,325` defeat static import-grep.
- **Session-scoping `current_time` and `mock_solcast_data`** in
  `tests/conftest.py` (original T8). Concrete-planner proved it's a
  no-op: `current_time` returns an immutable `datetime` (re-creation
  is free); `mock_solcast_data` has no consumers in the repo today.

**Deferred (filed mentally for follow-up if measurement justifies):**

- **Content-hash cache** replacing the commit-hash cache (original
  T5). Critic: hashing tens of MB on every cache check could exceed
  any reasonable "cache hit ≤2 s" budget. Scope-guardian: marginal
  value over commit-hash. Revisit if commit-hash misses are observed
  to be a real pain point.

**Reworks (kept but redesigned):**

- **T2 oversubscription** (original AC3). Critic and dev-proxy both
  flagged that running 5 gates concurrently while pytest holds `-n
  auto` workers is CPU oversubscription. The story now runs the 4
  cheap gates concurrently and pytest **after** them, serially.
- **T9 CI consolidation** (original AC9). Concrete-planner: invoking
  `quality_gate.py --full` from the `test` job would double-run ruff
  and mypy (they have dedicated jobs). The story now keeps separate CI
  jobs and just swaps the pytest cmd in `test`.
- **T3 progress reporter mode detection** (original AC10). Critic:
  AC10 allowed "disabled, just print 'xdist: progress unavailable'"
  as a way out, letting the task ship a no-op. Concrete-planner +
  critic both pointed out that detection-by-cmd-flag is fragile.
  Rewritten to a single output-mode-agnostic parser with three
  committed fixture-based parser tests.
- **AC2 coverage-equivalence wording** (original "±0 lines"). Critic:
  brittle, undefined. Rewritten to "diff of `--cov-report=term-missing`
  output is empty" with an explicit rollback (`unset COVERAGE_CORE`).
- **`_pytest_workers()` contract** (original T1, vague). Dev-proxy +
  concrete-planner: needs explicit fallback, env value semantics, and
  must NOT apply to the `--collect-only` count subprocess.

**Added (blocker surfaced by reviewers):**

- **B1: declare `pytest-xdist` in `requirements_test.txt`**.
  Concrete-planner: the package is installed in the local venv but
  not declared, so CI would fail because `pip install -r
  requirements_test.txt` doesn't pull it.
- **Smoke test** in `tests/test_quality_gate.py` asserting the
  pytest cmd contains `-n` and the env contains `COVERAGE_CORE=sysmon`
  (T10's commit-time regression guard). Earlier "not a committed test"
  framing wasn't sufficient per dev-proxy and scope-guardian.

### Decisions made

- **Drop the absolute "≤90 s cold / ≤2 s cache hit" SLA.** Both critic
  and scope-guardian flagged it as unsubstantiated. Replaced with
  "demonstrate at least 3× wall-clock reduction vs. baseline; numbers
  recorded in PR body" (AC9).
- **Keep T3 (progress reporter) in this story** rather than splitting
  it off (scope-guardian's suggestion). Rationale: the broken
  reporter currently masks failures during the implement phase
  (a `0/5066` line that should say `5066/5066`), so it's tightly
  coupled to the pytest-cmd work in T1. The marginal additional diff
  is small and tested.

### Known risks accepted

- **xdist + `pytest-homeassistant-custom-component` compatibility**:
  the plugin spins up `hass` per-test, and only one test file uses
  `freezegun`, so the risk surface is small but non-zero. Mitigation:
  T10's one-off coverage-equivalence check will also surface any
  parallelization-related test failures, and the rollback (drop `-n`
  from the cmd, or set `QS_QG_PYTEST_WORKERS=0`) is trivial.
- **Coverage diff under `sysmon` may be non-zero on edge cases**
  (decorators, branch boundaries). Mitigation: T10 explicit step
  blocks merge until the diff is investigated; rollback is `unset
  COVERAGE_CORE`.
- **GitHub-hosted runner core count**: `-n auto` on a 2-core runner
  resolves to 2 workers, on a 4-core runner to 4. Both improve over
  serial. Not pinning a fixed worker count to keep the change minimal
  — concrete-planner suggested pinning, but `auto` is the path of
  least surprise and easiest to revisit.
- **Removing `--cov-report=html` from `pytest.ini`**: no consumer in
  the repo or CI references `htmlcov/`. Devs who want the HTML report
  during ad-hoc local runs can pass `--cov-report=html` on the command
  line.
