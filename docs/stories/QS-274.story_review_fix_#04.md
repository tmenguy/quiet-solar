## Findings to fix

### [should-fix] Update person-trip-prediction.md to the shipped origin-string behavior
- File: `docs/agents/concepts/person-trip-prediction.md` (~lines 792-798)
- Severity: should-fix
- Source: blind-hunter
- Description: The doc table still describes the car-card origin string as
  `"Forecasted from <Person>: …"` and the no-person case as `"No proper Forecast"`.
  After fix #03 the code delegates to `get_car_person_readable_forecast_mileage()`,
  which produces `"<name>: <forecast>"` (e.g. `"Magali: 30km 14:30"`,
  `"Magali: No forecast"`) and `"No forecasted person"` when no person is attached
  — no `"Forecasted from"` prefix. The in-diff tests contradict the in-diff docs.
- Proposed fix: Update the table/prose to match the shipped strings: person
  attached → `"<name>: <forecast>"`; person attached but empty forecast →
  `"<name>: No forecast"`; no person → `"No forecasted person"`. Remove the
  `"Forecasted from …"` and `"No proper Forecast"` descriptions. Re-stamp
  `last_verified` if the doc convention requires it.

### [should-fix] Correct the PERSON_NO_FORECAST_STRING comment in const.py
- File: `custom_components/quiet_solar/const.py:6-10` (the
  `PERSON_NO_FORECAST_STRING` definition comment)
- Severity: should-fix
- Source: blind-hunter
- Description: The comment claims the constant is "shared so the car-card origin
  line can detect the empty-forecast case," but after fix #03 `car.py` no longer
  imports or references it — it is only used in `person.py` (where the sentinel
  is produced by `get_forecast_readable_string()`).
- Proposed fix: Rewrite the comment to accurately describe the constant's actual
  role (the sentinel string returned by `QSPerson.get_forecast_readable_string()`
  when there is no mileage/leave-time prediction). Do not remove the constant —
  it is still used by person.py.
