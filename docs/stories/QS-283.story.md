# QS-283 â€” Make `--impacted` reliable AND always-used by implement-task

- **Issue:** #283
- **Branch:** QS_283
- **Status:** DRAFT (story v3) â€” 2 review rounds folded in; all findings dispositioned

## Problem

Issue #283 reports: the implement-task agents run the **full** quality
gate (`python scripts/qs/quality_gate.py`, no flag) instead of the fast
`--impacted` inner loop QS-276 delivered, defeating the speedup.

Investigation shows the issue's framing (prompt ambiguity / agent
default) is the **symptom, not the disease**. Two compounding causes:

### Cause 1 â€” `--impacted` is unreliable, so agents (and humans) abandon it

Observed live in the QS_281 worktree (the user's transcript):

- A new accessor + its new test were added in one edit. `--impacted`
  reported the accessor's lines **uncovered** even though the new test
  covers them. `--seed-testmon` "found 0 changed" and refused to re-run
  the test â†’ a **false FAIL with no working recovery**.
- Manually purging `.testmondata` left `0 tests` selected, "yet partial
  coverage appears (stale `.coverage`)".
- The user suspected the data files lived in the main checkout, not the
  worktree.

Forensics on the live worktree confirm the mechanism:

- **10 orphaned `.coverage.<host>.pid*.* ` xdist shards, no combined
  `.coverage`** â†’ killed/crashed `--impacted` runs leave per-worker
  coverage shards that never get combined; the next `--cov-append` run
  folds those stale fragments into `coverage.xml` ("partial coverage
  appears").
- **A 4.3 MB un-checkpointed `.testmondata-wal`** â†’ killed runs leave a
  dirty WAL; reopening replays stale state.
- **testmon âŠ• coverage desync.** `--impacted` runs `pytest --testmon`,
  which **mutates `.testmondata` as a side effect**. If a run is killed
  after testmon records a test's fingerprint but before pytest-cov
  combines that test's coverage, the two stores diverge: testmon thinks
  the (covering) test is "seen/unchanged" â†’ deselects it â†’ its lines
  never get re-covered â†’ permanent false FAIL. `--seed-testmon` against
  the already-advanced baseline finds "0 changed" â†’ cannot recover. A
  **deadlock** only a manual `rm` could (sometimes) clear.
- The data files **do** live in the worktree (`REPO_ROOT` derives from
  `quality_gate.py`'s own resolved path; pytest runs with
  `cwd=REPO_ROOT`). The user's `0 tests after purge` came from the zsh
  glob `.testmondata.*` failing to match, leaving orphaned `-wal`/`-shm`
  sidecars that testmon reopened into an empty DB. The tool's own purge
  removes sidecars; manual recovery did not â€” and people only reach for
  manual recovery because the tool confused them.

### Cause 2 â€” the prompts permit an escape to the full gate

Even with `--impacted` working, the agent prompts let an agent bail out:

- `.claude/agents/qs-implement-task.md:90` / `qs-implement-setup-task.md:80`
  say **"Default to the impacted gate"** â€” "default" is soft.
- `qs-implement-task.md:100-102` explicitly licenses the full gate
  **"when you suspect coverage lost in *unchanged* code"** â€” an agent
  can always rationalise this, especially right after `--impacted` fails
  confusingly. **The same clause lives in `project-rules.md` L36-37**
  ("run locally on explicit request **or when you suspect coverage lost
  in UNCHANGED code**") and must be reconciled too.
- `qs-implement-task.md:108-109` "fix autonomously and re-run" nudges the
  agent to reach for the full gate as a diagnostic â€” exactly the QS_281
  transcript.

Only the three `qs-implement-task.md` copies carry the escape-clause +
"fix autonomously" wording; the three `qs-implement-setup-task.md` copies
have only the soft "Default to â€¦".

## Goal

1. **Reliability (must work):** `--impacted` is trustworthy under real
   conditions â€” killed runs, orphaned shards, dirty WAL, baseline drift â€”
   recovering **automatically** with **no manual file deletion** ever
   required. No false FAIL, no false PASS.
2. **Enforcement (must always run):** the implement agents run
   `--impacted` before every commit/PR and **never** substitute the full
   gate locally. The only local full-gate run is an explicit user
   request. Enforcement holds across all three harnesses.

Scope is **Option B** (harden the QS-278 `--cov-append` accumulation
model + enforce) â€” *not* a redesign of the accumulation model itself.

## Non-goals

1. **Redesigning the inner-loop coverage model** (dropping `--cov-append`
   accumulation for a self-contained per-run model) â€” Option C; a
   separate future issue if ever wanted. `--cov-append` stays load-bearing
   for the common "edit one file / no-op re-run" loop (testmon selects 0
   or a small subset; accumulation keeps changed-line coverage correct).
   *Acknowledged tension:* A4's recovery action is heavyweight (a full
   select-all coverage pass) precisely **because** `--cov-append` is kept â€”
   a cheaper recovery is impossible without the Option-C redesign. That
   cost is the Option-B boundary, paid only on the rare recovery path.
2. **Full-gate wall-clock optimization** â€” out of scope (QS-276 non-goal).
3. **CI changes** â€” CI already enforces whole-repo 100% on every PR; the
   safety split is unchanged. No `pr-quality.yml` edits.
4. **A hard pre-commit guardrail** â€” see "Cut from scope" + "Resolved
   decisions" (enforcement is prose + CI backstop, by decision).

## Design

All reliability fixes live in `scripts/qs/quality_gate.py`. The existing
subprocess seam they patch is `_stream_pytest(cmd)` (the pytest passes)
and `_run(cmd, timeout=â€¦)` (git / diff-cover), both already used by the
current `check_impacted` tests. Module path constants `REPO_ROOT`,
`SRC_DIR`, `TESTS_DIR`, `COVERAGE_DATA`, `COVERAGE_XML`, `TESTMON_DATA`
are resolved at import.

**Shared rebuild helper (new) â€” `_rebuild_testmon_baseline()`.** Purges
`.testmondata` (+ `-wal`/`-shm` via `_purge_testmon_db`) **and** clears
accumulated coverage (`_reset_coverage_data()`, which already unlinks the
combined `.coverage` **and** globs/unlinks its `.coverage.*` shards). So a
caller of this helper starts the next testmon pass with a clean DB **and**
zero coverage fragments. Used by **both** A3 (`seed_testmon`) and A4
(self-heal) so the rebuild â€” including shard hygiene â€” is spelled once.

**Definition â€” "select-all".** After a purge, testmon has no DB, so
`pytest --testmon` selects the **entire collected suite minus the
`--ignore`d `tests/test_quality_gate.py` self-tests**. As a literal
predicate: selected count == (total collected âˆ’ ignored self-tests).
Those self-tests never cover `custom_components/quiet_solar` production
lines, so excluding them leaves production-line coverage complete.

### A. Reliability â€” make `--impacted` self-coherent and self-healing

**A1 â€” Orphaned coverage-shard hygiene.** New `_clean_orphan_cov_shards()`
removes only pre-existing `.coverage.*` **shards** (never the combined
`.coverage`), reusing the **same glob expression** as `_reset_coverage_data`
(`COVERAGE_DATA.parent.glob(COVERAGE_DATA.name + ".*")`) so the two helpers
never drift into divergent shard-matching rules â€” they differ only in
whether the primary `.coverage` is also unlinked. Called at the **top of
`check_impacted()`** on the normal incremental path, **before** the
testmon/cov pytest subprocess spawns â€” so it can never reap live worker
shards mid-run, and the combined `.coverage` accumulation survives for the
warm-baseline case. (`seed_testmon` and the A4 retry do **not** call A1
separately â€” they go through `_rebuild_testmon_baseline()`, whose
`_reset_coverage_data()` already removes shards.) Single-writer assumption
(at most one `--impacted` per worktree at a time) is the same one already
documented for `COVERAGE_XML`. Unit test: a seeded `.coverage.host.pid`
shard is removed while a seeded combined `.coverage` survives.

**A2 â€” Sidecar-aware testmon DB hygiene.** In `_ensure_testmon_db_safe()`,
replace the current early `return` on absent `.testmondata` with a branch
that, when the primary is absent but `.testmondata-wal`/`-shm` linger,
unlinks the sidecars (reusing the suffix loop from `_purge_testmon_db`)
then returns â€” so testmon cannot reopen an empty DB against a stale WAL and
select `0 tests`. **This is a tool-reachable state, not just a manual-`rm`
artifact:** `_purge_testmon_db` unlinks the primary before its sidecars, so
a run killed mid-purge leaves exactly absent-primary + orphan-sidecar.
Unit test: `test_ensure_safe_removes_orphan_sidecars_when_primary_absent`.

**A3 â€” `--seed-testmon` becomes a true from-scratch rebuild.** Today
`seed_testmon()` runs `pytest --testmon` against the *existing* DB, so an
already-advanced baseline yields "0 changed" and recovery does nothing
(the QS_281 dead end). Fix: call `_rebuild_testmon_baseline()` at the top
of `seed_testmon()` (before the `pytest --testmon` select-all). The
coverage reset is included because seeding establishes the baseline the
*next* `--impacted` accumulates onto â€” a stale `.coverage` left behind
would re-introduce the desync A1 fixes. Unit test mocks
`_purge_testmon_db` / `_reset_coverage_data` / `_stream_pytest` and asserts
the rebuild runs before the pytest seam.
- **Concurrency (dev-proxy/critic):** `finish-task` invokes `--seed-testmon`
  detached/best-effort on `MAIN_DIR` **post-merge** â€” the main checkout,
  not an active worktree. Worktree data files are per-worktree (see
  Problem), so the destructive reset cannot race a developer's worktree
  `--impacted`. It could only race a gate run *in main* (rare). **No lock
  added â€” the loop is closed by self-heal:** a main-checkout DB left
  corrupt/half-rebuilt by such a race is repaired on the *next* run there
  by A2 (orphan sidecars) + A3/`_ensure_testmon_db_safe` (corrupt â†’ purge â†’
  select-all). A stale baseline only ever costs more tests, never
  under-selects.

**A4 â€” Self-heal on changed-line coverage FAIL (the desync killer).**
Refactor the post-diff-base pipeline (testmon-run â†’ coverage.xml check â†’
diff-cover) into a helper `_run_impacted_pass(base)` returning a verdict.
`check_impacted(*, allow_self_heal: bool = True)` resolves the diff base
**once**, then calls `_run_impacted_pass`. When the **incremental**
diff-cover reports changed lines <100% **and** `allow_self_heal` is set,
perform **exactly one** automatic rebuild: `_rebuild_testmon_baseline()` â†’
`_run_impacted_pass(base)` again (now a select-all under `--cov`). Only if
*that* still fails â†’ genuine gap â†’ exit 1. The retry reuses the
already-resolved `base`, so there is **no second `git fetch` / tooling
probe**, and "exactly one rebuild" is obviously local.
- **Trigger gate (committed signal):** capture
  `was_incremental = TESTMON_DATA.exists() and TESTMON_DATA.stat().st_size > 0`
  as a local **at the very top of `check_impacted`, before**
  `_clean_orphan_cov_shards()` / `_ensure_testmon_db_safe()` run (either of
  which can purge the DB). The retry fires only when `was_incremental` is
  true. If testmon already select-all'd this run (fresh/purged baseline,
  `was_incremental` false), a FAIL is genuine â†’ exit 1 immediately â€” **no
  wasted select-all on the normal TDD-red case**. (`_selected_count` is a
  *test-only* helper and is **not** used here â€” the production signal is
  the `.testmondata` predicate above, the same one the existing
  `if not TESTMON_DATA.exists()` reset already keys on.)
- **Soundness (proven, not asserted):** `_rebuild_testmon_baseline()`
  resets coverage AND clears shards, so the select-all retry writes a
  *fresh, complete* coverage picture over the whole suite (the
  `--cov-append` appends onto an empty store; no orphan shard can re-fold
  in). That is ground truth: a line with no covering test stays 0% (no
  false PASS); a line whose covering test was wrongly deselected gets
  re-run and covered (false FAIL cleared).
- **Bounded / coverable:** the recursive pass uses `allow_self_heal=False`,
  so there is no recursion; the unit seam drives both branches by
  controlling `_run_impacted_pass`'s two verdicts and `was_incremental`.
- **Killed-retry recovery:** if the retry's own pytest is killed, the next
  plain `--impacted` run repairs the half-finished rebuild on entry via A2
  (orphan sidecars) + `_ensure_testmon_db_safe` (corrupt â†’ purge â†’
  select-all) â€” so "no manual file deletion" holds even then.
- **Observability:** one-line `_emit("impacted", â€¦)` console notice (no
  trailing period), e.g.
  `rebuilding testmon baseline and re-checking (changed lines <100% on an incremental run)`.

### B. Enforcement â€” implement agents ALWAYS run `--impacted`

Harness-sync is a hard co-modification requirement (drift checker): every
edit to a `.claude/agents/*` file co-modifies its `.cursor/` + `.opencode/`
mirrors.

**B1 â€” Mandatory language (all six files).** Replace "Default to the
impacted gate" with: "**ALWAYS** run
`python scripts/qs/quality_gate.py --impacted` before commit/PR. Do **not**
run, or substitute, the full gate locally."

**B2 â€” Close the loophole (the three `qs-implement-task.md` copies +
`project-rules.md`).** Delete the "or when you suspect coverage lost in
*unchanged* code" clause from the three agent copies **and** from the
`project-rules.md` `## Commands` full-gate comment (L36-37), so the rules
doc cannot license the escape B2 removes. Detecting coverage lost in
unchanged code is **CI's exclusive job** (whole-repo 100% on every PR).
The *only* local full-gate run is an explicit user request.

**B3 â€” Constrain failure recovery (the three `qs-implement-task.md`
copies).** Replace "fix autonomously and re-run" with: if `--impacted`
fails, fix the **code/tests** â€” never switch to the full gate to diagnose.
`--impacted` self-heals (A4); a manual `--seed-testmon` + re-run is an
escalation-only last resort for the residual case (e.g. two consecutive
killed runs), after which escalate to the user. The full gate is not an
inner-loop debugging tool.

## Acceptance criteria

1. `_clean_orphan_cov_shards()` removes pre-existing `.coverage.*` shards
   (and **only** shards â€” the combined `.coverage` survives), reusing the
   `_reset_coverage_data` glob, called at the start of `check_impacted()`
   before the pytest spawn; a seeded orphan shard never reaches
   `coverage.xml`. (A1)
2. `_ensure_testmon_db_safe()` removes orphaned `.testmondata-wal`/`-shm`
   when `.testmondata` is absent, so testmon never selects `0 tests`
   against a stale WAL. (A2)
3. `seed_testmon()` calls `_rebuild_testmon_baseline()` (purge + coverage
   reset + shard clear) before the select-all rebuild; a reseed against an
   advanced baseline still fully re-fingerprints (no "0 changed" dead end). (A3)
4. With `allow_self_heal=True`, an **incremental** (`was_incremental`
   true, captured before any purge) changed-line FAIL triggers **exactly
   one** rebuild (`_rebuild_testmon_baseline()` â†’ `_run_impacted_pass(base)`
   on the already-resolved base) before exit 1; the recovery path **emits
   the self-heal `_emit` notice** (so a recovered-PASS is distinguishable
   from a never-failed PASS); a false FAIL recovers to PASS, a genuine gap
   still exits 1, a select-all run (`was_incremental` false) does **not**
   trigger the retry, and the retry never recurses (`allow_self_heal=False`
   on the inner call). (A4)
5. Integration repro reproduces the killed-run desync at the
   **`_run_testmon`+shard+`.testmondata` level** (matching the existing
   `TestImpactedIntegrationRealTestmon` harness, which sets
   `PYTEST_DISABLE_PLUGIN_AUTOLOAD=1` + explicit `-p testmon.pytest_testmon`
   in a `tmp_path` `pkg/` repo), then asserts A4's recovery by calling the
   helpers (`_rebuild_testmon_baseline` / `_reset_coverage_data`) directly
   with `TESTMON_DATA`/`COVERAGE_DATA`/`COVERAGE_XML` patched to the tmp
   repo â€” **not** by driving the full `check_impacted()` orchestrator
   (which would require repointing the import-time `SRC_DIR`/`TESTS_DIR`
   and disabling plugin autoload, neither expressible without changing
   `_build_testmon_cmd`/`_stream_pytest`). The repro shows the deadlock
   before the fix and clean recovery after; a genuinely-untested function
   still FAILs. (C)
6. All six agent files mandate `--impacted` before commit/PR (B1); the
   three `qs-implement-task.md` copies additionally delete the
   "coverage-lost-in-unchanged-code" clause (B2) and forbid the full-gate
   diagnostic escape (B3); the `.cursor`/`.opencode` mirrors stay aligned.
7. New/changed `quality_gate.py` orchestration code is 100% covered via the
   subprocess seam: A4 unit tests patch `_run_impacted_pass` (or its
   `_stream_pytest`+`_run` constituents) for the two-verdict retry,
   **patch `_rebuild_testmon_baseline`** (so the retry touches no disk),
   and control `was_incremental` (seed/clear `.testmondata` under
   `tmp_path`) to drive the incrementalâ†’retry vs select-allâ†’no-retry
   branches; the **real** purgeâ†’resetâ†’rerun ordering is owned by the C
   integration repro (AC#5), not the mock seam.
8. The change set passes the **whole-repo** gate at 100%, authoritative in
   **CI**. **Local verification of the new tests is
   `python scripts/qs/quality_gate.py --quick tests/test_quality_gate.py`**
   (an explicit known test path, allowed by the grammar rule) â€” because
   `--impacted` `--ignore`s `test_quality_gate.py` and would not run the
   new tests. The implementing agent does **not** run the full gate to
   self-verify; CI is the 100% backstop (consistent with B2).
9. Docs updated in the same atomic commit (drift co-modification); a
   mechanical `python scripts/qs/check_doc_drift.py` pass confirms no doc
   `covers:` tracks `scripts/qs/quality_gate.py` (and if it ever does,
