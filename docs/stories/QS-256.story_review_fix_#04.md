## Findings to fix

### [should-fix] Future-dated reset-ask drop leaves `_first_cmd_reset_done` set
- File: `custom_components/quiet_solar/ha_model/bistate_duration.py:203-211`
- Severity: should-fix
- Source: qs-review-blind-hunter (verified repo-grounded)
- Description: In `use_saved_extra_device_info`, the future-dated drop
  of `asked_for_reset_user_initiated_state_time` (`ask_age_s < -60.0`)
  clears only the ask-time, while the sibling drop-guards (orphan
  override at `:189-197`, expired/future override at `:220-231`) clear
  all four override fields. The orphaned
  `asked_for_reset_user_initiated_state_time_first_cmd_reset_done=True`
  is later read in `check_load_activity_and_constraints` (`:729`) on a
  path where the ask-time is None, triggering a one-time spurious
  constraint wipe + forced solve right after restore.
- Proposed fix: Also set
  `asked_for_reset_user_initiated_state_time_first_cmd_reset_done =
  None` in the future-dated reset-ask drop branch, making it consistent
  with the other guards. Add a test: restore with future-dated ask-time
  + `_first_cmd_reset_done=True` → both fields cleared, no spurious
  constraint reset on the next update cycle.

### [nice-to-have] Empty-slots early-return in hold-off repartition
- File: `custom_components/quiet_solar/home_model/constraints.py` (`TimeBasedHoldOffConstraint.compute_best_period_repartition`)
- Severity: nice-to-have
- Source: qs-review-blind-hunter
- Description: With an empty `power_available_power`,
  `last_slot = len(...) - 1 = -1`, which collides with the documented
  "-1/-1 no-energy-impact sentinel" semantics of the trailing return
  fields while being a real (degenerate) value here. Unreachable in
  practice (the solver never passes empty slot arrays), but unclamped
  unlike `first_slot`.
- Proposed fix: Early-return a clean empty repartition when
  `len(power_available_power) == 0` (empty commands/power arrays,
  sentinels, budget passed through). Add a small test pinning the
  empty-array behavior.

### [nice-to-have] Doc note: restart visibility cost of the conservative restore-expiry window
- File: `docs/agents/concepts/user-override.md`
- Severity: nice-to-have
- Source: qs-review-blind-hunter
- Description: `use_saved_extra_device_info` evaluates stored-override
  expiry against `override_duration`, which at restore time may still
  hold the config default (the number entity restores later). If the
  default is smaller than the configured value, a still-valid override
  is dropped on every restart. The code docstring documents the
  conservative direction; the user-facing agent doc does not mention
  the restart visibility cost.
- Proposed fix: One-line note in `user-override.md` (and only there —
  no code change) describing this restart-time limitation and its
  conservative rationale.

### [nice-to-have] Pin that restore guards run before a future-dated override can be scored
- File: `custom_components/quiet_solar/home_model/constraints.py` (`score()` recency term) + `tests/test_constraints.py`
- Severity: nice-to-have
- Source: qs-review-blind-hunter
- Description: `override_recency_score` clamps a future-dated
  `start_of_constraint` to age 0 via `max(0.0, ...)`, granting the
  maximum recency 0.5 to a future constraint. Benign because the
  restore drop-guards eliminate future-dated overrides before scoring,
  but that ordering is asserted nowhere.
- Proposed fix: Add a comment at the clamp noting the restore guards
  are the upstream defense, and a small test asserting a future-dated
  `start_of_constraint` scores with recency 0.5 (clamped, no error) —
  pinning the clamp behavior explicitly.

### [nice-to-have] Comment: idle expired-at-push guard relies on the 0.25h step floor
- File: `custom_components/quiet_solar/ha_model/bistate_duration.py:923-928`
- Severity: nice-to-have
- Source: qs-review-edge-case-hunter
- Description: The idle branch's `if end_schedule <= time:` guard
  catches only `override_duration == 0`; any non-zero duration pushes a
  hold-off constraint. This is safe because the number entity's
  `native_step=0.25` makes 900s the smallest non-zero window, but the
  guard's reliance on that step floor is implicit.
- Proposed fix: One-line comment at the guard stating it relies on the
  0.25h minimum step (no sub-tick windows possible); revisit if the
  step contract changes. No code change.

### [nice-to-have] Causality-guard cold-start gap (no anchor, no current command)
- File: `custom_components/quiet_solar/ha_model/bistate_duration.py:801-805`
- Severity: nice-to-have
- Source: qs-review-edge-case-hunter
- Description: The causality guard only runs when
  `last_command_execution_time is not None`. On a brand-new install /
  cleared `.storage` (no restored `current_command`, so the restore
  anchor never fires) a stale entity state at the first detection tick
  is classified as a user override before QS has issued any command.
  Defensible (there is no prior command to attribute the state to),
  but it is the one detection path the guard does not cover.
- Proposed fix: Conservative mitigation: skip override classification
  when `last_command_execution_time is None and current_command is
  None` (no causality baseline yet), with a debug log. Add tests:
  (a) cold start, stale state, no command ever → no override armed;
  (b) after the first real command executes, detection works normally.
  If implementation reveals this breaks a legitimate detection scenario
  covered by existing tests (e.g. first-tick genuine user action on a
  fresh install), fall back to a documentation-level resolution at the
  guard site explaining the accepted gap — note the chosen resolution
  in the PR body.
