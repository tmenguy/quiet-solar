# QS-173: Fix permission allowlist and worktree upstream bugs

## Story

**As a** Quiet Solar pipeline developer
**I want** agents to run pipeline scripts without permission prompts and
new worktrees to be published to origin as `QS_<N>` with the local
branch tracking that remote branch
**So that** the slash-command pipeline runs unattended and `git push` /
`git pull` from a fresh worktree work without flags and never target
`origin/main`.

## Root Cause

### Bug 1 — Permission allowlist patterns are relative-only

`.claude/settings.json` lists allow entries like
`Bash(python scripts/qs/context.py:*)`. Claude Code matches the literal
command text with a prefix check, so when an agent runs the same script
with an absolute path
(e.g. `python /Users/.../QS_173/scripts/qs/context.py`) no entry matches
and the user is prompted. Some QS agents run from the main checkout,
others from a worktree, so both forms occur in practice. Evidence: the
user had to add the absolute form manually to `settings.local.json`.

All seven QS agents that invoke scripts (`qs-setup-task`,
`qs-create-plan`, `qs-implement-task`, `qs-implement-setup-task`,
`qs-review-task`, `qs-finish-task`, `qs-release`) hit the same root
cause.

### Bug 2 — `worktree-setup.sh` creates branch with `origin/main` upstream

`scripts/worktree-setup.sh` (in the new-branch path, currently around
line 56) runs:

    git -C "$MAIN_DIR" worktree add "$WORKTREE_DIR" -b "$BRANCH" origin/main

With git's default `branch.autoSetupMerge=true` this records
`origin/main` as the upstream of the new `QS_<N>` branch. Confirmed
live in this worktree: `git config branch.QS_173.merge` returns
`refs/heads/main`. Result: `git push` / `git pull` without explicit
args target `origin/main` from a feature branch.

The existing-branch path (the `then` branch of the same `if`) inherits
whatever upstream config the branch already has — if the branch was
previously created with this bug, the bad config persists.

### Task 2: Publish the branch to origin and track it (AC-2, AC-3)

In `scripts/worktree-setup.sh`, immediately after the worktree-creation
`if … else … fi` block (the one with the `show-ref --verify` test) and
**before** the `# P2: Symlink venv using derived basename` comment,
insert:

    # Publish the branch to origin and set upstream → origin/${BRANCH}.
    # This makes `git push` / `git pull` Just Work without flags and
    # guarantees the upstream is the feature branch, never origin/main
    # (regardless of git's branch.autoSetupMerge default). Idempotent:
    # if origin/${BRANCH} already exists and matches the local tip, push
    # is a no-op but still (re)sets the upstream — repairing any
    # pre-existing branch that was created with the old buggy
    # origin/main upstream.
    echo "Publishing ${BRANCH} to origin and setting upstream..."
    git -C "$WORKTREE_DIR" push -u origin "$BRANCH"

`git push -u origin "$BRANCH"` does two things in one call:
- creates `origin/${BRANCH}` on the remote if missing (or no-ops if it
  exists and matches the local tip)
- (re)sets the local branch's upstream to `origin/${BRANCH}`, which
  overwrites any wrong pre-existing upstream (e.g. `origin/main` from a
  branch created under the old buggy setup) — satisfying AC-3 without
  a separate defensive check.

If the remote branch exists and has diverged from local, the push will
fail loudly under `set -e`. That's the correct behaviour — a divergent
remote is operator-visible state that should not be silently force-overwritten.

The previous draft used `--no-track origin/main` plus a defensive
`unset-upstream` post-check. That left the local branch with no
upstream at all, which is mildly inconvenient (the user has to set
`-u` themselves on first push). The current approach is cleaner: one
explicit `push -u` does both jobs and ends the worktree in the right
final state.

## Dev Notes

- **Scope:** dev-environment only. No files under
  `custom_components/quiet_solar/` are touched. Next phase is
  `/implement-setup-task` — that agent runs the quality gate per its
  contract, so this story intentionally does not add a "run quality
  gate" task.
- **No Python code touched:** the two-layer boundary, async,
  `const.py`, and logging rules from `project-rules.md` do not apply
  to this change.
- **Why `**/` (recursive glob):** matches both the bare path
  (`scripts/qs/foo.py`, no leading directory) and absolute paths
  rooted at any parent location (worktree or main checkout). Required
  so the same `.claude/settings.json` works from both invocation
  sites.
- **Why keep relative entries:** muscle memory and existing
  `settings.local.json` overrides are built around them. Removing
  them risks regressing other invocations we haven't enumerated.
- **Legacy OpenCode pipeline:** untouched per user direction; the
  `worktree-setup.sh` fix benefits it incidentally since the script is
  shared.
- **Current `QS_173` upstream:** per user direction the story includes
  NO retroactive-recovery instruction. The fix is forward-only for
  newly-created worktrees; the `push -u origin "$BRANCH"` call in
  Task 2 will repair this branch the next time its worktree is
  recreated.
- **No new tests:** per user direction. Permission allowlists are
  declarative config and shell-script integration tests aren't worth
  the maintenance burden for this surface. Verification is manual
  (Task 4).
- **Issue body context:** the GitHub issue body for #173 is
  intentionally placeholder text ("ok again a test task to probe the
  workflow a bit to enhance the setup"). The real two-bug scope was
  negotiated interactively during the `/create-plan` phase; the
  authoritative spec is this story and the eventual PR description.

  parent directory, not just `quiet-solar` and `quiet-solar-worktrees`.
  User explicitly preferred this over a tighter pattern.
