# Story: Improve review step — better implement back and forth

issue: 145
branch: "QS_145"
Status: ready-for-dev

## Story

As a developer using the OpenCode review workflow,
I want the review orchestrator to interactively triage each finding one-by-one, generate a fix plan artifact, and produce a ready-to-paste prompt for the build agent,
so that the review-to-fix loop is structured, traceable, and efficient.

## Acceptance Criteria

1. Given the review consolidation finds zero actionable findings (all invalid/dismissed),
   When the triage phase completes,
   Then the review agent immediately renders `qs-finish-task-QS-<N>` and hands off to the finish phase — no interactive triage is started.

2. Given the review consolidation finds one or more actionable findings (must-fix or should-fix),
   When the triage phase begins,
   Then the agent presents findings one at a time, in order (must-fix first, then should-fix, then nice-to-have), showing: finding text, severity, source reviewer(s), file/line, and proposed action.

3. Given a finding is presented to the user,
   When the user is prompted,
   Then the user may choose exactly one of: **fix** or **skip** (forget it), and the agent records the decision without acting yet.

4. Given all findings have been triaged,
   When the agent has collected every decision,
   Then it presents a decisions summary table (finding, severity, decision) and waits for user confirmation before proceeding.

5. Given the user confirmed the decisions summary and at least one finding is marked "fix",
   When the agent generates the fix plan,
   Then it writes a markdown file at `docs/stories/<plan_name>_review_fix_#<XX>.md` where `<plan_name>` is the base story file name (without `.md`) and `<XX>` is an auto-incremented counter starting at 1 (incrementing if prior fix plans exist).

6. Given the fix plan file is written,
   When the agent presents it to the user,
   Then the fix plan contains: a summary of what to fix, a per-finding section with file, line, description, and proposed fix approach, and references back to the original story file.

7. Given the user approves the fix plan,
   When the agent finalizes,
   Then it generates and displays a ready-to-copy prompt block for the build agent (`qs-implement-task-QS-<N>`) that instructs it to read the fix plan file and implement the fixes, run quality gates, and push.

8. Given no findings are marked "fix" (all skipped),
   When the decisions summary is confirmed,
   Then the agent proceeds directly to render `qs-finish-task-QS-<N>` and hand off — no fix plan is generated.

## Dev Notes

### Affected Files

- `_qsprocess_opencode/agent_templates/qs-review-task.md.tmpl` — primary file to modify (the review orchestrator template)

### Architecture Notes

- This is a **template modification only** — no Python scripts, no source code changes. The review-task template is rendered by `render_agent.py` into each worktree's `.opencode/agents/` folder.
- The template uses `{{VAR}}` placeholders (ISSUE, PR_NUMBER, TITLE, WORK_DIR, STORY_FILE, BRANCH, AGENT_NAME).
- The fix plan file naming convention: `<story_base>_review_fix_#<XX>.md` where `<story_base>` is derived from `{{STORY_FILE}}` minus the `.md` extension, and `<XX>` increments from 1.
- The build agent prompt should reference `qs-implement-task-QS-{{ISSUE}}` which is already rendered in the worktree.

### Key Design Decisions

- **One-by-one triage** replaces the current bulk triage to give the user granular control.
- **Fix/skip only** — simplified from the current fix/defer/reject to match the user's request for "fix it or forget it".
- **Fix plan as artifact** — stored alongside the story so it's traceable and can be referenced by the build agent.
- **Prompt generation** instead of auto-handoff — the user manually pastes the prompt to the build agent, keeping them in control.
- **Auto-increment counter** on fix plan files supports multiple review-fix cycles.
