# Story QS-155: Fix handoff: /reload in spawn prompt + clean agent files before worktree deletion

issue: 155
branch: "QS_155"
Status: ready-for-dev

## Story

As a pipeline operator,
I want the handoff between phases to reliably discover rendered agents and the worktree cleanup to not be blocked by expected untracked agent files,
so that phase transitions are fully automated and reproducible.

## Context

Three issues in the OpenCode workflow dev setup:

1. **`/reload` is now a programmatic API call**: OpenCode exposes `POST /instance/reload` (returns `true`). After rendering agent files, call this endpoint so the server discovers them. Then `spawn_session.py` can create a session that sees the new agent. Currently templates say "Run `/reload`" (a UI slash command agents can't execute). The fix: integrate `POST /instance/reload` into `spawn_session.py` (call reload automatically before creating the session), and update all templates to remove the manual `/reload` instruction.

2. **Agent files block worktree cleanup**: `cleanup_worktree.py` calls `check_worktree_status()` (step 1) before `remove_agent_files()` (step 3). Untracked agent `.md` files under `.opencode/agents/` cause the status check to report `action_required`, aborting cleanup before agent removal ever runs. The fix: remove agent files first, then check status.

3. **Plan reviewer sub-agents rendered too late**: The `qs-create-plan` template renders its 4 plan-reviewer sub-agents in Phase 1 (step 4) — within the already-running session. These agents are only loaded at session startup, so Task-spawning them fails with "unknown agent". The proven pattern from review (which works today): `implement-task` renders review sub-agents via `next_step.py` before spawning the review session. Apply the same pattern: `setup-task` should render the 4 plan reviewers alongside `create-plan`, and `create-plan.md.tmpl` should stop rendering them itself.

## Acceptance Criteria

1. **Given** agents have been rendered into `.opencode/agents/`,
   **When** `spawn_session.py` is called,
   **Then** it automatically calls `POST /instance/reload` before creating the session, and the spawned session discovers and activates the new agent without error.

2. **Given** a worktree has untracked agent files (`qs-*-QS-N.md`) and no other uncommitted changes,
   **When** `cleanup_worktree.py --work-dir <path> --issue N` is called (no `--force`),
   **Then** agent files are removed before the status check and the cleanup completes with `"status": "removed"`.

3. **Given** a worktree has untracked agent files AND real uncommitted changes (non-agent files),
   **When** `cleanup_worktree.py --work-dir <path> --issue N` is called (no `--force`),
   **Then** agent files are still removed but the safety gate fires `action_required` for the real uncommitted changes.

4. **Given** all agent templates that perform handoffs (create-plan, implement-task, implement-setup-task, review-task),
   **When** updated,
   **Then** the manual "Run `/reload`" instruction is removed (reload is now automatic in `spawn_session.py`).

5. **Given** `setup-task` renders agents for the create-plan phase,
   **When** it completes,
   **Then** it has rendered 5 agents: `qs-create-plan-QS-N` + 4 plan reviewers (`critic`, `concrete-planner`, `dev-proxy`, `scope-guardian`), mirroring how `implement-task` renders 5 agents for the review phase.

6. **Given** `create-plan` starts in a new session where plan reviewer agents were pre-rendered by `setup-task`,
   **When** it Task-spawns the 4 plan reviewers in Phase 4 (adversarial challenge),
   **Then** all 4 sub-agents are discovered and execute successfully.

## Adversarial Review Notes

**Reviewers:** Self-review (sub-agent spawn failed — this is exactly bug #3 above; reload API discovered during live debugging)
**Rounds:** 2

### Key findings incorporated:
- Agent removal should happen early in ALL paths (normal, force, push-first) for consistency
- `/reload` is now programmatic via `POST /instance/reload` — confirmed working (returns `true`)
- **Critical: reload scope is server-level, not session-level.** `POST /instance/reload` makes the OpenCode server re-scan agent files so **new sessions** see them. But the Task tool within the **current session** does NOT re-scan — agents rendered mid-session remain invisible to Task-spawn. This confirms that sub-agents MUST be rendered in the previous phase (before session startup). Reload fixes the spawn_session handoff only.
- Reload should be integrated into `spawn_session.py` so all handoffs get it automatically
- Templates should remove manual `/reload` instructions (now redundant)
- Plan reviewer rendering must move to setup-task, mirroring the proven review pattern
- This is a dev-setup task — must use `implement-setup-task` (not `implement-task`) for correct permissions and quality gate

### Decisions made:
- Integrate reload into `spawn_session.py` (not a standalone script) — this is the single choke point for all phase transitions
- Reload is best-effort: log failure but don't abort (the server might not be running in test/CI)
- No consolidation of `cleanup_agents.py` into `cleanup_worktree.py` — out of scope
- Review sub-roles are already handled correctly — no changes needed there

### Known risks accepted:
- `setup-task` is a static agent (`.opencode/agents/qs-setup-task.md`) — modifying it affects all future tasks. This is intentional.
- The `spawn_session.py` directory routing may not work correctly — all sessions show `directory: /Users/tmenguy` (global project) regardless of `x-opencode-directory`. Pre-existing issue, out of scope.
