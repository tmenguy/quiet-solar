### [must-fix] forward_min invariant broken across placements
- File: `custom_components/quiet_solar/home_model/constraints.py:1334, 1582-1633`
- Severity: must-fix
- Source: qs-review-blind-hunter + qs-review-edge-case-hunter

#### Description

Two distinct failure modes for the running-min guard.

**Mode A — Over-conservative (edge-case-hunter):** Initial `forward_min`
is built over `[k:num_slots]` (full array), but the in-place
`forward_min[i:last_slot+1] -= state_delta_wh` only updates a sub-slice.
When slots beyond `last_slot` have lower values than in-range slots,
after the first decrement `forward_min` underestimates the true running
min, skipping legitimate placements.

Reproducer: `bat=[5000,5000,5000,1000,1000]`, `last_slot=2`, floor=800.
Place 500 Wh at slot 0 → `forward_min` becomes `[500,500,500,1000,1000]`,
but true `min(bat[k:5])` for k=1 is still 1000. Subsequent placement
at slot 1 sees `max_drain_wh = max(0, 500-800) = 0` → skip. Real budget
at slot 1 was 200 Wh.

**Mode B — Stale (under-conservative, blind-hunter):** When iterating
backward (the `energy_delta >= 0.0` path: "start from the end"), a
later placement at `j < i` reads `forward_min[j]` that was never
decremented (the `[i:last_slot+1]` slice didn't cover j). The running
min anchor for `forward_min[j]` may have been pinned to a slot ≥ i;
decrementing those slots lowers the true running min at j, but
`forward_min[j]` stays stale → permits drain that breaches the actual
floor.

#### Proposed fix

Restrict `forward_min` to the working window AND back-propagate after
each placement:

```python
# at gate entry (replacing line 1334):
forward_min = np.empty_like(bat_charge_traj)
forward_min[first_slot:last_slot + 1] = (
    np.minimum.accumulate(
        bat_charge_traj[first_slot:last_slot + 1][::-1]
    )[::-1]
)

# after each placement at slot i (replacing the in-place block at
# constraints.py:1624-1635):
bat_charge_traj[i:last_slot + 1] -= state_delta_wh
forward_min[i:last_slot + 1] -= state_delta_wh
# Back-propagate: for k < i, forward_min[k] = min(forward_min[k], forward_min[i])
if i > first_slot:
    np.minimum(
        forward_min[first_slot:i],
        forward_min[i],
        out=forward_min[first_slot:i],
    )
```

Update the inline comment at `constraints.py:1626` to match the new
implementation (the comment is nice-to-have #5 — addressed here as a
companion to the must-fix).

#### Tests to add

In `tests/test_constraints_pre_discharge_guard.py`:

- `test_guard_running_min_correct_when_post_window_slots_lower`:
  build `bat_charge_traj=[5000,5000,5000,1000,1000]`, `last_slot=2`,
  `first_slot=0`, floor=800. Place a placement that consumes 500 Wh
  at slot 0; assert the next placement at slot 1 is NOT skipped (real
  budget is 200 Wh per Mode A).
- `test_guard_running_min_back_propagates_after_late_slot_placement`:
  build a scenario where backward iteration places at slot 2 first
  (decrementing `forward_min[2:]`), then attempts slot 0; assert
  `forward_min[0]` reflects the decrement (not the stale pre-placement
  value) per Mode B.

### [must-fix] Multi-pass loop double-decrements `bat_charge_traj`
- File: `custom_components/quiet_solar/home_model/constraints.py:1629`
- Severity: must-fix
- Source: qs-review-edge-case-hunter

#### Description

`adapt_repartition` runs a multi-pass loop (`num_passes=2` when
`num_max_on_off` is set). Pass-2 reads
`current_command_power = existing_commands[i].power_consign` (NOT
`out_commands[i]`, which is what pass-1 mutated). If pass-1 placed at
slot `i` (e.g., adjacent to a pre-existing ON slot — free transition
cost), pass-2 may revisit the SAME slot with the same baseline, compute
a fresh `delta_power`, OVERWRITE `out_delta_power[i]`, and STILL apply
its own `bat_charge_traj[i:last_slot+1] -= state_delta_wh`. Net:
trajectory decrement is pass-1 + pass-2, but realized placement is
only pass-2's.

Reproducer: `existing_commands[i] = CMD_AUTO_GREEN_ONLY(1500W)`, pass-1
bumps to 2500W (delta=1000, state_delta=1000*dt/3600), then pass-2
revisits same slot and bumps to 3500W (delta=2000, state_delta=2000*dt/3600).
`bat_charge_traj` is decremented by `3000*dt/3600`, but `out_delta_power[i]`
is overwritten to 2000.

#### Proposed fix

Add a guard at the top of the placement loop in pass-2 (and as defense-
in-depth in pass-1, which is harmless since `out_commands[i]` is
already None there):

```python
# at the top of `for i in sorted_available_power:` (around constraints.py:1470):
if out_commands[i] is not None:
    continue
```

This is the user's preferred fix direction.

#### Tests to add

In `tests/test_constraints_pre_discharge_guard.py`:

- `test_multi_pass_does_not_double_decrement_bat_charge_traj`:
  build a 3-slot scenario with `existing_commands = [CMD_AUTO_GREEN_ONLY(1500W), None, None]`,
  `num_max_on_off=4`, multi-step commands `[1500, 2500, 3500]`.
  Force a scenario where pass-1 would place at slot 0 with free
  transition and pass-2 would revisit. Assert:
  - `sum(d * dt / 3600 for d in out_delta_power) == initial_bat[last_slot] - final_bat[last_slot]`
  - `out_commands[0].power_consign` matches the pass-1 placement (since pass-2 skipped via the new guard).

### [should-fix] `probe_window_end = i` before break — double-counts surplus
- File: `custom_components/quiet_solar/home_model/solver.py:1363`
- Severity: should-fix
- Source: qs-review-edge-case-hunter

#### Description

In the refill-shrink loop, `probe_window_end = i` is set BEFORE the
`nrj_to_recharge <= 0.0` break check. The break-slot's surplus is
conceptually accounted twice: subtracted from the refill budget AND
included in the probe window (so loads placed there can also consume
it).

#### Proposed fix

```python
nrj_to_recharge -= slot_surplus
if nrj_to_recharge <= 0.0:
    probe_window_end = i + 1  # exclude the slot whose surplus satisfied the refill
    break
probe_window_end = i  # this slot's surplus did NOT satisfy the refill
```

(Note: bounds checking — `probe_window_end` must remain ≤ `last_surplus_idx`
since `i + 1` may overshoot when the break fires on the first
iteration. Clamp with `min(i + 1, last_surplus_idx + 1)` if needed.)

Add a focused test that reproduces the scenario described in
edge-case-hunter's reproducer and asserts `probe_window_end` excludes
the refill-satisfying slot.

### [should-fix] `battery_drain_budget_wh` uses NOW-time charge, not future peak
- File: `custom_components/quiet_solar/home_model/solver.py:1334`
- Severity: should-fix
- Source: qs-review-edge-case-hunter

#### Description

`battery_drain_budget_wh = current_charge - get_value_empty() - safety_margin`
uses NOW-time charge; the trigger gate uses the SIMULATED trajectory.
When the battery is currently low but the trajectory climbs to 100%
mid-day, `battery_drain_budget_wh ≈ 0` caps `energy_to_be_spent` even
though the LATER full battery could absorb meaningful pre-discharge.

#### Proposed fix

Use the future peak charge as the drain budget basis:

```python
# Probe-window peak charge — captures "how much room the battery has
# over the probe window", not "how much I can drain right now".
max_future_charge_wh = float(
    np.max(bat_charge_traj[probe_window_start:probe_window_end + 1])
)
battery_drain_budget_wh = max(
    0.0,
    max_future_charge_wh - self._battery.get_value_empty() - safety_margin_wh,
)
```

Add a test `test_drain_budget_uses_future_peak_not_current_charge` with
`current_charge` near empty and `bat_charge_traj` peaking above the
safety margin: assert `_constraints_delta` is called with
`energy_delta > 0` (pre-discharge fires) — which would NOT happen under
the current `current_charge`-based gate.

(Note: this supersedes should-fix #8's use of `current_charge_wh`.
Apply should-fix #8's caching pattern to any remaining `current_charge`
reads in the same block.)

### [nice-to-have] Dusk-finder `i = j + 1` skips single-slot sun spike
- File: `custom_components/quiet_solar/home_model/solver.py:530`
- Severity: nice-to-have
- Source: qs-review-edge-case-hunter

#### Description / Fix

When the inner sustain loop breaks on `pv[j] >= threshold`, the outer
loop's `have_seen_sun` flag is not updated for that slot. Currently
harmless because the prior main-branch path always sets the flag, but
fragile.

```python
while j < num_slots and float(pv[j]) < _DUSK_THRESHOLD_W:
    sustained_s += float(durations[j])
    if sustained_s >= _DUSK_SUSTAIN_S:

# Inner loop exited because pv[j] >= threshold (or j == num_slots).
# Mark have_seen_sun for the threshold-crossing slot so subsequent

if j < num_slots:
    have_seen_sun = True
i = j + 1
```
