### [must-fix] M1 — JS syntax error: backticks inside HTML comment in template literal — card fails to load
- File: `custom_components/quiet_solar/ui/resources/qs-pool-card.js:447-457`
- Severity: **must-fix (BLOCKER)**
- Source: user-reported runtime error confirmed by qs-review-edge-case-hunter
- Description: Review-fix #01 N6 added an HTML comment
  (`<!-- Review-fix #01 N6 ... -->`) **inside** the
  `this._root.innerHTML = \`...\`` template literal. The comment body
  contains literal backticks at lines 449-450 (around `` `dragMove` ``
  and `` `_fmt(..., false)` ``). JavaScript template literals treat
  backticks as string delimiters — they do NOT honour HTML comment
  syntax. The JS parser closes the outer template literal at the first
  inner backtick, then tries to parse the next tokens
  (`dragMove`, `_fmt(..., false)`, ...) as JavaScript. The `, false`
  on line 450 produces the user-reported runtime error:
  `Uncaught SyntaxError: Failed to execute 'appendChild' on 'Node':
  Unexpected token 'false'`. The card never loads in HA.
- Why our tests didn't catch this: `tests/test_dashboard_rendering.py`
  reads `qs-pool-card.js` as **text** for structural regex
  assertions. It never `eval`s, `import`s, or syntax-parses the JS,
  so a JS syntax error passes the entire quality gate silently. This
  is a systemic hole in our JS-card testing strategy.
- Proposed fix:
  1. **Move the comment out of the template literal.** Delete the
     HTML comment block at lines 447-457. Replace it with a `//`-style
     JS comment placed **immediately above** the template-literal
     line `this._root.innerHTML = \`` (or above the surrounding
     statement block). Example (place near the top of `_render`,
     adjacent to the existing N6 reference in the comment block above
     `defaultDuration`):

     ```js
     // Review-fix #01 N6 (target-value span at the "Actual / Target
     // Hours" block) — the displayTargetHours value is rendered with
     // _fmt(displayTargetHours, false) (un-rounded) so the committed
     // display matches the live drag preview (dragMove emits the
     // half-hour snap value via _fmt(..., false)). Without false,
     // 8.5h would render as "9h" on release while the drag handle
     // still showed "8.5". The family (radiator / water-boiler /
     // on_off_duration / climate) defaults to round=true and shares
     // this minor mismatch; pool diverges because the half-hour grid
     // is more visible on the 24h scale.
     ```

     (Strip ALL backticks and `${...}` patterns from the comment
     text — use plain prose. Even with the comment moved outside the
     template literal, keeping bare backticks in JS source comments
     is a future-foot-gun if anyone copy-pastes the comment back into
     a template literal.)

  2. **Re-verify the AC-2 / N6 test still matches.** The current
     `test_pool_card_target_value_uses_palette_color_not_theme_variable`
     pins the span with `_fmt(displayTargetHours, false)`. Moving the
     comment does NOT change the rendered span, so the test should
     still pass — re-run it to confirm.

  3. **Add a JS-syntax-validation test** (NEW — guards the whole
     class of bug). Two options; pick whichever fits the existing
     test infrastructure:

     - **Option A (preferred):** add a pytest case that invokes
       `node --check <path-to-qs-pool-card.js>` (and the other
       sibling JS cards in `custom_components/quiet_solar/ui/resources/`)
       via `subprocess`. Skip with a clear `pytest.skip(...)` if
       `node` is not on `PATH`. Mark as `@pytest.mark.js_syntax`. The
       check is fast (~50ms per file) and catches the entire class
       of "JS parse error" bugs that structural-regex tests miss.

     - **Option B (Python-only fallback):** add a hand-rolled
       template-literal-balance checker — count backticks at the JS
       top level, ignoring escapes (`\\\``) and string literals
       (`'...'`, `"..."`). Assert the count is even. This won't
       catch all syntax errors but it catches THIS specific class
       (unbalanced backticks). Less robust than Option A but
       node-free.

     Either way: add a regression test
     `test_pool_card_js_syntax_is_valid` (and ideally the same for
     the other duration-card JS files — radiator, water-boiler,
     on_off_duration, climate). Iterate across all
     `custom_components/quiet_solar/ui/resources/qs-*.js` paths so
     the regression coverage scales automatically.

  4. **Manual verification step before pushing:** open the pool card
     in HA after the fix, confirm no console errors, confirm the
     handle renders, drag-commits, and the target-value span shows
     the expected value.

### [nice-to-have] N1 — `progressRatio` overshoots during HA boot
- File: `custom_components/quiet_solar/ui/resources/qs-pool-card.js:290-292`
- Severity: nice-to-have
- Source: qs-review-edge-case-hunter
- Description: With review-fix #01 N5's `_safeNumber(sDefaultOnDuration, 1)`
  fallback, if `hoursRun > 1` when the entity is still booting
  (`unknown`/`unavailable`), `progressRatio = min(1, hoursRun/1) = 1`
  → the water-fill renders FULL until the entity registers. Pre-fix-plan
  this rendered EMPTY in the same boot window.
- Proposed fix: detect the fallback path explicitly and zero the
  ratio when the sensor state is missing. One approach: capture the
  raw sensor state separately and gate `progressRatio`:

  ```js
  const defaultDurationKnown =
      sDefaultOnDuration?.state != null
      && sDefaultOnDuration.state !== ''
      && sDefaultOnDuration.state !== 'unknown'
      && sDefaultOnDuration.state !== 'unavailable';
  // In default mode, only show progress when the default_on_duration
  // sensor has a real value; the `1` fallback is purely for handle
  // positioning, not for the water-fill ratio.
  const progressRatio =
      displayTargetHours > 0 && (!isDefaultMode || defaultDurationKnown)
          ? Math.max(0, Math.min(1, hoursRun / displayTargetHours))
          : 0;
  ```

  Add a structural test that asserts the new gate is in place.

### [nice-to-have] N2 — Strict equality on `poolMode` won't trim whitespace
- File: `custom_components/quiet_solar/ui/resources/qs-pool-card.js:231`
- Severity: nice-to-have
- Source: qs-review-edge-case-hunter
- Description: `isDefaultMode = poolMode === 'bistate_mode_default'`
  uses strict equality. A padded state string
  (`'bistate_mode_default '` with trailing whitespace, e.g. via a
  custom integration) would flip `isDefaultMode` to `false`. Family
  cards share the pattern, so this is family-consistent.
- Proposed fix: add defence-in-depth trimming:

  ```js
  const poolMode = (selPoolMode?.state || 'bistate_mode_default').trim();
  ```

  No test required — defence-in-depth.

### [nice-to-have] N3 — Silent commit when `e.default_on_duration` is undefined
- File: `custom_components/quiet_solar/ui/resources/qs-pool-card.js:358`
- Severity: nice-to-have
- Source: qs-review-edge-case-hunter
- Description: `canDragHandle = isEnabled && isDefaultMode` does not
  gate on `e.default_on_duration` being defined. If the dashboard
  template omits the key, drag wires up but `_setNumber(undefined, …)`
  no-ops, then the value snaps back to the fallback (1h after N5,
  more visible than 0h pre-N5).
- Proposed fix: gate on `e.default_on_duration` truthy:

  ```js
  const canDragHandle = isEnabled && isDefaultMode && !!e.default_on_duration;
  ```

  And add a console.warn in the wire path when `canDragHandle` is
  true but `entityId` is falsy (defensive — should never happen
  given the gate above, but cheap insurance). Add a structural test
  pinning the three-term gate.
