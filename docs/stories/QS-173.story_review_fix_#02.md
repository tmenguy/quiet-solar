# QS-173 Review Fix #02

Phase 4 re-review of PR #174 (after fix #01 + the scope-aware-prompt
follow-up) surfaced 2 must-fix items and 5 should-fix items. The user
selected "fix all" — this plan lands every must-fix and every should-fix.
Nice-to-haves are explicitly out of scope.

## Findings to address

### must-fix

**MF-1 (stale Task 3 prose reference, blind-hunter):** at
`docs/stories/QS-173.story.md` lines 426-427, the Dev Note still says
`"Verification is manual (Task 3)."` Since commit 3e92eab inserted a new
Task 3 (scope-aware prompt) above the manual-verification task, manual
verification is now Task 4 — the prose reference is stale.

**MF-2 (substring grep in tracked-worktree detection, edge-case-hunter):**
at `scripts/worktree-setup.sh:49-50`, `grep -qF "worktree ${WORKTREE_DIR}"`
is an unanchored substring match. If any sibling tracked-worktree path
has `$WORKTREE_DIR` as a prefix (e.g. checking `QS_17` while `QS_173`
exists), the check falsely classifies the current dir as a tracked
worktree and then tries `git worktree remove --force` on a path that
isn't actually registered. Fix: line-anchored match (`grep -qxF`).

### should-fix

**SF-1 (process violation by 3e92eab, blind-hunter):** commit 3e92eab
introduced a brand-new Task 3 with new scope, violating fix #01's
explicit `"Do NOT re-introduce a 'Task 3' with new scope"`. The behavioural
consequence (stale prose) is MF-1; once MF-1 lands, SF-1 is resolved. No
separate action required, but recorded here so the discipline note isn't
lost.

**SF-2 (destructive recovery overwrites uncommitted work, edge-case-hunter):**
at `scripts/worktree-setup.sh:53-68`, the partial-state recovery branch
runs `git worktree remove --force` (or `rm -rf`) without checking
whether the operator has uncommitted edits, staged changes, or stashes
in the worktree — silent data loss if recovery fires after a failed
push while the operator was inspecting/editing.

**SF-3 (no fallback on worktree remove failure, edge-case-hunter):**
same recovery block — if `git worktree remove --force` fails (locked
worktree, corrupted `.git` file, permission), `set -e` aborts and the
operator gets no actionable error message.

**SF-4 (non-fast-forward push retry loops, edge-case-hunter):** with the
relaxed guard, re-running after a failed `push -u` to a diverged
`origin/QS_<N>` always re-attempts the push and re-hits the failure ad
infinitum until the operator intervenes manually. Detect divergence
explicitly and surface a clear "remote diverged" message instead of
retrying blindly.

**SF-5 (`scripts/qs/utils.py` not in AC-1 glob list, acceptance-auditor):**
`scripts/qs/utils.py` exists but has no glob pair in `.claude/settings.json`.
Parent story said: *"If extra `.py` files exist (e.g. a new script added
since this story was written), add allow patterns for them in the same
pair-form."* `__init__.py` is also new but is never invoked as a script
(only imported), so it's exempted from the pattern requirement.

### Task B — Worktree-setup hardening (`scripts/worktree-setup.sh`)

**B.1 (MF-2)** — At lines 49-50, change
`grep -qF "worktree ${WORKTREE_DIR}"`
to
`grep -qxF "worktree ${WORKTREE_DIR}"`
(the `-x` flag requires a full-line match; porcelain emits one entry per
line with the exact path, so this is safe and tight).

**B.2 (SF-2)** — In the partial-state recovery branch (currently at
lines 53-68), before running `git worktree remove --force` or
`rm -rf`, check for uncommitted work in `$WORKTREE_DIR`:
```bash
DIRTY=""
if [ -d "${WORKTREE_DIR}/.git" ] || [ -f "${WORKTREE_DIR}/.git" ]; then
    DIRTY="$(git -C "$WORKTREE_DIR" status --porcelain 2>/dev/null || true)"
    STASHES="$(git -C "$WORKTREE_DIR" stash list 2>/dev/null || true)"
fi
if [ -n "$DIRTY" ] || [ -n "$STASHES" ]; then
    echo "Error: worktree ${WORKTREE_DIR} has uncommitted changes or stashes."
    echo "  Status: ${DIRTY:-(none)}"
    echo "  Stashes: ${STASHES:-(none)}"
    echo "Refusing to destructively recover. Resolve manually:"
    echo "  cd ${WORKTREE_DIR} && git status"
    exit 1
fi
```
…then proceed with the existing `remove --force` / `rm -rf` path.

**B.3 (SF-3)** — Wrap `git worktree remove --force "$WORKTREE_DIR"` in
an explicit failure check (the `||` form rather than relying on `set -e`):
```bash
if ! git -C "$MAIN_DIR" worktree remove --force "$WORKTREE_DIR"; then
    echo "Error: 'git worktree remove --force' failed on ${WORKTREE_DIR}."
    echo "The worktree may be locked. Try:"
    echo "  git worktree unlock ${WORKTREE_DIR}"
    echo "  git worktree remove --force ${WORKTREE_DIR}"
    echo "…then re-run this script."
    exit 1
fi
```

**B.4 (SF-4)** — Detect non-fast-forward divergence **before** running
`push -u`. After the HEAD verification block (around line 152), before
the push at line 154:
```bash
REMOTE_TIP="$(git -C "$WORKTREE_DIR" ls-remote origin "$BRANCH" 2>/dev/null | awk '{print $1}')"
if [ -n "$REMOTE_TIP" ]; then
    LOCAL_TIP="$(git -C "$WORKTREE_DIR" rev-parse "$BRANCH")"
    if [ "$REMOTE_TIP" != "$LOCAL_TIP" ] && \
       ! git -C "$WORKTREE_DIR" merge-base --is-ancestor "$REMOTE_TIP" "$LOCAL_TIP" 2>/dev/null; then
        echo "Error: origin/${BRANCH} has commits not in local ${BRANCH}."
        echo "  Remote tip: ${REMOTE_TIP}"
        echo "  Local tip:  ${LOCAL_TIP}"
        echo "Refusing to push (would be non-fast-forward)."
        echo "Resolve manually:"
        echo "  cd ${WORKTREE_DIR} && git fetch && git rebase origin/${BRANCH}"
        echo "…then re-run this script (or just 'git push')."
        exit 1
    fi
fi
```
