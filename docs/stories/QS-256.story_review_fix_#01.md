## Findings to fix

### [should-fix] Naive-datetime restore crash in override-expiry check
- File: `custom_components/quiet_solar/ha_model/bistate_duration.py:185-187`
- Severity: should-fix
- Source: qs-review-edge-case-hunter
- Description: `use_saved_extra_device_info` computes
  `datetime.now(pytz.UTC) - self.external_user_initiated_state_time`. If a
  legacy or hand-edited `.storage` entry holds a tz-naive isoformat string,
  `fromisoformat` yields a naive datetime and the subtraction raises
  `TypeError: can't subtract offset-naive and offset-aware`, aborting
  restore for that load.
- Proposed fix: if `self.external_user_initiated_state_time.tzinfo is None`,
  coerce with `.replace(tzinfo=pytz.UTC)` before the age computation. Apply
  the same coercion to the pre-existing comparison site that consumes this
  field later in `check_load_activity_and_constraints` (around line 644) so
  both consumers are safe. Add a test restoring a naive-isoformat stored
  override (both expired and valid variants).

### [should-fix] ON-override with `override_duration == 0` lacks the expired-at-push guard
- File: `custom_components/quiet_solar/ha_model/bistate_duration.py:858-883`
- Severity: should-fix
- Source: qs-review-edge-case-hunter
- Description: the new idle-override branch has an explicit
  `end_schedule <= time` expired-at-push guard (reset directly), but the ON
  branch does not — with the user-settable override-duration number at its
  minimum of 0 (`number.py`, `native_min_value=0`), an external ON pushes a
  zero-length `TimeBasedSimplePowerLoadConstraint`
  (`start_of_constraint == end_of_constraint == time`, `target_value == 0`).
  The two override directions behave asymmetrically for the same user input.
- Proposed fix: mirror the idle branch's
  `if end_schedule <= time: self.reset_override_state_and_set_reset_ask_time(time)`
  guard in the ON branch before constructing/pushing the constraint. Add a
  test: `override_duration = 0.0` + external ON → direct reset, no
  constraint pushed.

### [nice-to-have] Score-dominance term relies on un-asserted invariants
- File: `custom_components/quiet_solar/home_model/constraints.py:~344`
- Severity: nice-to-have
- Source: qs-review-blind-hunter
- Description: the QS-256 (D4) override score term's dominance ("== 1e14, 5x
  margin") depends on `type_score` staying capped and on the span constants;
  a future change to type scoring could silently break override dominance —
  only AC11's test would catch it indirectly.
- Proposed fix: make the invariant explicit — e.g. a module-level named
  constant for the override term and/or an assertion (test-level is fine)
  that the override term strictly exceeds the maximum possible sum of all
  lower-order terms computed from the actual span constants (not hardcoded
  literals).

### [nice-to-have] Two simultaneous USER_OVERRIDE constraints tie at the override term
- File: `custom_components/quiet_solar/home_model/constraints.py:335-351`
- Severity: nice-to-have
- Source: qs-review-edge-case-hunter
- Description: two simultaneously-live USER_OVERRIDE constraints (e.g. a
  stale restored hold-off alongside a freshly-pushed one) both receive the
  identical `+1e14` term, so same-end-time cluster dedup falls back to
  lower-order terms with no override-vs-override determinism. The
  single-override-per-load invariant normally prevents this.
- Proposed fix: document the single-override invariant at the score-term
  site (comment), and/or add a stable secondary tie-break (newer
  `start_of_constraint` wins). Documentation-level resolution is acceptable.

### [nice-to-have] Duplicated causality-anchor block
- File: `custom_components/quiet_solar/home_model/load.py:653-657` and `:733-737`
- Severity: nice-to-have
- Source: qs-review-blind-hunter
- Description: the identical
  `if is_command_set is True: self.last_command_execution_time = time`
  block appears in both `launch_command` and `force_relaunch_command`; the
  two copies can drift.
- Proposed fix: extract a tiny private helper (e.g.
  `_anchor_causality_guard_if_executed(is_command_set, time)`) and call it
  from both sites. No behavior change; existing tests must stay green.

### [nice-to-have] Defensive `first_slot` clamp in hold-off repartition
- File: `custom_components/quiet_solar/home_model/constraints.py:2749-2750`
- Severity: nice-to-have
- Source: qs-review-edge-case-hunter
- Description: `TimeBasedHoldOffConstraint.compute_best_period_repartition`
  indexes `time_slots[first_slot]` right after `bisect_left` without a
  bounds check; if `current_start_of_constraint` exceeded every slot,
  `bisect_left` returns `len(time_slots)` → IndexError. Unreachable in
  practice (hold-off starts at "now") and copied verbatim from the base
  method, but cheap to harden in the new code.
- Proposed fix: clamp `first_slot = min(first_slot, len(power_available_power) - 1)`
  before the `time_slots[first_slot]` comparison in the new class only (do
  NOT touch the base method — out of scope).
