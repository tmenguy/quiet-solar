# QS-178: Aggressive battery pre-discharge for big-sun days

## Story

**As a** Quiet Solar user with a forecast-rich PV setup and a flexible-load fleet (car, hot water, …)
**I want** the solver to pre-discharge the battery on forecast-proven big-sun days, including via overnight / early-morning load placements
**So that** the upcoming midday peak production has room to charge the battery, and we waste less solar energy to inverter DC clamping or grid feedback.

## Root Cause

[`solver.py`](../../custom_components/quiet_solar/home_model/solver.py)
lines 1161-1273 already contain an "aggressive surplus" block, but it
under-fires for four concrete reasons:

1. **6h horizon cap** (line 1179) — too short for an overnight / early
   morning run targeting a midday peak.
2. **Waste counter ignores DC clamp loss** — only `_available_power[i] < 0`
   is counted; `_battery_charge_power_by_inverter_AC_clamping` (already
   computed at line 103) is ignored, so DC-coupled clipping is invisible.
3. **Budget cap uses `battery.get_value_full()`** (line 1184) — when the
   battery is near empty and the surplus is large, the cap doesn't
   actually bound drain.
4. **Probe window starts ~2h before first surplus** (lines 1191-1197) —
   can't reach overnight slots where the only available draw is from the
   battery.

There is also a **latent over-discharge gap** that emerges once the
budget is widened: today's `power_headroom` array inside
`MultiStepsPowerLoadConstraint.adapt_repartition` is computed once per
constraint from a pre-call battery sim. Within a single call placing
loads at several no-sun slots, the per-slot envelope does not propagate
cross-slot drain — so a wider budget combined with constraint windows
that funnel placements into no-sun slots can dip `battery_charge` below
its safety floor. The fix's Layer 3 (a per-slot battery-charge guard
with forward-min lockstep propagation, gated on the consumption-increase
path) closes this gap, and a failing regression test written FIRST is
the structural proof that Layer 1 + Layer 2 are insufficient on their
own.

The existing `_constraints_delta` → `adapt_repartition` path already
places loads on no-sun slots because `_max_possible_production[i]`
includes `battery_possible_discharge[i]`
([`solver.py`](../../custom_components/quiet_solar/home_model/solver.py)
line 181), so `power_headroom[i]` is positive at night while the battery
has charge. Option A (enabling `additional_energy_to_deplete` at line
966) is **not** needed.

## Acceptance Criteria

### AC-1: Dusk finder returns the start of the sustained low-pv tail

```gherkin
Given a solar production array self._solar_production and per-slot
  durations self._durations_s and local time_slots
When _find_next_dusk_idx() is called
Then it returns the first slot index i such that:
  - some earlier slot has pv >= _DUSK_THRESHOLD_W (have_seen_sun guard)
  - time_slots[i].astimezone(tz=None).hour >= _DUSK_EARLIEST_LOCAL_HOUR (= 15)
  - pv[i:j] < _DUSK_THRESHOLD_W for j-i durations summing to
    >= _DUSK_SUSTAIN_S (= 90 min)
Or returns None when no such slot exists (cloudy-all-day) or when the
  current run starts post-sunset and have_seen_sun stays False.
```

### AC-2: Expected solar waste accounts for both AC excess and DC clamp loss

```gherkin
Given self._available_power, self._durations_s,
  self._battery_charge_power_by_inverter_AC_clamping (may be None),
  and a battery_charge trajectory
When _compute_expected_solar_waste(battery_charge) is called
Then it iterates [0, horizon_idx] where
  horizon_idx = _find_next_dusk_idx() if not None else num_slots - 1
And for each slot i where self._battery.is_value_full(battery_charge[i])
  it sums waste_w = max(0, -self._available_power[i])
                  + (self._battery_charge_power_by_inverter_AC_clamping[i]
                     if that array is not None else 0)
And it returns the tuple
  (expected_waste_wh, first_surplus_idx, last_surplus_idx, horizon_idx)
  where first/last surplus indices are the first/last i with waste_w > 0
  AND is_value_full(battery_charge[i]).
```

### AC-3: Global energy envelope bounds total extra-load energy added

```gherkin
Given the refactored surplus block reaches its trigger gate
  (expected_waste_wh >= SOLAR_WASTE_TRIGGER_THRESHOLD_WH and
   first_surplus_idx is not None)
When energy_to_be_spent is computed
Then it equals
  min(SOLAR_WASTE_CONFIDENCE_FACTOR * expected_waste_wh,
      solar_in_probe_wh + battery_drain_budget_wh)
Where solar_in_probe_wh is sum over [probe_window_start,
  probe_window_end] of max(0, -_available_power[i]) * dt[i] / 3600
And battery_drain_budget_wh = max(0, current_charge
  - battery.get_value_empty()
  - SOLAR_WASTE_SAFETY_MARGIN_FRACTION * usable_capacity)
And on big-sun days this typically resolves to
  0.7 * expected_waste_wh (the additive form is the upper bound for
  pathological tiny-waste cases; battery drain safety is policed by
  Layer 3 per-slot, not by the envelope itself).
```

### AC-4: Probe window starts at slot 0 and ends at last_surplus_idx

```gherkin

Then probe_window_start = 0
And probe_window_end = last_surplus_idx
(So overnight / early-morning slots are reachable from the constraint
```

### AC-6: Per-slot battery-charge guard preserves the safety floor inside `adapt_repartition`

```gherkin
Given MultiStepsPowerLoadConstraint.adapt_repartition is called with
  init_energy_delta >= 0.0 and bat_charge_traj is not None
When use_battery_floor evaluates True
Then at call entry forward_min = np.minimum.accumulate(
  bat_charge_traj[::-1])[::-1]
And inside the slot loop, for each slot i,
  max_delta_power = max(0, forward_min[i] - battery_min_wh)
                  * 3600 / power_slots_duration_s[i]
And if the proposed delta_power exceeds max_delta_power, the placement
  is clamped to the largest command from
  _get_lower_consign_idx_for_power that fits, or skipped if even the
  smallest command exceeds max_delta_power
And after each placement,
  state_delta_wh = delta_power * power_slots_duration_s[i] / 3600
  bat_charge_traj[i:last_slot+1] -= state_delta_wh
  forward_min[i:last_slot+1]     -= state_delta_wh
And forward_min stays a valid running minimum of bat_charge_traj across
  the multi-pass loop (num_passes ∈ {1, 2}) because both passes only
  decrement.

Given the same function is called with init_energy_delta < 0.0
  (reclaim path used by _prepare_battery_segmentation cap loop)
Or with bat_charge_traj is None
When use_battery_floor evaluates False
Then behavior is byte-identical to today's baseline (no guard applied).
```

### Task 4 — Implement `_find_next_dusk_idx` (AC-1)

[`custom_components/quiet_solar/home_model/solver.py`](../../custom_components/quiet_solar/home_model/solver.py)

Add the helper method on `PeriodSolver`:

```python
def _find_next_dusk_idx(self) -> int | None:
    num_slots: int = len(self._durations_s)
    pv: npt.NDArray[np.float64] = self._solar_production
    durations: npt.NDArray[np.float64] = self._durations_s
    time_slots: list[datetime] = self._time_slots

    have_seen_sun = False
    i = 0
    while i < num_slots:
        if float(pv[i]) >= _DUSK_THRESHOLD_W:
            have_seen_sun = True
            i += 1
            continue
        if not have_seen_sun:
            i += 1
            continue
        if time_slots[i].astimezone(tz=None).hour < _DUSK_EARLIEST_LOCAL_HOUR:
            i += 1
            continue
        sustained_s = 0.0
        j = i
        while j < num_slots and float(pv[j]) < _DUSK_THRESHOLD_W:
            sustained_s += float(durations[j])
            if sustained_s >= _DUSK_SUSTAIN_S:
                return i
            j += 1
        i = j + 1
    return None
```

```python
                constraints_evolution, constraints_bounds, probe_window_start, is_off_grid,
            )
            if constraints:
                _LOGGER.info("solve: surplus pre-discharge envelope %.0f Wh", energy_to_be_spent)
                while True:
                    solved, has_changed, energy_to_be_spent = self._constraints_delta(
                        energy_to_be_spent,
                        constraints,
                        constraints_evolution,
                        constraints_bounds,
                        actions,
                        probe_window_start,
                        probe_window_end,
                        # NEW: thread Layer 3 plumbing
                        battery_min_wh=self._battery.get_value_empty() + safety_margin_wh,
                    )
                    if solved or not has_changed:
                        break
```

## Dev Notes

- **TDD ordering matters.** Task 1 writes the failing regression test
  BEFORE Task 7-9 land Layer 3. If the test passes without Layer 3
  (i.e., today's `power_headroom` already covers the gap), the
  implementer must revert Layer 3 from this story and reduce scope to
  the four under-firing reasons (horizon, DC-clamp waste, smart budget,
  probe window start). The TDD step is the self-correcting empirical
  gate.
- **Architecture: domain-pure invariant preserved.** All changes are
  in `home_model/`; no `homeassistant.*` imports introduced. The
  abstract-signature change is contained within the `home_model/`
  domain layer.
- **Forward-min identity** within a single `adapt_repartition` call:
  `forward_min[i] = min(bat_charge_traj[i:last_slot+1])` is preserved by
  both passes of the multi-pass loop because each placement subtracts
  the same `state_delta_wh` from every `bat_charge_traj[i:last_slot+1]`
  and every `forward_min[i:last_slot+1]` — a constant subtraction over
  the same slice preserves the running minimum.
- **Cross-call ownership**: `_constraints_delta` owns `bat_charge_traj`
  via the entry `.copy()`. `adapt_repartition` mutates in place by
  reference; the mutation is discarded after the next sim refresh
  reseeds the array. No aliasing is carried across constraint
  iterations.
- **Numpy typing**: use `npt.NDArray[np.float64]` consistently (not
  `np.ndarray`). The project's idiom — see existing `solver.py`
  signatures.
- **Lazy logging**: convert any remaining f-string log calls in the
  rewritten surplus block (current lines 1161-1273) to lazy `%s`
  formatting while you're there. Project rule, no trailing periods.
- **Constants location**: `SOLAR_WASTE_*` in `const.py`
  (user-visible-ish tuning); `_DUSK_*` in `solver.py` module-level
  (single user, internal). All non-user-facing in v1 — no
  config-flow / `strings.json` work.
- **No post-loop WARNING**: the safety invariant is locked by tests, not
  by a runtime check. If a test ever shows the invariant violated, the
  fix is to harden the guard, not to log-and-continue.
- **Why no per-slot `_battery_charge_power_by_inverter_AC_clamping[i]`
  math in the guard**: the trajectory `bat_charge_traj` is initialized
  from `_battery_get_charging_power()[1]`, which integrates ALL of: AC
  excess to battery, DC clamp to battery (DC-coupled, not-full slots),
  and discharge to UA/loads. So the trajectory is the ground-truth
  pre-load battery state. Adding a load reroutes `delta_power × dt /
  3600` of energy from the battery (or from solar that would have
  charged it) to consumption — net `−delta_power × dt / 3600` whenever
  the battery isn't at exact full. The edge case at exact full is
  over-conservative (we still subtract; the actual state change is
  zero), but the conservatism is on the safe side and can be optimized
  later if profiling shows real surplus left on the table.
