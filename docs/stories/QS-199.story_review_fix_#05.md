## Findings to fix

### [should-fix] S1 — align the drag gauge max and the snap-list max by construction
- Files:
  - `custom_components/quiet_solar/ui/resources/shared/qs-ring-duration-base.js:125` (`_clampMaxHours`) and `:143` (`_allowedHalfHours`)
  - the `maxHours` derivation (both default and non-default `targetHours`
    branches) in all four duration cards:
    `qs-radiator-card.js:~209/~212`, `qs-on-off-duration-card.js:~72`,
    `qs-climate-card.js:~695`, `qs-water-boiler-card.js:~734`
- Severity: should-fix
- Source: qs-review-edge-case-hunter (S1) + qs-review-blind-hunter (NH1) — same root cause
- Scope: residual of the round-4 ES1 source-clamp; the SAME divergence class, now triggered by fractional/edge configs instead of `>168`
- Description: The round-4 fix clamps `maxHours` and feeds one value to both the
  gauge math and the snap list, which fixed the `>168` desync. But "gauge max ==
  max draggable snap value" is still NOT guaranteed by construction. Three faces
  of the one class:
  1. **Fractional config** — `max_default_hours: 13.3`: `_clampMaxHours` returns
     `Math.min(13.3, 168) = 13.3`, the ring's 100% draws 13.3h, but
     `_allowedHalfHours` builds `0, 0.5, … ` and stops at the last 0.5-step `<=`
     cap (13.0), so the handle can only reach 13.0h → ~0.5h dead zone at the top
     of the ring. (Empirically: cap 13.3 → max snap 13.0; integer/0.5-multiple
     caps all match exactly.)
  2. **Runtime sensor** — the non-default-mode branch sets `maxHours = targetHours`
     straight from `_safeNumber(sDurationLimit)` with no 0.5 rounding AND no
     168 clamp, so (a) a fractional duration-limit sensor (13.3) reproduces the
     same dead zone, and (b) a pathological sensor value (e.g. 1e6) feeds the
     gauge and `_allowedHalfHours` unbounded — the `Number.isFinite` guard stops
     `Infinity` but there's no upper bound, so the snap-list `for` loop could be
     enormous. The 168 cap is currently asymmetric between the two branches.
  3. **Tiny positive config** — `0 < max_default_hours < 0.5` yields a snap list
     of exactly `[0]`: target permanently pinned at 0, undraggable, while the
     gauge renders a 0..n scale. (Degenerate tail of the same boundary; see N-fold.)
- Proposed fix (root-cause — close the class, don't patch a range): Normalize
  `maxHours` ONCE, on EVERY branch (default and non-default `targetHours`), to a
  snap-grid-aligned value before it feeds either the gauge or the snap list:
  - round to the nearest (or floor to the) 0.5 multiple so the gauge max lands
    exactly on a snap step,
  - floor at 0.5 (so the list is never just `[0]`),
  - ceiling at 168 (kills the huge-array tail on both branches),
  - keep `Number.isFinite` as defense-in-depth.
  Put this in ONE shared helper (extend `_clampMaxHours` to also grid-align, and
  have every card + `_allowedHalfHours` consume its output) so the gauge and the
  snap list are mathematically guaranteed equal. This single change resolves
  faces (1), (2a), (2b), and (3).
- Tests: assert `gauge_max == max(snap_list)` for a representative set —
  `13.3`, `167.8`, `0.3`, `1e6`, `Infinity`, `12`, `24`, `168` — and that the
  non-default `targetHours` branch goes through the same normalization.

### [nice-to-have] N1 — single source for the `12` default fallback
- File: `custom_components/quiet_solar/ui/resources/shared/qs-ring-duration-base.js:125,143`
- Severity: nice-to-have
- Source: qs-review-blind-hunter (NH2)
- Description: `_clampMaxHours` falls back to `12` for non-positive/non-finite
  input, and `_allowedHalfHours` independently falls back to `12` for its own
  non-finite cap. The double-hardcoded magic number is harmless defense-in-depth
  today, but a future change to one default would silently desync the two
  helpers.
- Proposed fix: Define the `12` default once (module constant) and reference it
  from both helpers. Folds naturally into the S1 single-helper refactor.

### [nice-to-have] N2 — harden the concurrency test to cover the sweep
- File: `tests/test_ui_dashboard.py` (`test_async_update_resources_serialized_by_lock`, ~line 2017)
- Severity: nice-to-have
- Source: qs-review-edge-case-hunter (N2) + qs-review-blind-hunter (NH3)
- Description: The test patches out `_async_remove_orphan_temps`, so it only
  proves the COPY runs inside the lock, not the SWEEP. Production code DOES hold
  the lock around the sweep (verified by reading the source), so this is a
  test-coverage gap, not a live bug — but every one of the prior three
  regressions was exactly "the fix moved the bug just outside what the test
  covered," so the sweep belongs inside the asserted critical section.
  Separately, the module-level `_RESOURCE_UPDATE_LOCK` persists across tests in
  the same event loop; a future test that acquires it and fails to release could
  hang this test rather than fail fast.
- Proposed fix: (a) Let a non-mocked sweep participate in the serialization
  assertion (e.g. have both `fake_copy` and the real sweep bump/observe the
  `active`-invocation counter, asserting it never exceeds 1). (b) Ensure lock
  release / test isolation — assert `not _RESOURCE_UPDATE_LOCK.locked()` after
  each invocation, and/or reset the lock in a fixture teardown.
