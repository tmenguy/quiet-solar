# Story: Improve plan agent — adversarial planning

issue: 146

## Story

As a developer using the OpenCode workflow,
I want the create-plan phase to use adversarial multi-persona review (like the review-task phase does),
so that story artifacts are maximally unambiguous and complete before the implement phase begins.

## Acceptance Criteria

1. Given the create-plan agent template exists,
   When the plan agent runs,
   Then it follows a multi-phase protocol: (1) gather context, (2) draft the plan, (3) spawn 4 parallel adversarial reviewer sub-agents, (4) consolidate findings, (5) present to user, (6) iterate or finalize.

2. Given the create-plan agent needs adversarial reviewers,
   When the agent templates are rendered,
   Then 4 hidden sub-agent templates exist: `qs-plan-critic.md.tmpl`, `qs-plan-concrete-planner.md.tmpl`, `qs-plan-dev-proxy.md.tmpl`, `qs-plan-scope-guardian.md.tmpl` — each with a distinct personality and review lens.

3. Given the 4 plan reviewer sub-agents are spawned,
   When they execute,
   Then they run in parallel via 4 simultaneous Task tool calls from the orchestrator (same pattern as `qs-review-task`).

4. Given each reviewer returns findings,
   When findings are consolidated,
   Then the orchestrator deduplicates, classifies into `critical`/`redesign`/`improve`/`clarify`/`dismiss`, and presents a structured report to the user.

5. Given the user reviews findings,
   When the user approves the plan,
   Then all accepted findings are incorporated into the story artifact before committing.

6. Given the user wants another round,
   When the user says "revise",
   Then the plan is updated and re-submitted to the 4 reviewers (max 3 total rounds).

7. Given the plan is finalized,
   When the agent commits,
   Then the commit, push, render-next-agent, and handoff steps remain unchanged from today's template.

8. Given the `qs-setup-task` or `qs-create-plan` agent renders plan reviewers,
   When rendering happens,
   Then `render_agent.py` is called 4 additional times to produce the hidden sub-agents in `.opencode/agents/`.

9. Given `cleanup_agents.py` runs at finish-task,
   When it cleans up,
   Then all `qs-plan-*-QS-<N>.md` files are also removed (already covered by existing `qs-*-QS-<N>` glob pattern).

  - Normalize all findings into unified format: source, category, finding, evidence, suggestion

  **Phase 6 — Present & Resolve**:
  - Show structured findings report with counts per category
  - Walk through `critical` and `clarify` items with user

## Dev Notes

### Architecture Constraints

- **Two-layer agent architecture**: The create-plan agent becomes an orchestrator (like review-task), spawning hidden sub-agents via the Task tool. Sub-agents are `mode: subagent`, `hidden: true`.
- **No new scripts needed**: `render_agent.py` already supports arbitrary `--phase` values — it just substitutes `{{VAR}}` placeholders in the corresponding `.tmpl` file. Adding 4 new templates is sufficient.
- **Cleanup already works**: `cleanup_agents.py` uses `qs-*-QS-<N>` glob, which matches `qs-plan-critic-QS-146.md` etc.
- **Hands-off paths**: Do NOT modify `_qsprocess/plan-reviewers/` (those are Claude Code's personas). The OpenCode sub-agent templates are independent files that draw inspiration from them but are self-contained.

### Personality Design — The 4 Reviewers

The personalities are chosen to create maximum productive tension:

1. **Critic** (pessimist) — Assumes the plan is wrong. Pure logic check, no codebase needed. Catches hand-waving, contradictions, missing failure modes.
2. **Concrete Planner** (pragmatist) — Demands implementation-level specificity. Has codebase access to verify file paths. Catches vague tasks.
3. **Dev Proxy** (anxious implementer) — Simulates the next agent in the pipeline. Has project rules. Catches rule violations, missing dev notes, untestable ACs.
4. **Scope Guardian** (minimalist) — NEW persona not in Claude Code's set. Replaces External Challenger (which requires `--plan` input). Challenges scope creep, over-engineering. Has issue body to compare against.

The Scope Guardian replaces External Challenger because:
- External Challenger only activates when `--plan` is provided (rare in OpenCode flow)
- Scope creep is the #1 failure mode in AI-generated plans
- Having a dedicated "less is more" voice balances the Critic's "you missed X" tendency
