# Story QS-147: Fix flow when working on dev environment itself vs the code

issue: 147
branch: "QS_147"
Status: ready-for-dev

## Story

As a developer working on the Quiet Solar dev tooling,
I want the create-plan phase to detect whether I'm working on the real code or on the dev environment itself,
so that the correct implement agent is rendered with appropriate permissions and quality gates.

## Context

Currently, `create-plan` always renders `qs-implement-task`, which has edit access to `custom_components/quiet_solar/**` and `tests/**` and runs the full quality gate (`python scripts/qs/quality_gate.py`). When working on the dev environment itself (scripts, templates, tests under `_qsprocess_opencode/` and `scripts/qs_opencode/`), this is wrong: the agent should NOT touch the real code/tests, and the quality gate should only run the dev-environment tests.

**Additionally**, rendered per-task agent files (`qs-*-QS-*.md`) are currently **not gitignored** and have been accidentally committed and merged to main (e.g., the QS-146 agent files). These ephemeral files must be excluded from git tracking.

## Acceptance Criteria

1. **Given** per-task agent files are rendered into `.opencode/agents/`,
   **When** a developer runs `git add` or `git status`,
   **Then** files matching `qs-*-QS-*.md` are ignored by git (via `.gitignore`).

2. **Given** the QS-146 per-task agent files were previously committed to main,
   **When** this fix is applied,
   **Then** they are removed from git tracking (but the static agents `qs-setup-task.md` and `qs-release.md` remain tracked).

3. **Given** a new template `qs-implement-setup-task.md.tmpl` exists,
   **When** it is rendered,
   **Then** its edit permissions allow `_qsprocess_opencode/**` and `scripts/qs_opencode/**` but deny `custom_components/quiet_solar/**` and `tests/**`.

4. **Given** the `qs-implement-setup-task` agent runs its quality gate,
   **When** it executes tests,
   **Then** it runs `source venv/bin/activate && pytest _qsprocess_opencode/tests/ -v` instead of `python scripts/qs/quality_gate.py`.

5. **Given** `render_agent.py` is invoked with `--phase implement-setup-task`,
   **When** it looks up the template,
   **Then** it finds `qs-implement-setup-task.md.tmpl` in `PHASE_TEMPLATES`.

6. **Given** `next_step.py` receives `--phase implement-setup-task` as the finishing phase,
   **When** it computes the transition,
   **Then** it transitions to `review-task` (same downstream as `implement-task`).

7. **Given** the create-plan template,
   **When** it is rendered for a dev-environment issue,
   **Then** it uses a `{{IMPLEMENT_PHASE}}` placeholder (defaulting to `implement-task`) that can be overridden to `implement-setup-task` at render time via `--extra IMPLEMENT_PHASE=implement-setup-task`.

## Dev Notes

### Gitignore fix
The `.opencode/.gitignore` currently only ignores `node_modules`, `package.json`, etc. Per-task agent files (`qs-*-QS-*.md`) are not ignored. The pattern must be specific enough to NOT match the two static agents:
- `qs-setup-task.md` — static, tracked
- `qs-release.md` — static, tracked

The pattern `qs-*-QS-*.md` is perfect because static agents don't have `-QS-` in their names. Best location: `.opencode/agents/.gitignore` with one line: `qs-*-QS-*.md`.

The QS-146 files already on main must be removed with `git rm --cached` so they stop being tracked while staying on disk (the cleanup_worktree script handles physical deletion).

### Architecture constraints
- Templates live in `_qsprocess_opencode/agent_templates/`
- `render_agent.py` maps phase names to template files via `PHASE_TEMPLATES` dict
- `next_step.py` maps phase transitions via `PHASE_TRANSITIONS` dict
- The `{{VAR}}` substitution is simple string replacement — no conditionals
- The create-plan template has 7 phases including adversarial review with 4 plan sub-agents

### Key design decision: `{{IMPLEMENT_PHASE}}` placeholder
The create-plan template currently hardcodes `implement-task` in Phase 7. By replacing with `{{IMPLEMENT_PHASE}}`, the decision is made at **create-plan render time** by `setup-task`, not at execution time. This is clean because:
- No runtime detection logic needed in the create-plan agent
- The template system already supports `--extra KEY=VALUE`

### Quality gate for dev environment
The `implement-setup-task` agent uses a lighter gate:
- `source venv/bin/activate && pytest _qsprocess_opencode/tests/ -v`
- No ruff/mypy/translations on the main codebase
- The bash permission list drops `python scripts/qs/quality_gate.py*` and `python scripts/qs/*`
