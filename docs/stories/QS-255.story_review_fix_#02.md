### [must-fix] Auto contradiction does not latch for cars that cannot use percent constraints — per-cycle re-flag + notification spam
- File: `custom_components/quiet_solar/ha_model/car.py:891-907`
- Severity: must-fix
- Source: qs-review-edge-case-hunter
- Description: In the `manual=False` branch, `check_charger_assignment_contradiction`
  sets `_car_api_stale = True` and (by design) skips the inferred flags. The only
  cycle-surviving state is `_enter_stale_percent_mode(time)`, which is gated on
  `can_use_charge_percent_constraints()`. For a car where that guard is `False`
  (`car_is_invited`, or `car_battery_capacity` is `None`/`<= 0`), nothing durable
  latches. On the next `_update_car_api_staleness` cycle in `CAR_STALE_MODE_AUTO`,
  `self._car_api_stale = raw_stale` (car.py:700-701) resets the flag when the API
  publishes fresh timestamps with wrong values (exactly the incident scenario),
  `is_car_effectively_stale` returns `False`, the periodic gate re-opens, and the
  check fires again — re-sending the "detected on … disagrees" notification every
  state cycle. This is a regression: the pre-PR code's unconditional
  `_car_api_inferred_home/plugged = True` survived across cycles and suppressed
  re-trigger. The test suite only exercises percent-capable cars
  (`car_battery_capacity: 60000`); no test drives the no-percent auto path through
  two cycles.
- Proposed fix: Add a cycle-surviving latch for the auto path that is independent
  of percent mode. Recommended shape: dedup the contradiction handling on
  `_car_api_stale_since` (or a dedicated `_contradiction_notified`-style marker) —
  when a prior contradiction was recorded and the same assignment + contradiction
  persist, skip the re-flag/re-notification (or at least the notification) instead
  of treating each cycle as a fresh contradiction. Clear the latch when the
  assignment changes, the contradiction resolves, or the car exits stale state.
  Do NOT set `car_api_stale_percent_mode` for cars failing the guard, and do NOT
  set the inferred flags on the auto path (that is the core D1 design). Add a test:
  auto-attached car with `can_use_charge_percent_constraints() == False` (e.g.
  invited or zero-capacity), API reporting fresh-but-contradicting data, driven
  through two `_update_car_api_staleness` cycles — assert exactly one notification
  and no inferred flags.

### [must-fix] `get_best_car`: possible `None` deref on stale stored car name
- File: `custom_components/quiet_solar/ha_model/charger.py:2866-2874`
- Severity: must-fix
- Source: qs-review-coderabbit (Critical)
- Description: When the user-originated `car_name` marker refers to a car that no
  longer exists, `self.home.get_car_by_name(...)` returns `None` and the next line
  `car.get_user_originated(USER_ORIGINATED_CHARGER_NAME)` raises
  `AttributeError`. Pre-existing deref, but the line was touched by this PR.
- Proposed fix:
  ```diff
  -                if car.get_user_originated(USER_ORIGINATED_CHARGER_NAME) == FORCE_CAR_NO_CHARGER_CONNECTED:
  -                    car = None
  +                if car is not None and (
  +                    car.get_user_originated(USER_ORIGINATED_CHARGER_NAME) == FORCE_CAR_NO_CHARGER_CONNECTED
  +                ):
  +                    car = None
  ```
  Add a test where the charger's stored `car_name` marker names a car absent from
  `home._cars` and assert `get_best_car` does not raise.

### [should-fix] Direct `_user_originated` dict write bypasses constant and API
- File: `custom_components/quiet_solar/ha_model/charger.py:3099`
- Severity: should-fix
- Source: qs-review-coderabbit (Major)
- Description: `self._boot_car._user_originated["charger_name"] = self.name`
  hardcodes the marker literal and writes the private dict directly, bypassing
  both `USER_ORIGINATED_CHARGER_NAME` and `set_user_originated()` — the exact
  contract this PR's constants migration establishes.
- Proposed fix:
  ```diff
  -                    self._boot_car._user_originated["charger_name"] = self.name
  +                    self._boot_car.set_user_originated(USER_ORIGINATED_CHARGER_NAME, self.name)
  ```
  Verify `set_user_originated` has no side effects beyond the dict write that
  would change boot behavior; if it does (e.g. persistence scheduling), keep the
  semantics identical and note it.
