## Findings to fix

### [must-fix] tz-naive `asked_for_reset_user_initiated_state_time` crashes the cooldown check
- File: `custom_components/quiet_solar/ha_model/bistate_duration.py:766`
- Severity: must-fix
- Source: qs-review-coderabbit
- Description: The cooldown branch computes
  `time - self.asked_for_reset_user_initiated_state_time` with no tz
  normalization. `home_model/load.py` restores this field via
  `datetime.fromisoformat(...)` with no tzinfo fix, so a legacy or
  hand-edited tz-naive stored value raises `TypeError` when subtracted
  from the tz-aware `time`. The PR added coercion for
  `external_user_initiated_state_time` (review fix #01) but missed this
  sibling field.
- Proposed fix: Coerce `asked_for_reset_user_initiated_state_time` to
  UTC (`.replace(tzinfo=pytz.UTC)` when `tzinfo is None`) before the
  subtraction, mirroring the existing coercion pattern for
  `external_user_initiated_state_time`. Prefer coercing once at the
  restore/consumer boundary rather than at every use. Add a test with a
  tz-naive stored value exercising the cooldown path.

### [must-fix] `execute_command` override interception can still phantom-ack a suppressed relaunch
- File: `custom_components/quiet_solar/ha_model/bistate_duration.py:503-528`
- Severity: must-fix
- Source: qs-review-coderabbit
- Description: When a stale `running_command` is retried under an
  active override via `force_relaunch_command()`, the interception
  branch (1) returns `True` if the entity state already matches
  `override_state` — downstream `AbstractDevice.force_relaunch_command()`
  treats `True` as a real execution and acks/re-anchors the pre-override
  command (phantom ack); and (2) falls through to
  `execute_command_system()` when the entity state is unavailable —
  resending a conflicting command against the user override. This
  reopens the "don't fight the user / no phantom ack" contract the PR
  enforces at the `launch_command` drop point.
- Proposed fix: Make the interception suppress without acking: handle
  override suppression before `force_relaunch_command()` calls
  `execute_command()` (e.g. reuse `is_command_suppressed_by_override`
  there and drop the running command the same way `launch_command`
  does), or introduce a distinct non-acked "suppressed" outcome instead
  of overloading `True`. The unavailable-state branch must NOT fall
  through to `execute_command_system()` while the override is active.
  Add tests: (a) stale running command + entity matching override state
  → no ack, no counter increment; (b) entity unavailable under override
  → no transport call.

### [should-fix] Causality guard compares `state.last_changed` without tz-naive coercion
- File: `custom_components/quiet_solar/ha_model/bistate_duration.py:742-762`
- Severity: should-fix
- Source: qs-review-blind-hunter
- Description: The new causality guard compares
  `state.last_changed <= self.last_command_execution_time` directly.
  Real HA states carry aware-UTC `last_changed`, but every sibling path
  in this PR coerces defensively; a naive `last_changed` would raise
  `TypeError` inside `check_load_activity_and_constraints`.
- Proposed fix: Coerce `state.last_changed` to UTC before the
  comparison (local variable; do not mutate the HA state object),
  matching the defensive coercion used elsewhere. Add a test with a
  fake state whose `last_changed` is tz-naive.

### [should-fix] Future-dated stored override never dropped at restore
- File: `custom_components/quiet_solar/ha_model/bistate_duration.py:173-201`
- Severity: should-fix
- Source: qs-review-edge-case-hunter
- Description: `use_saved_extra_device_info` drops a stored override
  only when `age_s > 3600.0 * self.override_duration`. A stored
  `external_user_initiated_state_time` ahead of `datetime.now(pytz.UTC)`
  (clock skew, NTP correction, restore from a machine whose clock was
  ahead) yields negative `age_s`, so the poison override is retained
  and re-arms the loop the story is trying to kill.
- Proposed fix: Also drop when the timestamp is clearly in the future,
  e.g. `age_s < -tolerance` with a small tolerance (e.g. 60s), clearing
  the same four override fields. This matches the documented
  conservative "drop rather than keep poison" intent. Add a test with a
  stored timestamp `now + 1h`.

### [should-fix] Weak solver test assertions for the hold-off window
- File: `tests/test_solver.py:2686-2698`
- Severity: should-fix
- Source: qs-review-coderabbit
- Description: `all(cmd.is_off_or_idle() for cmd in in_window)` passes
  vacuously when `in_window` is empty, so it does not prove the solver
  emitted any idle coverage before `boundary_floor`. `on_seconds` is
  then accumulated until `end_time`, letting runtime scheduled after
  `daily.end_of_constraint` satisfy the 3h assertion — a solver that
  misses the noon deadline but back-loads energy later would still
  pass.
- Proposed fix: Assert `in_window` is non-empty (real hold-off prefix
  present), and clamp the `on_seconds` accumulation to the daily
  constraint's `end_of_constraint` so the 3h runtime must land before
  the deadline.

### [nice-to-have] `USER_OVERRIDE_SCORE_OFFSET` comment understates the lower-order sum
- File: `custom_components/quiet_solar/home_model/constraints.py:41-51`
- Severity: nice-to-have
- Source: qs-review-blind-hunter
- Description: The comment claims the lower-order terms sum to
  "~2.0001e13" with a "5x margin"; the actual worst case is roughly
  `1e13 (type term) + 1e13 (user term) + 1e12 + ~1e6 ≈ 2.1e13`, a
  ~4.7x margin. Dominance still holds (asserted in tests), but the
  inline figure is stale.
- Proposed fix: Correct the comment to the accurate worst-case sum and
  margin (the test remains the source of truth for the invariant).

### [nice-to-have] Degraded-override check resolves the wrong constraint
- File: `custom_components/quiet_solar/ha_model/bistate_duration.py:467-486`
- Severity: nice-to-have
- Source: qs-review-edge-case-hunter (+ qs-review-blind-hunter perf note)
- Description: `is_command_suppressed_by_override` uses
  `get_current_active_constraint(time)`, which returns the FIRST active
  constraint — not necessarily the override constraint. A chained
  non-override constraint that is active and non-mandatory can satisfy
  `not ct.is_mandatory and command.is_off_or_idle()` and let a
  conflicting off/idle command through even though a different
  constraint drives the load. (Mirrors the pre-existing
  `execute_command` nuance, so behavior is at least consistent.)
- Proposed fix: Resolve the override constraint explicitly by its
  originator key (`load_info[CONSTRAINT_ORIGINATOR_KEY] ==
  CONSTRAINT_ORIGINATOR_USER_OVERRIDE`) instead of taking the first
  active constraint; short-circuit when the command is not off/idle to
  avoid the lookup entirely. Apply the same resolution in the
  `execute_command` interception branch if it shares the nuance.

### [nice-to-have] Causality guard skipped when `state.last_changed is None`
- File: `custom_components/quiet_solar/ha_model/bistate_duration.py:749`
- Severity: nice-to-have
- Source: qs-review-edge-case-hunter
- Description: The guard requires `state.last_changed is not None`; if
  a backing entity exposes `last_changed = None`, the guard is skipped
  and a genuinely stale post-restart state is classified as a fresh
  user override — the exact failure class QS-256 targets.
- Proposed fix: Treat `last_changed is None` (while
  `last_command_execution_time` is set) as "cannot prove freshness" →
  suppress the override classification (conservative direction), with a
  debug log. Add a test for the None branch.

### [nice-to-have] No deterministic override-vs-override score tie-break
- File: `custom_components/quiet_solar/home_model/constraints.py:344-356`
- Severity: nice-to-have
- Source: qs-review-edge-case-hunter
- Description: Two simultaneously-live USER_OVERRIDE constraints (e.g.
  a hold-off restored from storage coexisting with a freshly detected
  override before the detection block wipes constraints) both receive
  the identical `+USER_OVERRIDE_SCORE_OFFSET`; cluster dedup then falls
  back to lower-order terms with no override-aware winner. Fix #01
  resolved this at documentation level only; "fix all" upgrades it to a
  code-level tie-break.
- Proposed fix: Add a stable secondary tie-break within the override
  tier (e.g. a small additive term derived from `start_of_constraint`
  recency so the NEWER override wins, scaled to stay strictly below the
  next-order term so it cannot perturb non-tie ordering). Assert the
  tie-break and the preserved dominance invariant in
  `tests/test_constraints.py`.
