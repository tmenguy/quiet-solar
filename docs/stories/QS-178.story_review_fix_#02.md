### [must-fix] Floor-clamp skip/lookup uses absolute power instead of delta
- File: `custom_components/quiet_solar/home_model/constraints.py:1608-1637`
- Severity: must-fix
- Source: qs-review-coderabbit + qs-review-edge-case-hunter

#### Description

`max_delta_power_floor` bounds the **incremental drain** at a slot, but
the clamp's skip check and `_get_lower_consign_idx_for_power` lookup
compare it against **absolute** `power_consign` values. When
`current_command_power > 0`, valid intermediate placements are wrongly
skipped.

Reproducer: `existing_commands[i].power_consign = 1500W`,
`power_sorted_cmds = [500, 1500, 2000, 2500]`, floor budget
`max_delta_power_floor = 800W`. Original `delta_power = 2500 - 1500 =
1000 > 800` triggers clamp. Skip check `800 < 500` → False (don't
skip). Lookup `_get_lower_consign_idx_for_power(cmds, 800)` finds j=0
(cmds[0]=500 ≤ 800). `delta_power = 500 - 1500 = -1000 < 0` → continue.
But the safe placement was `cmds[2]=2000` (delta=500 ≤ 800).
Opportunity lost.

#### Proposed fix

```python
piloted_offset = (
    possible_power_piloted_delta if is_current_empty_command else 0.0
)
allowed_base_power = (
    current_command_power + max_delta_power_floor - piloted_offset
)
if allowed_base_power < power_sorted_cmds[0].power_consign:
    continue
j = self._get_lower_consign_idx_for_power(
    power_sorted_cmds, allowed_base_power
)
```

The existing post-lookup `if delta_power <= 0: continue` correctly
handles the post-pick safety check.

#### Tests to add

`tests/test_constraints_pre_discharge_guard.py`:
- `test_guard_picks_intermediate_step_when_current_command_nonzero`:
  - existing `power_consign = 1500W`, sorted cmds `[500, 1500, 2000, 2500]`,
    `max_delta_power_floor = 800W`.
  - Assert chosen command is `cmds[2]=2000W` (delta=500 ≤ 800), NOT
    skip and NOT `cmds[0]` (which would yield negative delta).

### [must-fix] Open-ended `bat_charge_traj` / `forward_min` decrement past `last_slot`
- File: `custom_components/quiet_solar/home_model/constraints.py:1652-1662`
- Severity: must-fix
- Source: qs-review-coderabbit

#### Description

Physical battery charge carries forward across slot boundaries. A
placement at slot `i` decrements the trajectory for ALL slots ≥ `i`,
not just `[i, last_slot]`. The current `bat_charge_traj[i:last_slot+1]`
slice leaves the tail (`slots > last_slot`) stale relative to physical
reality. Subsequent constraints in the SAME `_constraints_delta` call
(before the cross-call reseed at line 879) inherit the stale tail and
may permit placements that would breach the floor in reality.

#### Proposed fix

```python
# Open-ended slices: physical battery state carries forward across
# the constraint's last_slot boundary into all later slots.
bat_charge_traj[i:] -= state_delta_wh
if use_battery_floor and forward_min is not None:
    forward_min[i:] -= state_delta_wh
```

Slots outside `[first_slot:last_slot+1]` in `forward_min` were
uninitialized before this fix; after the open-ended decrement they
remain uninitialized + a constant, still never read by the current
loop. (Nice-to-have #11 — `np.full_like(..., np.inf)` — closes that
defensiveness gap.)

#### Tests to add

`tests/test_constraints_pre_discharge_guard.py`:
- `test_open_ended_decrement_protects_downstream_constraint_floor`:
  - Build a 2-constraint scenario in a single `_constraints_delta`
    call: constraint A spans `[2, 4]`, constraint B spans `[6, 8]`.
  - Force A to place enough at slot 2 that the physical battery
    trajectory at slots `≥ 6` dips toward the floor.
  - With the open-ended decrement: B's Layer-3 check sees the
    correctly-decremented `bat_charge_traj[6:]` and refuses an unsafe
    placement.
  - Without it (the prior `[i:last_slot+1]` slice): B sees stale
    tail and would mistakenly allow the placement (test should fail
    if the open-ended slice is reverted — paired RED/GREEN).

### [must-fix] Refresh `battery_charge` before surplus block trigger gate
- File: `custom_components/quiet_solar/home_model/solver.py:1311-1321 / 1345`
- Severity: must-fix
- Source: qs-review-coderabbit + qs-review-edge-case-hunter (nice-to-have rating)

#### Description

Between the last `battery_charge` refresh (~line 1216-1223) and the
surplus-block read at line 1345
(`_compute_expected_solar_waste(battery_charge)` plus
`np.max(battery_charge[probe_window_start:probe_window_end+1])`), the
second `_allocate_constraints(..., always_use_available_only_power=True)`
mutates `_available_power_no_battery`. The local `battery_charge` is
the pre-allocation snapshot.

Consequences:
- Slots that were "full" in the stale trajectory but would not be full
  post-allocation are still flagged in `full_mask`, inflating
  `expected_waste_wh`.
- `max_future_charge_wh` from the stale array may be inflated, then
  must-fix #4 needs to shrink it after the refill loop.

#### Proposed fix

Immediately after `_allocate_constraints(..., always_use_available_only_power=True)`
and BEFORE `_compute_expected_solar_waste(battery_charge)`:

```python
# Refresh battery trajectory: the second _allocate_constraints pass
# absorbed surplus, so battery_charge from the earlier refresh is
# stale relative to the post-allocation surplus picture.
_refreshed = self._battery_get_charging_power(
    existing_battery_commands=battery_commands
)
battery_charge = _refreshed[1]
```

(Coordinate with must-fix #4: the refreshed `battery_charge` becomes
the basis for the pre-shrink `max_future_charge_wh` AND the post-shrink
recomputation.)

### [must-fix] Recompute `max_future_charge_wh` AFTER the refill shrink
- File: `custom_components/quiet_solar/home_model/solver.py:1346 + post-shrink`
- Severity: must-fix
- Source: qs-review-blind-hunter

#### Description

`max_future_charge_wh = float(np.max(battery_charge[probe_window_start : probe_window_end + 1]))`
is computed against the un-shrunk `probe_window_end = last_surplus_idx`.
The refill-shrink loop narrows `probe_window_end`. The second
`min(...)` at line 1408-1415 re-uses `battery_drain_budget_wh` without
recomputing peak charge over the narrower window. When peak charge
sits in the cut tail, the drain budget overstates head-room actually
available to placements inside the shrunk window.

#### Proposed fix

After the refill-shrink loop completes (after should-fix #2's Option A
exclusion lands):

```python
# After shrink: recompute peak charge over the actual placement window
# so battery_drain_budget_wh reflects only head-room reachable from
# inside [probe_window_start, probe_window_end].
if probe_window_end >= probe_window_start:
    max_future_charge_wh = float(
        np.max(battery_charge[probe_window_start : probe_window_end + 1])
    )
    battery_drain_budget_wh = max(
        0.0,
        max_future_charge_wh
        - self._battery.get_value_empty()
        - safety_margin_wh,
    )
```

#### Tests to add

`tests/test_solver_pre_discharge.py`:
- `test_drain_budget_recomputed_over_shrunk_window`:
  - Scenario: refill-shrink narrows `probe_window_end` AND peak charge
    sits in the cut tail (`bat_charge_traj` rises sharply between
    shrunk end and `last_surplus_idx`).
  - Capture `energy_to_be_spent` passed to `_constraints_delta`.
  - Assert it does NOT include the tail-peak head-room (i.e., it is
    bounded by the in-window peak, not the global peak).
