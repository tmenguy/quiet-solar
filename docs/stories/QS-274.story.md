# QS-274: Show charge-target origin (manual / calendar / person forecast) on the car card

## Problem

On the car card, Magali (non-technical household member) cannot tell
**where the current charge target / finish time came from** — whether
she entered it manually, it came from a calendar event, or it was
derived from a person forecast. Today the card's "Mode" indicator
collapses both manually-entered times and calendar-driven times into a
single `"Scheduled"` type, so two of the three origins are
indistinguishable.

## Goal

Make the charge-target origin coherent in the backend and visible on the
car card:

1. **Backend coherence:** stamp `CONSTRAINT_ORIGINATOR_KEY` consistently
   at the constraint-creation sites the card distinguishes (manual,
   calendar/agenda, person), using named constants.
2. **Card Mode icon:** drive an origin-aware icon from
   `get_charge_type()` — rabbit for as-fast-as-possible, hand for
   manual, calendar for calendar, person for person forecast — keeping
   all existing error / solar / state icons.
3. **Origin-responsive context line:** add a **new** sensor whose string
   adapts to the active constraint origin, and render it in the card's
   context row (replacing what that row shows). The existing
   person-forecast sensor is left untouched.

The finish-time popup is intentionally **not** changed (dropped after
review — see Adversarial Review Notes, finding 5/B).

### Charge-type detection (`get_charge_type(return_charge_errors=True)`, `ha_model/charger.py`)

**Charger-error suppression.** `get_charge_type()` takes
`return_charge_errors: bool = True`. When `True` (default — existing
callers `get_car_charge_type()` and `convert_auto_constraint_to_manual_if_needed()`
unchanged) it still short-circuits to `Faulted` / `No Power To Car` /
`Not Plugged`. When `False`, those error short-circuits are skipped and
the method returns the *actual* charge type — this is what the new origin
string helper needs (we never want a charger-error string in the origin
line).

**As-fast branch — user vs automation.** When `ct.as_fast_as_possible`:

- `ct.load_info and ct.load_info.get(CONSTRAINT_ORIGINATOR_KEY) == CONSTRAINT_ORIGINATOR_USER_OVERRIDE`
  → `CAR_CHARGE_TYPE_MANUAL_AS_FAST_AS_POSSIBLE` (the user pressed
  force-charge; the force constraint at `charger.py:3419` carries
  `CONSTRAINT_ORIGINATOR_USER_OVERRIDE` + `from_user=True`).
- else → `CAR_CHARGE_TYPE_AS_FAST_AS_POSSIBLE` (an automation/person-ASAP
  constraint forcing as-fast to meet a target).

**Deadline branch — backward-compatible precedence** (legacy/in-flight
person constraints carry only `{"person": name}` with no originator, so
the `"person"` check stays first):

1. `ct.load_info and "person" in ct.load_info` →
   `CAR_CHARGE_TYPE_PERSON_AUTOMATED` (unchanged).
2. `elif ct.load_info and ct.load_info.get(CONSTRAINT_ORIGINATOR_KEY) == CONSTRAINT_ORIGINATOR_AGENDA`
   → `CAR_CHARGE_TYPE_CALENDAR`. The `ct.load_info and …` guard is
   **mandatory** — without it a constraint with `load_info=None` raises
   `AttributeError` on `None.get(...)`.
3. `else` (manual-tagged or untagged / `load_info=None`) →
   `CAR_CHARGE_TYPE_MANUAL`.

New constants in `const.py`: `CAR_CHARGE_TYPE_MANUAL = "Manual"`,
`CAR_CHARGE_TYPE_CALENDAR = "Calendar"`,
`CAR_CHARGE_TYPE_MANUAL_AS_FAST_AS_POSSIBLE = "Manual As Fast As Possible"`.

**Remove `CAR_CHARGE_TYPE_SCHEDULE` entirely** (user decision). The
deadline branch now returns Manual / Calendar / Person Automated, so the
constant is dead. Blast radius (all must be cleared):
`const.py` definition, the `charger.py` import + usage, the
`qs-car-card.js` `"Scheduled": "mdi:clock-outline"` icon entry, and the
`CAR_CHARGE_TYPE_SCHEDULE` imports/assertions in `tests/test_charger_methods.py`
and `tests/ha_tests/test_charger.py`. **Do NOT touch**
`WallboxChargerStatus.SCHEDULED` (`charger.py:25`) or
`tests/test_wallbox_fallback.py` — that `"Scheduled"` is an unrelated
wallbox *hardware status*, a different concept.

`convert_auto_constraint_to_manual_if_needed()` (`car.py:1151`) branches
on `CAR_CHARGE_TYPE_PERSON_AUTOMATED`, which is preserved — this consumer
is intentionally unaffected and is verified by test.

### New origin sensor + origin-responsive context line

Add a **new** sensor `qs_car_charge_origin` (new
`SENSOR_CAR_CHARGE_ORIGIN = "qs_car_charge_origin"` constant in
`const.py`, registered in `sensor.py` next to `qs_car_person_forecast`).
Register it with **`key=SENSOR_CAR_CHARGE_ORIGIN`** (mirroring the
person-forecast sensor, whose `key == SENSOR_CAR_PERSON_FORECAST`) so the
dashboard template's `ha.get("qs_car_charge_origin")` resolves — the
`ha_entities` dict is keyed by the description `key`. Its `value_fn`
calls a new helper `get_car_charge_origin_readable_string()` in
`ha_model/car.py`.

The **existing** `qs_car_person_forecast` sensor and
`get_car_person_readable_forecast_mileage()` are **left unchanged** (kept
for the raw forecast string / history / automations).

**Signature / data access.** `get_car_charge_origin_readable_string()` is
a method on `QSCar` that reads
`self.charger.get_charge_type(return_charge_errors=False)` (the car
already holds the charger reference, as `get_car_charge_type()` does at
`car.py:1139–1143`). Passing `return_charge_errors=False` is essential —
the origin line must never show a charger-error state. It takes no
external arguments. It is **pure / CPU-only** — it only reads in-memory
constraint/load state; its callees `get_charge_type()` and
`get_readable_next_target_date_string()` are themselves I/O-free (no
`hass` calls, no executor jobs), so the helper is safe inside a sync
sensor `value_fn`.

It returns, based on `(type, ct) = self.charger.get_charge_type(return_charge_errors=False)`:

| case | line content |
| --- | --- |
| `type == PERSON_AUTOMATED` and a person is resolvable | `f"Forecasted from {get_car_person_readable_forecast_mileage()}"` → `"Forecasted from <Person>: <km>km <HH:MM>"` (e.g. `"Forecasted from Magali: 30km 14:30"`); time is the person's predicted leave time |
| `type == PERSON_AUTOMATED` but no resolvable person (`self.current_forecasted_person is None`) | `"No proper Forecast"` |
| `type == CALENDAR` | `"Calendar · <target-time>"` (constraint target time only — no event title) |
| `type == MANUAL` | `"Manually set to <target-time>"` |
| `type == AS_FAST_AS_POSSIBLE` | `"Automatically forced to charge as fast as possible"` |
| `type == MANUAL_AS_FAST_AS_POSSIBLE` | `"Manual as fast as possible charge"` |
| any other type (Solar / Solar Priority / Target Met / Not Charging / none) | if `self.current_forecasted_person is not None` → `get_car_person_readable_forecast_mileage()` (today's forecast string); else → `"No proper Forecast"` (the new helper returns this itself for the no-person case — it does **not** delegate the empty case, so the existing `get_car_person_readable_forecast_mileage()` is left untouched) |

The Person row reuses the existing helper's native string prefixed with
`"Forecasted from "`. The Calendar / Manual lines build `<target-time>`
from the active constraint via
`get_readable_next_target_date_string(for_small_standalone=True)`
(already used by `get_car_charge_time_readable_name()`, `car.py:1156`).
**Caveat:** that formatter returns `"%H:%M"` only when the target is
within ~1 day; for further-out targets it returns the two-line
`"%m-%d\n%H:%M"`. So the rendered string may be e.g. `"Calendar · 06-25\n14:30"`.
The Calendar / Manual coverage tests must construct a **near-term**
`end_of_constraint` to deterministically assert the `HH:MM` form.
With `return_charge_errors=False`, charger-error states (Faulted / No
Power / Not Plugged) never reach this switch — they fall through to the
"other type" branch only via `Not Charging`/`Solar`, never as an error
string.

**Person-first invariant:** detection of `PERSON_AUTOMATED` relies solely
on the presence of a `"person"` key in `load_info`, so the design assumes
**no non-person constraint ever puts a `person` key in `load_info`** —
otherwise an agenda/manual constraint would be shadowed as Person.

**Orphaned-person reachability:** `PERSON_AUTOMATED` is keyed off the
constraint's stamped `load_info["person"]` (set at creation), while the
`"No proper Forecast"` branch reads the car's **live**
`self.current_forecasted_person` — independent state. The branch is
reached when a person-tagged constraint outlives a now-`None`
`current_forecasted_person` (person entity removed, forecast cleared, or
leave-time elapsed). The coverage test builds an active deadline
constraint with `load_info={"person": name}` **and** forces
`car.current_forecasted_person = None`.

**Icon/text consistency:** the card's Mode icon (from the `car_charge_type`
sensor) and this context line (from `qs_car_charge_origin`) both recompute
from the same `get_charge_type()` on each update cycle, so they stay in
agreement; any sub-cycle skew during a transition is a non-issue for a
status card.

Card change (`ui/resources/qs-car-card.js`) — three exact spots:
1. **Read** the new entity near `:831` (next to
   `sPersonForecast = this._entity(e.person_forecast)`):
   `sChargeOrigin = this._entity(e.charge_origin)`.
2. **Compute** the display string near `:1194` (mirroring the existing
   `personForecastStr`/`forecastDisplay` validity handling).
3. **Render** at `:1418` — the load-bearing edit. Today the row is
   `<div class="forecast-row">Forecast: ${forecastDisplay}</div>`
   (`sChargeType` for the icon is in scope at `:827`).
   **Replace** its binding with the new origin string (drop the
   `Forecast: ` prefix) and pick the leading icon from `sChargeType`
   (already in scope). This is a **replace, not append**: the
   `.forecast-row` must render the new `charge_origin` string instead of
   `qs_car_person_forecast`, so the old sensor is no longer shown in this
   row (it remains registered for history/automations but is not
   surfaced on the card, avoiding a duplicate row).

### Translations

The `CAR_CHARGE_TYPE_*` values are raw sensor states consumed directly by
JS — they are **not** translatable `strings.json` entries today (no
`state:` block), so no charge-type strings are added. The origin
context-line text is built as plain Python strings, consistent with the
existing `get_car_person_readable_forecast_mileage()`. The only
`strings.json` change is the **friendly name** for the new
`qs_car_charge_origin` sensor; then run
`bash scripts/generate-translations.sh`. Never edit `translations/en.json`
directly. (Note: pre-existing duplicate-key typo `qs_car_person_forecast:`
at `strings.json:879` — flag only / out of scope **unless** the
translations gate or `generate-translations.sh` trips on the duplicate
once the new adjacent key is added, in which case fix the typo as part of
this change.)

   `convert_auto_constraint_to_manual_if_needed` still fires only for

5. The car-card Mode icon shows: rabbit (as-fast and manual-as-fast),
   hand (manual), calendar (calendar), `mdi:account-clock` (person
   forecast); all existing error / solar / state icons still render;
   `mdi:auto-fix` and the `Scheduled` key are no longer referenced.
6. A new `qs_car_charge_origin` sensor exists; its string adapts to
   origin per the table (incl. the as-fast / manual-as-fast / orphaned-
   person cases); the "other type" case returns today's person-forecast

7. The card renders the new origin string in the context row with an
   origin-matched icon, and no longer shows the `Forecast:` prefix.
8. New sensor friendly name in `strings.json`; `en.json` regenerated, not
   hand-edited.
9. Full quality gate passes: `python scripts/qs/quality_gate.py`
   (pytest 100% coverage + ruff + mypy + translations).
