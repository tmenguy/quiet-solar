# QS-255 ‚Äî Charger-assignment contradiction check: manual vs auto split (D1)

- **Issue:** #255
- **Branch:** `QS_255`
- **Next phase:** `implement-task` (product code under `custom_components/quiet_solar/`)

## Problem

Incident 2026-06-04 night: the Zoe was **automatically** attached to
`wallbox 3 portail` via plug-time correlation (`get_best_car` score 2532)
while its Renault cloud API was frozen (`home=False`). The periodic
contradiction check `check_manual_assignment_contradiction()`
(`ha_model/car.py`, method near line 822) then:

1. **Bug 1 ‚Äî misleading log**: logged *"Car Zoe manually assigned to
   charger ‚Ä¶"* although no human acted. The method has two call sites ‚Äî
   `user_set_selected_charger_by_name` (always manual) and the periodic
   `_update_car_api_staleness` (manual **or** automatic) ‚Äî but only one
   wording.
2. **Bug 2 ‚Äî manufactured evidence**: set `_car_api_inferred_home = True`
   and `_car_api_inferred_plugged = True` regardless of assignment origin.
   These override raw API readings in `is_car_home()` / `is_car_plugged()`,
   which feed back into `get_car_score()` ‚Äî the incident log shows the match
   score jumping 2532 ‚Üí 102535 right after flagging. Circular: "we guessed
   it's this car ‚Üí therefore the car is home ‚Üí therefore the guess scores
   higher". A wrong auto-match would be entrenched by its own contradiction
   flag.

The stale flagging + stale-percent (estimate) mode + notification themselves
behaved correctly and are kept for both origins.

## Design (D1, agreed in issue)

| | Manual assignment | Automatic attach |
|---|---|---|
| Flag stale + stale-percent mode + notification | ‚úÖ | ‚úÖ |
| Notification body | "Your {name} assignment contradicts its car API data ‚Äî using estimated charge data" | "Your {name} was detected on {charger_name} but its API disagrees ‚Äî check the car card" |
| Notification title | `Warning: {name}` (unchanged) | `Warning: {name}` (unchanged) |
| Log wording | "manually assigned to charger" | "auto-attached to charger" |
| Set `_car_api_inferred_home/plugged = True` | ‚úÖ user's word is ground truth | ‚ùå never ‚Äî identity is only a heuristic |

Why inferred flags are manual-only: a car can only be auto-matched with API
`plugged=True` + (home OR plug-time correlation), so `inferred_plugged` is
always redundant at auto-attach time and `inferred_home` is exactly the
missing evidence being manufactured. The cable proves *a* car is present,
never *which* car. Manual assignment is different: the user's explicit action
IS ground truth, so the inferred flags (and the score boost they imply) are
justified; the marker is cleared on physical unplug and when another charger
claims the car, which bounds any damage from a mistaken manual pick.

### Manual-origin detection (periodic call site)

Only two user entry points exist, both leaving durable `_user_originated`
markers (persisted across reboots, re-linked at boot, cleared on unplug /
other-charger claim); automatic `attach_car()` touches neither:

```python
is_manual = (
    self.get_user_originated("charger_name") == self.charger.name
    or self.charger.get_user_originated("car_name") == self.name
)
```

- `get_user_originated(key)` (`home_model/load.py:173`) returns the stored
  value or `None` when unset ‚Äî `None == <real name>` is `False`, so unset
  markers are safe.
- The sentinels `FORCE_CAR_NO_CHARGER_CONNECTED` / `CHARGER_NO_CAR_CONNECTED`
  cannot equal real device names, so the **name-equality itself** excludes
  them ‚Äî no extra sentinel-handling code is added (the note is explanatory
  only).
- The two detection forms are equivalent by construction: the literal
  `manual=True` at the user entry point holds *by definition* (we are inside
  the user action ‚Äî independent of the marker-write ordering trap at the
  cross-call), and the markers are the durable trace of that same action for
  the periodic path. This rationale goes in the method docstring.

## Acceptance criteria

**AC1 ‚Äî Manual contradiction keeps full behavior**
Given the check is invoked with `manual=True`
When the car API raw readings report `home=False` or `plugged=False`
Then the car is flagged stale (`_car_api_stale=True`), stale-percent mode is
entered when `can_use_charge_percent_constraints()` is true (pre-existing
guard, verified independent of the inferred flags ‚Äî no new branching),
`_car_api_inferred_home=True` and `_car_api_inferred_plugged=True` are set,
the log reads "manually assigned to charger", and the notification body is
"Your {name} assignment contradicts its car API data ‚Äî using estimated
charge data" with title `Warning: {name}`.

**AC2 ‚Äî Auto contradiction never sets inferred flags**
Given the check is invoked with `manual=False`
When the car API raw readings report `home=False` or `plugged=False`
Then the car is flagged stale, stale-percent mode is entered under the same
pre-existing guard, the log reads "auto-attached to charger", the
notification body is "Your {name} was detected on {charger_name} but its API
disagrees ‚Äî check the car card" with title `Warning: {name}`, and
`_car_api_inferred_home` / `_car_api_inferred_plugged` remain `False`.
("disagrees" rather than "reports it away": the trigger is home=False OR
plugged=False, so an "away" claim would be wrong for the plug-only case.)

**AC3 ‚Äî Periodic call site detects origin**
Given `_update_car_api_staleness` reaches the contradiction block
(`self.charger is not None` guard)
When either durable marker matches (`is_manual` expression above, computed
inside the non-None guard)
Then the check is called with `manual=True`; with neither marker matching,
`manual=False`.

**AC4 ‚Äî Manual entry point passes the literal**
Given `user_set_selected_charger_by_name` resolves a real charger
Then it calls the check with `manual=True` (pure rename + appended kwarg; the
`datetime.now(tz=pytz.UTC)` argument and imports are unchanged), and a test
pins that the entry point invokes the check with `manual=True`.

**AC5 ‚Äî Guards unchanged**
The `FORCE_NOT_STALE` override and the already-stale / stale-percent early
returns behave identically for both origins (no notification spam, no
behavior change).

**AC6 ‚Äî Rename complete (code scope)**
`check_manual_assignment_contradiction` no longer appears in any `.py` file
(production + tests); new name `check_charger_assignment_contradiction` with
required keyword-only `manual: bool`. Historical story docs
(`docs/stories/QS-82.story.md`) are immutable records and are explicitly
excluded.
