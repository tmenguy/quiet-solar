### [must-fix] M1 ‚Äî `--fix` + concurrent ruff format / ruff lint write race

- File: `scripts/qs/quality_gate.py` ‚Äî `main()` cheap-gate `ThreadPoolExecutor`
- Severity: must-fix
- Source: qs-review-edge-case-hunter
- Description: When the gate is invoked as
  `python scripts/qs/quality_gate.py --full --fix`, both
  `check_ruff_format(fix=True)` and `check_ruff_lint(fix=True)` run
  **concurrently** inside the same `ThreadPoolExecutor`. `ruff format`
  rewrites files for style; `ruff check --fix` rewrites the same files
  for lint fixes. The two subprocesses race on identical paths under
  `custom_components/quiet_solar/` ‚Üí lost edits, truncated writes, or
  non-deterministic results across runs. This is a **new regression**
  introduced by AC3's parallelization (the old code ran the two gates
  serially).
- Proposed fix: In `main()`, when `args.fix` is truthy, **serialize**
  the two write-mutating gates (`check_ruff_format`, `check_ruff_lint`)
  so they run one after the other on the same thread, while
  `check_mypy` and `check_translations` can still run concurrently with
  them. Concretely:
  - Split `cheap` into `cheap_readonly = (check_mypy,
    check_translations)` and `cheap_write = (check_ruff_format,
    check_ruff_lint)`.
  - In a single pool: submit the two readonly futures, plus **one**
    composite future that calls `check_ruff_format(fix=args.fix)` then
    `check_ruff_lint(fix=args.fix)` sequentially and returns the two
    result dicts.
  - Wait, flatten, then sort by `gate_order` as today.
  - When `args.fix` is `False`, the two gates are pure reads ‚Äî they can
    stay parallel; gate the serial wrapper on `args.fix`.
- Test additions in `tests/test_quality_gate.py`:
  - `test_fix_mode_serializes_ruff_gates` ‚Äî patch both `ruff format` and
    `ruff lint` to record their start/finish timestamps in a thread-safe
    list; run `main()` with `--fix`; assert `ruff_format.finish <=
    ruff_lint.start` (or vice versa) ‚Äî the windows do not overlap.
  - `test_no_fix_mode_keeps_ruff_gates_parallel` ‚Äî same instrumentation,
    no `--fix`; assert `ruff_format` and `ruff_lint` windows DO overlap
    (concurrency preserved when safe).

### [must-fix] M2 ‚Äî `_has_xdist()` probes wrong interpreter

- File: `scripts/qs/quality_gate.py:85-93` (`_has_xdist`)
- Severity: must-fix
- Source: qs-review-coderabbit
- Description: `_has_xdist()` calls `importlib.util.find_spec("xdist")`
  in the **orchestrator process**, but pytest is launched via
  `VENV_PYTHON` (the project venv's `python`). If the orchestrator is
  run from a different interpreter (e.g. the user's pyenv shim, or a
  system Python that happens to be the agent harness's runtime),
  detection lies in both directions: xdist installed in venv but not in
  orchestrator ‚Üí `-n` silently dropped (slow run, no warning); xdist
  installed in orchestrator but not in venv ‚Üí `-n` added, pytest crashes
  with "unrecognized arguments: -n auto".
- Proposed fix: Replace the in-process `find_spec` probe with a
  subprocess probe against `VENV_PYTHON`. Use `_run` (no shell), pass
  `-c "import importlib.util, sys; sys.exit(0 if
  importlib.util.find_spec('xdist') else 1)"`, time-bound it via the
  existing `_run` plumbing (it already supports `cwd`; the cmd is fast,
  no timeout needed). Cache the result as today via `_HAS_XDIST_CACHE`.
  Update the docstring to say "checks `VENV_PYTHON`'s site-packages,
  not the orchestrator's".
- Test additions in `tests/test_quality_gate.py`:
  - `test_has_xdist_probes_venv_python` ‚Äî patch `_run` to record the
    cmd; call `_has_xdist.cache_clear()` (or reset
    `_HAS_XDIST_CACHE`); call `_has_xdist()`; assert the recorded cmd
    starts with `[VENV_PYTHON, "-c", ‚Ä¶]` and the `-c` body contains
    `find_spec('xdist')`.
  - `test_has_xdist_true_when_venv_returns_zero` ‚Äî patch `_run` to
    return `(0, "", "")`; assert `_has_xdist()` is `True`.
  - `test_has_xdist_false_when_venv_returns_nonzero` ‚Äî patch `_run`
    to return `(1, "", "")`; assert `_has_xdist()` is `False`.

### [should-fix] S1 ‚Äî `--cov-report=` ordering may silently disable term-missing

- File: `scripts/qs/quality_gate.py` ‚Äî `check_pytest()` cmd construction
  (around the `--cov-report=` empty override line)
- Severity: should-fix
- Source: qs-review-blind-hunter
- Description: The cmd currently appends `--cov-report=term-missing`
  **then** `--cov-report=` (empty). Per `pytest-cov`'s docs, an empty
  `--cov-report=` value disables **all** reports ‚Äî passed after
  term-missing, it can wipe the very report the downstream coverage
  parser depends on. The intent (override `pytest.ini`'s leftover html
  default) is better expressed by putting the empty override **first**
  to clear the inherited reports, then re-adding term-missing.
- Proposed fix: Reorder so `--cov-report=` (empty) precedes any
  positive `--cov-report=...` entry. Concretely, in `check_pytest()`'s
  cmd list, move the empty form to come right after `--cov=...` and
  before `--cov-report=term-missing`. Add a one-line code comment
  explaining the ordering invariant.
- Test additions in `tests/test_quality_gate.py`:
  - `test_cov_report_empty_precedes_term_missing` ‚Äî build the cmd via
    `check_pytest()` (or extract it via the existing fake-`_run`
    pattern); assert `cmd.index("--cov-report=") < cmd.index(
    "--cov-report=term-missing")`.

### [should-fix] S2 ‚Äî Parser drops `skipped`, `xfailed`, `xpassed` from final total

- File: `scripts/qs/quality_gate.py` ‚Äî `_parse_pytest_output` and
  `_stream_pytest` final-total computation (~`:367-373`)
- Severity: should-fix
- Source: qs-review-coderabbit + qs-review-edge-case-hunter
- Description: `final_total = counts["passed"] + counts["failed"] +
  counts["errors"]` omits `counts["skipped"]`. Worse, `_COUNT_RE`'s
  `passed|failed|errors?|skipped` whitelist drops `xfailed` and
  `xpassed` entirely, so suites with xfail markers report incomplete
  totals. The char-level tally counts `x` / `X` / `s` as "skipped",
  but the authoritative-summary path drops them ‚Äî the two parsing
  modes disagree on the same run.
- Proposed fix:
  - Extend `_COUNT_RE` to `(\d+) (passed|failed|errors?|skipped|
    xfailed|xpassed|deselected|warnings?)` (or at minimum add
    `xfailed`, `xpassed`, `skipped`).
  - Track these labels in the counts dict (collapse `errors`/`error`
    ‚Üí `errors`; keep `xfailed` and `xpassed` as separate keys).
  - Recompute `final_total = passed + failed + errors + skipped +
    xfailed + xpassed`.
  - Update the final stderr line to include skipped/xfailed/xpassed
    when non-zero (e.g. `done (N/M) | passed=P failed=F errors=E
    skipped=S xfailed=Xf xpassed=Xp`).
- Test additions in `tests/test_quality_gate.py` (extend
  `TestStreamPytestParser`):
  - New fixture `tests/fixtures/quality_gate/seq_q_with_skips.txt`
    with a summary line like `=== 3 passed, 2 skipped, 1 xfailed in
    0.50s ===`. Assert `passed=3, skipped=2, xfailed=1, final_total=6`.
  - `test_parse_with_xpassed` ‚Äî synthetic summary with `xpassed`;
    assert it's tracked.

### [should-fix] S3 ‚Äî `_COUNT_RE` not anchored ‚Äî future log lines could be misclassified

- File: `scripts/qs/quality_gate.py` ‚Äî `_COUNT_RE`
- Severity: should-fix
- Source: qs-review-blind-hunter
- Description: The regex `(\d+) (passed|failed|errors?|skipped)` is
  unanchored and matches anywhere on a line. A future log line like
  `"5 passed checks remaining"` (or a coverage missing-line range
  formatted by a plugin) would falsely match as a summary line and
  overwrite the authoritative count from the actual `=== ‚Ä¶ in N.NNs
  ===` summary.
- Proposed fix: Anchor the regex (or the summary-line detection step)
  to pytest's canonical summary shape. Options:
  - Match the trailing `in <duration>s` token: only treat a line as a
    summary if `re.search(r"in [\d.]+s\b", line)` is also true.
  - Or detect the surrounding `=== ... ===` envelope first (via
    `_SUMMARY_LINE_RE = re.compile(r"^={2,}.* in [\d.]+s.*={2,}$")`)
    and only run `_COUNT_RE` over matched lines.
- Test additions in `tests/test_quality_gate.py`:
  - `test_parser_ignores_non_summary_lines_with_passed_word` ‚Äî feed a
    line like `"5 passed checks remaining\n10 passed, 0 failed in
    1.23s\n"` and assert the parsed `passed == 10` (the legitimate
    summary), not 5.

### [should-fix] S4 ‚Äî `QS_QG_PYTEST_WORKERS` value not normalized/validated

- File: `scripts/qs/quality_gate.py` ‚Äî `_pytest_workers` (~`:107`)
- Severity: should-fix
- Source: qs-review-edge-case-hunter
- Description: The env var value is passed verbatim to `pytest -n
  <val>`. A leading space (`" 4"`), a typo (`"four"`), or a negative
  number (`"-1"`) all silently break pytest's argparse with an
  unfriendly error rather than the gate either rejecting or
  normalizing the value.
- Proposed fix: In `_pytest_workers`:
  1. Read the env var; strip whitespace.
  2. If empty or equal to `"0"` ‚Üí return `None` (serial).
  3. If equal to `"auto"` (case-insensitive) ‚Üí return `"auto"`.
  4. Else try `int(value)` ‚Äî if it's a positive integer, return
     `str(parsed_int)`. If not, write a one-line stderr warning
     (`[pytest] invalid QS_QG_PYTEST_WORKERS=<value>, falling back to
     "auto"`) and return `"auto"`.
- Test additions in `tests/test_quality_gate.py`:
  - `test_workers_strips_whitespace` ‚Äî set env to `" 4 "`; assert cmd
    contains `-n 4`.
  - `test_workers_invalid_value_warns_and_uses_auto` ‚Äî set env to
    `"four"`; capture stderr; assert cmd contains `-n auto` and a
    warning was emitted.
  - `test_workers_negative_value_falls_back_to_auto` ‚Äî set env to
    `"-1"`; assert cmd contains `-n auto` and a warning was emitted.

### [should-fix] S6 ‚Äî `ThreadPoolExecutor` exception handling kills the pipeline

- File: `scripts/qs/quality_gate.py` ‚Äî `main()` cheap-gate pool
- Severity: should-fix
- Source: qs-review-edge-case-hunter
- Description: If any one of the four cheap-gate functions raises an
  **unhandled** exception (e.g. `FileNotFoundError` when `ruff` is
  missing from the venv, or any subprocess plumbing crash),
  `f.result()` re-raises inside the list comprehension and bubbles out
  of `main()`. The pool's `__exit__` still waits for the other three
  but the traceback escapes ‚Äî no cache write, no failure summary, no
  pytest run. Previously (serial code) each gate produced
  `passed=False` and the run continued.
- Proposed fix: Wrap the `f.result()` call in a per-future
  try/except. On exception, synthesize a result dict
  `{"name": gate_name, "passed": False, "stderr": f"<exception>:
  {exc}", "stdout": ""}` and append it to results so the failure shows
  up in the standard `FAILED gates: ‚Ä¶` summary path. Use the
  `futures: dict[Future, str]` mapping to recover the gate name for
  each failed future.
- Test additions in `tests/test_quality_gate.py`:
  - `test_cheap_gate_exception_does_not_crash_main` ‚Äî patch one cheap
    gate (e.g. `check_mypy`) to raise `RuntimeError("boom")`; invoke
    `main()` with the other gates mocked to succeed; assert `main()`
    exits with code 1, stderr contains `"FAILED gates:"` and the
    failing gate name, and pytest still ran (or the synthesized result
    dict for the failure is in `results`).

### [should-fix] S11 ‚Äî `test_cheap_gates_run_concurrently` is wall-clock flaky

- File: `tests/test_quality_gate.py:581-616` ‚Äî
  `test_cheap_gates_run_concurrently`
- Severity: should-fix
- Source: qs-review-coderabbit + qs-review-blind-hunter
- Description: The test relies on `elapsed < sleep_s * 3` (i.e.
  `< 0.9 s` with `sleep_s = 0.3`) to assert that 4 cheap gates ran
  in parallel rather than serially. Under loaded CI runners or
  scheduler jitter, this threshold can be exceeded even when the
  gates DID run concurrently, producing flaky failures.
- Proposed fix: Replace the wall-clock assertion with a
  `threading.Barrier(4)` rendezvous. Each mocked cheap gate calls
  `barrier.wait(timeout=2.0)` before returning. If any gate ran
  serially, the barrier times out and the test fails deterministically;
  if all four ran in parallel, the barrier releases all of them at
  once. The remaining assertion can stay: total elapsed should still
  be bounded (use a generous `<5 s` ceiling as a safety net, not the
  primary signal).
- Test additions: rewrite of the existing test only.
