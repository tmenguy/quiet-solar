## Findings to fix

### [should-fix] TOCTOU `FileNotFoundError` on `TESTMON_DATA.stat()` in `check_impacted`
- File: `scripts/qs/quality_gate.py:1437`
- Severity: should-fix
- Source: qs-review-edge-case-hunter
- Description: `was_incremental = TESTMON_DATA.exists() and TESTMON_DATA.stat().st_size > 0`
  is the one filesystem probe in the new code that is NOT protected against the
  file vanishing between the `.exists()` check and the `.stat()` call. If a
  concurrent `--seed-testmon` / `--impacted` run (or another worktree sharing
  the path, or a mid-`_purge_testmon_db`) unlinks `.testmondata` in that window,
  `.stat()` raises `FileNotFoundError` and crashes the gate with a traceback
  instead of treating the run as non-incremental. The rest of the module
  deliberately tolerates exactly this race everywhere else — via `missing_ok=True`
  (`_purge_testmon_db`, the QS-283 A2 sidecar loop) and via the
  `FileNotFoundError`/`OSError` guards at lines ~172 and ~396 — so this single
  unguarded `.stat()` is an inconsistency with the project's stated
  single-writer/tolerate-vanish concurrency model.
- Proposed fix: Make the `was_incremental` probe race-safe. Preferred: wrap the
  `stat()` in a `try/except (FileNotFoundError, OSError)` that falls back to
  `was_incremental = False` (treat a vanished/unreadable baseline as
  non-incremental, matching the existing guards at lines ~172/~396). Equivalent:
  call `TESTMON_DATA.stat()` once inside the try and derive both existence and
  size from it, dropping the separate `.exists()` pre-check. Add a unit test that
  simulates the race (e.g. patch `Path.stat` to raise `FileNotFoundError` after
  `exists()` returns True, or unlink between the two calls) and asserts
  `check_impacted` treats it as non-incremental and does not raise.

### [nice-to-have] Vestigial `allow_self_heal` parameter on `check_impacted`
- File: `scripts/qs/quality_gate.py` (`check_impacted` signature)
- Severity: nice-to-have
- Source: qs-review-blind-hunter
- Description: The `allow_self_heal` keyword param is vestigial — no production
  caller passes `False`, and recursion is already prevented structurally by the
  retry calling `_run_impacted_pass` directly (not `check_impacted`). The
  parameter exists only to be toggled by a test.
- Proposed fix: Either drop the parameter and adjust the one test that toggles
  it to exercise the no-retry path another way (e.g. via `was_incremental` /
  select-all state), OR keep it but add a brief comment noting it is a
  test-only seam. Do not introduce new uncovered branches — whatever is kept
  must remain 100% covered.
