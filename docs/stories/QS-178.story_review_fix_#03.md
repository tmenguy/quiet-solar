### [must-fix] `_battery_get_charging_power` argument inconsistency between budget and floor-guard
- File: `custom_components/quiet_solar/home_model/solver.py:514, 444, 463`
- Severity: must-fix
- Source: qs-review-blind-hunter

#### Description

The surplus block at line 514 computes:
```python
battery_charge = self._battery_get_charging_power(
    existing_battery_commands=battery_commands
)[1]
```
…to derive `max_future_charge_wh` and `battery_drain_budget_wh`.

But `_constraints_delta` seeds the floor-guard trajectory via:
```python
# at entry (line 444):
bat_charge_traj = self._battery_get_charging_power()[1].copy()
# at per-placement reseed (line 463):
bat_charge_traj = self._battery_get_charging_power()[1].copy()
```
…with NO `existing_battery_commands` kwarg.

When `battery_commands` contains `CMD_FORCE_CHARGE` or
`CMD_GREEN_CHARGE_ONLY` slots, the internal sim defaults to
`CMD_GREEN_CHARGE_AND_DISCHARGE`, producing a LOWER charge trajectory.
The floor-guard therefore reads a different baseline from the one used
to size the envelope — placements can be:
- Skipped that the budget thought were feasible (over-conservative), OR
- Admitted when the budget assumed a higher peak that doesn't exist in
  the floor-check view (over-aggressive).

#### Proposed fix

Thread `battery_commands` into `_constraints_delta` as a new
keyword-only parameter, and pass `existing_battery_commands=battery_commands`
in both seeding calls:

```python
def _constraints_delta(
    self,
    energy_delta: float,
    constraints: list[tuple[BaseConstraint, float]],
    seg_start: int,
    seg_end: int,
    *,
    bat_charge_traj_seed: npt.NDArray | None = None,
    battery_min_wh: float = 0.0,
    battery_commands: list | None = None,  # NEW
) -> tuple[bool, bool, float]:
    ...
    # at entry:
    if bat_charge_traj_seed is not None:
        bat_charge_traj = self._battery_get_charging_power(
            existing_battery_commands=battery_commands
        )[1].copy()
    ...
    # at per-placement reseed (inside the has_changes branch):
    refreshed = self._battery_get_charging_power(
        existing_battery_commands=battery_commands
    )
    bat_charge_traj = refreshed[1].copy()
```

Caller side (surplus block in `solve()`): pass `battery_commands` through:
```python
self._constraints_delta(
    energy_to_be_spent,
    constraints,
    probe_window_start,
    probe_window_end,
    bat_charge_traj_seed=...,
    battery_min_wh=...,
    battery_commands=battery_commands,  # NEW
)
```

The reclaim/segmentation call site (where `bat_charge_traj_seed is None`)
passes `battery_commands=None` (or omits) since it doesn't use the
floor-guard.

#### Tests to add

`tests/test_solver_pre_discharge.py`:
- `test_constraints_delta_uses_battery_commands_for_floor_guard`:
  - Build a scenario where `battery_commands` includes a
    `CMD_FORCE_CHARGE` for slots 0-2.
  - Patch `_battery_get_charging_power` to record its calls.
  - Run `solve()` and assert that every call originated from
    `_constraints_delta` while inside the surplus block uses
    `existing_battery_commands=battery_commands`.

### [must-fix] Multi-pass skip applied unconditionally to all `adapt_repartition` call sites
- File: `custom_components/quiet_solar/home_model/constraints.py:1443-1454`
- Severity: must-fix
- Source: qs-review-blind-hunter

#### Description

Round-1 must-fix #2 added `if out_commands[i] is not None: continue` at
the top of the placement loop, justified as preventing pass-2 from
double-decrementing `bat_charge_traj` after pass-1 placed at the same
slot. But the skip runs for ALL `adapt_repartition` calls — including
the 49+ existing call sites that pass `bat_charge_traj=None`.

In multi-pass mode (`num_max_on_off` set → `num_passes=2`), pass-2
previously could revisit a slot placed in pass-1 to upgrade it (e.g.,
from `cmds[1]` to `cmds[2]`). The unconditional skip short-circuits
that legitimate upgrade path.

Risk: legacy multi-pass scenarios that relied on revisiting could now
silently produce sub-optimal placements. The change was not validated
against the broader test suite for upgrade-on-pass-2 regressions.

#### Proposed fix

Gate the skip on `use_battery_floor` so legacy behavior is preserved
when no trajectory is passed:

```python
# at the top of `for i in sorted_available_power:` (around line 1443):
if use_battery_floor and out_commands[i] is not None:
    continue
```

The double-decrement hazard only exists when `bat_charge_traj` is
being mutated — exactly when `use_battery_floor` is True. Outside
that scope, pass-2's upgrade path is restored.

#### Tests to add

`tests/test_constraints_pre_discharge_guard.py`:
- `test_multi_pass_upgrade_path_restored_when_no_battery_floor`:
  - Build a scenario with `bat_charge_traj=None`, `num_max_on_off=4`,
    `existing_commands=[CMD_AUTO_GREEN_ONLY(1500W), None, None]`,
    cmds `[500, 1500, 2500, 3500]`.
  - Force pass-1 to place at slot 0 (e.g., upgrade to 2500W).
  - Assert pass-2 successfully upgrades slot 0 to 3500W (final
    `out_commands[0].power_consign == 3500`).
- `test_multi_pass_skip_active_when_use_battery_floor`:
  - Same setup but with `bat_charge_traj` passed.
  - Assert pass-2 does NOT revisit slot 0 (existing round-1 test
    coverage; verify it still passes).

### [must-fix] Degenerate window `[probe_window_start, probe_window_start]` double-counts surplus
- File: `custom_components/quiet_solar/home_model/solver.py:1408-1417 + 1454`
- Severity: must-fix
- Source: qs-review-edge-case-hunter

#### Description

The refill-feasibility shrink does
`probe_window_end = max(probe_window_start, i - 1)`, intending to
exclude the break-slot. The degenerate-window guard at line 1454 uses
non-strict `probe_window_end >= probe_window_start`, so the
`[probe_window_start, probe_window_start]` window (from a
first-iteration break with `i = probe_window_start`) passes through.

Concrete trigger: `last_surplus_idx == probe_window_start` (single-slot
surplus region in the probe) AND `_available_power[probe_window_start]
* dur / 3600 <= -energy_to_be_spent` (break-slot's surplus alone
satisfies refill). Result: the break-slot's surplus is counted in
BOTH `nrj_to_recharge` (refill) AND `solar_in_shrunk_probe_wh`
(placement).

#### Proposed fix

Change the degenerate-window guard to strict `>`:

```python
# at line 1454 (degenerate-window guard):
if probe_window_end <= probe_window_start:
    # Refill consumed the entire probe, OR the only remaining slot was
    # the break-slot itself. Skip the surplus block.
    ...  # match surrounding control flow (likely `continue` or skip)
```

ALTERNATIVELY: drop the `max(probe_window_start, ...)` clamp and let
`probe_window_end = i - 1` go to `-1` so the strict `<` against
`probe_window_start = 0` collapses cleanly.

#### Tests to add

`tests/test_solver_pre_discharge.py`:
- `test_surplus_block_skips_single_slot_break_satisfy`:
  - Build a fixture with at least one pluggable load AND
    `last_surplus_idx == probe_window_start` (single-slot surplus
    in the probe), refill-satisfying surplus at that slot.
  - Patch `_constraints_delta` to record calls.
  - Run `solve()`.
  - Assert `len(surplus_calls) == 0` (degenerate window skipped).
  - Replaces / supplements `test_surplus_block_skips_degenerate_post_shrink_window`
    (which is vacuous per must-fix #6).

Recommended: **Option B** — Option A complicates the loop structure
for a bounded slack.

### [should-fix] AC-6 story text describes closed-window decrement
- File: `docs/stories/QS-178.story.md` AC-6 lines 145-148
- Severity: should-fix
- Source: qs-review-acceptance-auditor

#### Description / Fix

```
bat_charge_traj[i:last_slot+1] -= state_delta_wh
```

- Severity: nice-to-have

The dusk-finder ignores slots with `pv < threshold AND hour < EARLIEST_LOCAL_HOUR`,

intended bias).

Add a docstring note rather than fixing:

```python
def _find_next_dusk_idx(self) -> int | None:
    """Return the index of the first slot in the sustained low-pv tail.

    ...

    Note: dusk detection is biased LATER than reality when a sustained
    low-pv run starts before EARLIEST_LOCAL_HOUR. The pre-EARLIEST slots
    are skipped (not accumulated into the sustain budget), so the
    return value may overshoot the actual dusk transition by up to
    `EARLIEST_LOCAL_HOUR - run_start_hour` minutes. This bias widens the
    waste-counting horizon, which is the intended direction on
    big-sun days.
    """
```
