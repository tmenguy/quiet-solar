# QS-288 — Audit whole harness pipeline: find discrepancies and speed improvements

**Issue:** #288 · **Branch:** `QS_288` · **Story:** v2 (round-1 review folded in)

## Problem

The dev pipeline has grown to a large surface: 16 agents mirrored across
three harness directories (`.claude/`, `.cursor/`, `.opencode/` — 48
files; 7 phase agents + 9 reviewer sub-agents), 7 slash-command
fallbacks, `scripts/qs/` with 14 files (13 modules — including
`quality_gate.py` at ~77K and `spawn_session.py` at ~30K) plus
`launchers/` with 6 files (5 modules), 6 workflow docs (~1,170 lines),
the `docs/agents/` hierarchy, `CLAUDE.md`, and the retired `legacy/`
tree.

Two problems have accumulated:

1. **Discrepancies.** `scripts/qs/check_doc_drift.py` enforces
   *file-level co-modification* across harness mirrors, but nothing
   verifies *content alignment* — protocol text can silently drift
   between mirrors, between agents and `docs/workflow/*`, between slash
   commands and the agents they delegate to, and between adjacent
   phases' handoff contracts. Plan-time evidence: `harness.md` declares
   agent bodies *identical* across harnesses while
   `project-rules.md` (Harness sync) says handoff/session-spawn
   sections *legitimately differ* — the two contract docs contradict
   each other; and issue #288 itself was mislabeled by setup-task
   (`area:car/ui/config` on a dev-infra issue).
2. **Unmeasured latency.** The mechanical layer (`scripts/qs/*`) has
   never been timed and the phase protocols have never been analyzed
   for latency. Known suspect: `scripts/qs/context.py` makes two serial
   `gh` network calls on *every* agent startup. **Scope honesty:** this
   task delivers *static latency analysis plus mechanical-layer
   timings*; instrumented end-to-end profiling of a whole phase run is
   out of scope here and becomes a ranked follow-up proposal if the
   static analysis leaves big unknowns.

## Goal

Produce a **prioritized, evidence-backed audit report** covering
(1) discrepancies across the whole pipeline and (2) — above all —
**speed improvements** ranked by a computable score; apply
user-confirmed trivial text/doc fixes inline; create follow-up GitHub
issues for user-accepted big-ticket proposals.

**Disposition authority:** the implementer *proposes* every disposition
(`fixed-inline` / `follow-up` / `wontfix`); the **user decides**.

### Non-goals

- No script **behavior** changes in this task (no logic edits to
  `scripts/qs/*.py`).
- No protocol redesign — redesigns are proposals in the report, not
  changes.
- No new **shipped** tooling (e.g., a content-level harness-sync
  checker is a *proposal*, not a deliverable). Scratch, **non-committed
  audit scaffolding** (throwaway diff/timing wrappers) is explicitly
  allowed — audit scaffolding is not shipped tooling.
- No instrumented end-to-end phase profiling (candidate follow-up
  proposal, see C4).

## Design

### Deliverable

`docs/stories/QS-288.audit-report.md` — committed with the PR
(companion-artifact pattern already used for review-fix files).
Structure:

1. **Executive summary** — top findings, top 3 speed wins by score.
2. **Baseline measurements** — real timed runs; for every timing the
   **exact command line**, N=5 repeats with **median + min/max**,
   environment (machine, Python, venv state), `.testmondata`
   pre-existence, and **network-bound timings reported separately**
   from CPU-bound ones.
3. **Discrepancy findings** — ID `D-<n>`, severity (P0/P1/P2), evidence
   (`file:line` or diff excerpt), proposal, effort (S/M/L), disposition.
   Ends with an explicit **"not audited"** list for anything timeboxed
   out.
4. **Speed findings** — ID `S-<n>`, measured or estimated cost,
   frequency assumption, proposal, estimated saving, effort, **score**
   (see ranking model), disposition.
5. **Applied inline fixes** — finding IDs → commit short-SHAs
   (appended by T7 as commits land).
6. **Follow-up issues** — title + body draft per proposal; URL for each
   user-accepted one; an explicit note if zero were accepted.

### Ranking model (speed findings)

`score = (estimated seconds saved × estimated events/week) ÷ effort
weight`, with effort weights **S=1, M=3, L=9** and the events/week
assumption stated per finding (e.g., agent startups/week ≈ tasks/week ×
phases × sessions/phase). The report **shows the arithmetic** for every
S-finding; §4 is sorted by score.

### Method (execution order: A → C → B → D, speed before discrepancies)

**A — Baseline measurements.** Protocol: run on the **untouched tree
before any edits**; N=5 repeats per timing (plus one recorded **cold**
run where defined); record exact command lines in §2. "Cold" =`
first invocation in a fresh shell on the untouched worktree; for the
gate, cold additionally means missing `.testmondata` (record whether it
pre-existed; if a cold select-all baseline build happens, report it as
a **separate number**, not mixed into the warm figure). Prerequisites:
`gh` authenticated, network up — `context.py` timings are
network-bound and reported as such.

- `time python scripts/qs/context.py` — cold + 5 repeats
  (network-bound: two serial `gh` calls).
- `time python scripts/qs/next_step.py --next-cmd create-plan
  --work-dir "$PWD" --issue 288 --title "timing probe"
  --harness claude-code` — pinned invocation (all required args; writes
  `/tmp/qs_launch_288.sh`, a benign recorded side effect).
- `time python scripts/qs/check_doc_drift.py` — default run.
- Quality gate: `--impacted` warm; `--quick
  tests/qs/launchers/test_phase_mapping.py` (pinned small target); one
  full-gate reference timing plus a `--cache` hit — sanctioned, see
  Constraints.

**C — Speed audit (top priority, runs before B):**

- **C1 latency walk** over the **7 phase agents** (qs-setup-task,
  qs-create-plan, qs-implement-task, qs-implement-setup-task,
  qs-review-task, qs-finish-task, qs-release) **and the 9 reviewer
  sub-agents** (5 `qs-plan-*`, 4 `qs-review-*` — the fan-out
  multiplier): mandated startup reads (approx. token volume), serial
  subprocess/network calls, fan-out sizes; flag redundant or
  serializable work. Includes **static call-structure analysis of
  `release.py` and `create_pr.py`** (their `gh`/network call chains
  under the finish-task and release phases) — timing them is skipped
  because both have side effects.
- **C2 inner loop**: `quality_gate.py --impacted` — measure end-to-end
  wall time and pytest's own reported `in Xs` test time; the remainder
  is attributed to selection/coverage/diff-cover overhead. Finer split
  (optional): a **scratch, non-committed** line-timestamping wrapper
  keyed on the gate's emitted progress lines. If that's insufficient,
  per-stage instrumentation becomes a follow-up proposal — the gate
  does not time its own stages today and this task won't edit it.
- **C3 context discovery**: `gh` call latency in `context.py` /
  `next_step.py`; parallelization or caching proposals.
- **C4 ranking**: all speed proposals scored per the ranking model;
  top 3 in the executive summary. **Instrumented end-to-end phase
  profiling** is included in §4 as an explicit candidate follow-up
  (proposed or rejected with rationale).

**B — Discrepancy audit** (timeboxed: breadth-first across B1–B6,
prioritized depth; anything cut is listed in §3 "not audited"):

- **B1 mirror drift**: strip **YAML frontmatter only** — no body
  sections are pre-stripped — then pairwise-diff the full bodies of all
  agent triples (16 at plan time — **verify the count at implement
  time**). Classify every delta **by delta class** (grouped, with
  representative evidence — not per-hunk bookkeeping) against the
  declared contract, tagging each class as *legitimate-but-undeclared
  harness difference* (⇒ a doc-amendment finding) or *protocol drift*.
  The contract itself is contradictory — `harness.md` ("bodies are
  identical… only frontmatter differs, handoff mechanics isolated in
  Python") vs `project-rules.md` Harness sync ("harness-specific
  sections — session-spawn logic, handoff commands — legitimately
  differ") — record this as a **seed D-finding** and propose the
  reconciled contract wording. Known seed cases to classify (not
  discard): the "Commit and hand off" section length divergence in
  `qs-create-plan` (~36 lines claude / ~67 opencode / ~29 cursor); the
  "Code intelligence (LSP)" section present in cursor+opencode
  `qs-create-plan` but absent from the claude copy.
- **B2 agents ↔ docs**: cross-check each phase agent against
  `docs/workflow/phase-protocols.md`, `overview.md`,
  `adversarial-review.md`, `harness.md`, `project-rules.md`, and the
  `CLAUDE.md` tables. Includes auditing **setup-task's labeling
  instructions against the actual repo label taxonomy** (the #288
  mislabel's root cause).
- **B3 command parity**: each `.claude/commands/<phase>.md` vs its
  agent — does the fallback delegate faithfully?
- **B4 scripts**: read all 13 modules under `scripts/qs/` + the 5
  launcher modules — **explicitly including `release.py`,
  `create_pr.py`, `fetch_issue.py`** — for dead code, duplicated
  helpers, stale references, inconsistent conventions. Plus a
  **`legacy/` quarantine pass**: confirm nothing live references
  `legacy/`, and flag legacy docs that contradict the current protocol.
  Analysis only — no behavior edits.
- **B5 docs hierarchy** (justification: this audits
  `check_doc_drift.py`'s *effectiveness*, in scope via Goal 1 "stale or
  contradictory instructions"). Deterministic sample: all `docs/agents/`
  docs whose `covers:` targets fall under `scripts/qs/` or
  `docs/workflow/` (if any), **plus** the 5 docs with the oldest
  `last_verified` per `check_doc_drift.py --json`.
- **B6 phase-handoff walk** (inter-phase protocol mismatches): for each
  adjacent phase pair in setup→plan→implement→review→finish (+release),
  check that what phase N writes/hands off matches what phase N+1
  mandates reading — story-file headings, review-fix artifacts, PR
  state assumptions, "Adversarial Review Notes" round-trips,
  `next_step.py` payloads — including internal consistency of
  `phase-protocols.md` itself across phases.

**D — Fixes & follow-ups:**

- **Inline-fix threshold:** pure text/doc discrepancy fixes — stale doc
  references, `CLAUDE.md`/workflow-doc table corrections, small mirror
  re-syncs. Nothing that changes script behavior or redesigns a
  protocol.
- **Canonical-source rule (mirror re-syncs):** the winning text is the
  version **consistent with `docs/workflow/*`** (process authority per
  `project-rules.md`); if the docs don't decide it, **git history**
  does (most recent deliberate change wins over rot). **Any delta with
  semantic disagreement between mirrors is automatically `follow-up`,
  never `fixed-inline`.** The chosen canonical side is recorded per
  finding ID in §3.
- **T7 gate & size cap:** the user confirms the concrete inline-fix
  list (finding IDs) **before any fix commit** — same gating as T8. A
  fix exceeding ~30 changed lines per file or ~6 files per finding →
  `follow-up`. Bulk mirror re-syncs → a dedicated follow-up issue, not
  inline.
- Fixes land as separate commits *after* the report is written, one
  finding-ID batch per commit; **after each batch commit, T7 appends
  `D-n → <short-hash>` to report §5** and commits the report amendment.
- **Follow-up issues:** created via
  `python scripts/qs/create_issue.py --title … --body … --labels …`
  (repo convention; labels per the taxonomy verified in B2), after the
  user confirms the final list at implement time; each issue links back
  to the report; URLs recorded in §6.
- **#288 relabel:** propose `gh issue edit 288` swapping the wrong
  `area:*` labels for the correct dev-infra labels — executed only with
  user confirmation, alongside T8.

### Constraints

- Harness-sync rule: any agent-file fix edits all three mirrors
  (`check_doc_drift.py` enforces co-modification).
- **Full-gate benchmark carve-out:** the single Method-A full-gate
  timing is a one-off benchmark **explicitly requested by this story
  (AC2)** — sanctioned under the project-rules "explicit user request"
  carve-out, not an inner-loop diagnostic.
- **TDD waiver:** deliverables are markdown-only (report + doc-text
  fixes; no Python changes), so the implement phase's failing-test-first
  step does not apply. The local pre-commit gate is
  `quality_gate.py --impacted` before **each** commit (dev-infra fast
  path); the whole-repo full gate is owned by CI per the QS-276
  invariant.
- Prerequisites: `gh` authenticated and network available (baselines
  and T8 depend on it).
- The report is a point-in-time artifact under `docs/stories/` — not a
  durable doc under `docs/workflow/` — so no `covers:` frontmatter.

## Acceptance criteria

1. `docs/stories/QS-288.audit-report.md` exists, committed, with the
   six sections defined above.
2. §2 contains real timed numbers for the **pinned invocations** of
   `context.py`, `next_step.py`, `check_doc_drift.py`,
   `quality_gate.py --impacted` (warm), `--quick` on the pinned target,
   and one full-gate reference — each with exact command line, N=5
   median + min/max, environment, `.testmondata` state, and
   network-bound timings separated.
3. The mirror-drift audit covers **all agent triples** (16 at plan
   time — count verified at implement time) via frontmatter-only
   stripping and full-body pairwise diffs; every delta is classified
   **by delta class** with representative evidence; the
   `harness.md` ↔ `project-rules.md` contract contradiction is
   addressed as a finding.
4. B2–B6 each list findings **or** an explicit "no findings"
   statement: agents↔docs (incl. setup-task labeling), command parity,
   scripts (incl. `release.py`/`create_pr.py` and the `legacy/`
   quarantine pass), docs-hierarchy sample, and the **phase-handoff
   walk**. Anything timeboxed out appears in the §3 "not audited" list.
5. Every D-finding carries ID, severity, evidence, proposal, effort,
   disposition; every S-finding carries ID, cost, frequency assumption,
   estimated saving, effort, **score**, disposition (severity applies
   to D-findings only).
6. §4 shows the scoring arithmetic per S-finding and is sorted by
   score; the top 3 appear in the executive summary; instrumented
   end-to-end phase profiling appears as an explicit
   proposed-or-rejected entry.
7. All inline fixes: user-confirmed list, within threshold and size
   cap, canonical-source rule applied and recorded per finding, three-
   mirror co-edits where applicable, `check_doc_drift.py` clean and
   `--impacted` green before each commit; §5 maps finding IDs →
   commit SHAs.
8. For each **user-accepted** proposal (possibly zero) a follow-up
   issue exists via `create_issue.py` with its URL in §6; if zero were
   accepted, §6 records that explicitly.

## Task breakdown (speed before discrepancies, per review)

1. **T1 — Baseline measurements** (method A) on the untouched tree,
   before any edits; write report §2. _[report only]_
2. **T2 — Speed audit** (C1–C4): latency walk (7 phase agents + 9
   sub-agents), inner loop, context discovery, scored ranking; report
   §4. _[report only]_
3. **T3 — Mirror-drift audit** (B1): frontmatter-only normalization,
   full-body 3-way diffs, delta-class classification, seed cases +
   contract-contradiction finding; report §3. _[report only]_
4. **T4 — Agents↔docs, command parity, phase-handoff walk**
   (B2+B3+B6); report §3. _[report only]_
5. **T5 — Scripts, legacy, docs-hierarchy audit** (B4+B5); report §3.
   _[report only]_ — T3–T5 are breadth-first and timeboxed; cuts go to
   the "not audited" list.
6. **T6 — Assemble report**: write
   `docs/stories/QS-288.audit-report.md` §1–§6 (with §5/§6 as
   placeholders pending T7/T8). _[new file]_
7. **T7 — Apply inline fixes**: propose the fix list → **user
   confirms** → one finding-ID batch per commit (canonical-source rule,
   size cap, three-mirror co-edits) → append `D-n → <short-hash>` to §5
   after each batch. _[potentially: `.claude/agents/*`,
   `.cursor/agents/*`, `.opencode/agents/*`, `.claude/commands/*`,
   `docs/workflow/*`, `CLAUDE.md`, `docs/agents/*`]_
8. **T8 — File follow-up issues**: user confirms the list →
   `python scripts/qs/create_issue.py` per accepted proposal; propose
   the #288 relabel; record URLs in §6. _[gh only]_
9. **T9 — Doc maintenance**: run
   `python scripts/qs/check_doc_drift.py --paths <touched files>` on
   the *actual* T7 set; update covered docs or record `Doc-OK`.

**Doc-OK (plan-time):** the definite new files
(`docs/stories/QS-288.story.md`, `docs/stories/QS-288.audit-report.md`)
are point-in-time story artifacts not covered by any `docs/agents/`
doc. T7's exact file set is unknowable until the audit runs, so T9
mandates re-running the drift checker on the real touched set at
implement time.
