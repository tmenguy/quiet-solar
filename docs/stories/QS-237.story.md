## Context and root cause

The pool card's "Actual / Target Hours" big text displays
`targetHours = duration_limit` (= `qs_bistate_current_duration_h`,
i.e. the current active **constraint's** duration). When the user
drags the handle in `bistate_mode_default` and releases:

1. `_setNumber(default_on_duration, 12)` writes 12 to the
   `default_on_duration` number entity. HA's number-entity push
   updates this state IMMEDIATELY.
2. The **constraint** is NOT rebuilt synchronously. It picks up the
   new `default_on_duration` only on the next solver cycle (which can
   be tens of seconds away —
   `ha_model/bistate_duration.py:441 target_value =
   self.default_on_duration * 3600.0` is only re-evaluated on solve).
3. So `duration_limit` keeps reporting the **old** 8h.
4. On the next HA push, the pool card re-renders:
   - Big target text: reads `targetHours = duration_limit = 8` → shows
     "8h" — mismatched against the dragged value the user just
     committed.
   - Handle: still at 12 via the `_localTargetPct` 5-second hold.
   - Target color: shifts from the inline `colors.primary`
     (`#2196F3`, electric blue) used by `dragMove` to
     `var(--primary-color)` (the HA theme blue) used by the rendered
     `<span>`. Two different blues = "different shade" symptom.
5. After 5 seconds, `_localTargetPct` clears. If the constraint still
   hasn't rebuilt, the handle ALSO drops to 8 → full visual reset.

The five-line root cause: pool is the lone outlier in the
5-duration-card family. Radiator (line 207-214), water-boiler (line
729-744), on_off_duration (line 67-78), and climate all switch on
`isDefaultMode` and compute
`displayTargetHours = isDefaultMode ? defaultDuration : targetHours`,
then use that SINGLE value for both the handle position AND the big
text. Pool reads raw `duration_limit` for both — and they diverge the
moment the user changes `default_on_duration`. QS-235's family
consolidation didn't catch pool because pool's local internal shape
(`hasValidTarget` / `handleTargetHours` / `targetHours`) was
self-consistent — just diverged from the documented pattern.

## Acceptance criteria

### AC-1 — Source-of-truth unification

Pool's `_render()` computes
`displayTargetHours = isDefaultMode ? defaultDuration : targetHours`
and uses that **single** value for BOTH the handle position AND the
big "Actual / Target Hours" target span. After the change,
`targetHours` (= `duration_limit`) feeds ONLY the non-default branch
of `displayTargetHours`.

**Pinning**: structural test asserts the source contains
`isDefaultMode = poolMode === 'bistate_mode_default'` AND
`displayTargetHours = isDefaultMode ? defaultDuration : targetHours`
AND that `displayTargetHours` appears (a) in the `_buildRingHTML(...)`
call site and (b) inside the `.target-value` span's `_fmt(...)`.

### AC-2 — Target-value color via palette token

The `.target-value` big-text span's color attribute references the
`colors.primary` palette token via `${colors.primary}` (matching the
inline color used by `dragMove` and the family pattern in
water-boiler / radiator). The pre-QS-237 `var(--primary-color)` theme
variable is gone from that span.

**Pinning**: structural test asserts pool source contains
`${colors.primary};">${this._fmt(displayTargetHours)}h` AND does NOT
contain `var(--primary-color);">${this._fmt(`. The test pins the
**token reference** `colors.primary`, not the hex literal `#2196F3` —
the palette object owns the value, so a palette tweak doesn't break
the test.

> **Superseded by review-fix #01 (N6).** See
> `docs/stories/QS-237.story_review_fix_#01.md` — the span is rendered
> via `_fmt(displayTargetHours, false)` (un-rounded) so the committed
> display matches the live drag preview on the half-hour grid; the
> pinning regex tolerates this form. See also review-fix #02 (M1)
> for the JS-syntax-safety rule on the comment that documents this
> divergence inline.

### AC-3 — Mode-gated drag (behavior change, intentional)

`canDragHandle = isEnabled && isDefaultMode && displayTargetHours > 0`.
In any non-default pool mode (`bistate_mode_auto`,
`bistate_mode_exact_calendar`, `pool_winter_mode`) the
`<circle id="target_handle">` is NOT rendered. The pre-QS-237
`onBeforeCommit` silent-mode-switch hack is removed.

> **Superseded by review-fix #01 (S1) and review-fix #02 (N3).** See
> `docs/stories/QS-237.story_review_fix_#01.md` — the drag gate is
> two-term `canDragHandle = isEnabled && isDefaultMode` (the
> `displayTargetHours > 0` clause was removed to prevent drag-to-zero
> self-lockout). Then `docs/stories/QS-237.story_review_fix_#02.md`
> added a third defensive term `&& !!e.default_on_duration` so drag
> isn't wired when the dashboard template omits that entity key.
> The final shipped form is
> `canDragHandle = isEnabled && isDefaultMode && !!e.default_on_duration`.

**Migration impact** (user-acknowledged): users previously dragging
in auto/winter/exact_calendar mode silently switched the pool to
default mode on commit. After QS-237 they must switch the mode
explicitly via the existing mode pill first. This is the documented
family UX — radiator, water-boiler, on_off_duration, and climate all
behave this way.

**Pinning**: structural test asserts the source contains
`canDragHandle = isEnabled && isDefaultMode && displayTargetHours > 0`
AND does NOT contain `onBeforeCommit:`.

### AC-4 — Half-hour snap (UX harmonization)

The pool drag snap list is
`const allowedHours = this._allowedHalfHours(maxHours);`. Pre-QS-237
pool used `Array.from({ length: 25 }, (_, i) => i)` (integer 0..24).
This is a **UX harmonization with the family**, not a bug fix:
matches radiator (line 513), water-boiler (line 1320), on_off_duration
(line 323), climate. Justified by user decision #1 + the "fix for
everyone" mandate.

**Pinning**: structural test asserts the source contains
`this._allowedHalfHours(maxHours)` AND does NOT contain
`Array.from({ length: 25 }`.

### AC-5 — Pool keeps `maxHours = 24` literal (user-authorized exemption)

Pool's draggable range maximum stays a card-local `const maxHours = 24;`
literal (NOT `this._clampMaxHours(cfg.max_default_hours)` like the
family). Justification: pool's 24h cap is genuinely different from
the family's 12h default (pool filter runs can be much longer); the
dashboard template doesn't pass `max_default_hours` for pool.
`_allowedHalfHours(24)` is grid-aligned by construction (24 is a
multiple of `SNAP_STEP_HOURS = 0.5`) and bounded below
`MAX_HOURS_CEILING = 168`, so the snap list and the gauge stay
consistent.

Pool stays EXCLUDED from the
`test_card_derives_max_hours_via_clamp_helper` parametrize list
(`tests/test_dashboard_rendering.py` lines 7865-7873) — that exclusion
is intentional and documented.

**Pinning**: structural test asserts the source contains
`const maxHours = 24;` (with the explanatory comment immediately
above).

### AC-6 — Defensive `_safeNumber` adoption

The three raw `Number(sensor?.state || N)` reads for `targetHours`,
`hoursRun`, and `defaultDuration` are replaced with
`this._safeNumber(sensor, default)`. This adopts the documented
hardened pattern (project-rules doc, "Safe numeric coercion (S8,
water-boiler)") and prevents NaN propagation into the SVG arc-path
math. Defaults are spelled out:

- `targetHours = this._safeNumber(sDurationLimit, 0)` — 0 collapses
  `displayTargetHours` to 0 in non-default mode if the constraint
  sensor is unknown, which safely disables drag.
- `hoursRun = this._safeNumber(sCurrentDailyRunDuration, 0)` — 0 hours
  run when the sensor is unknown.
- `defaultDuration = this._safeNumber(sDefaultOnDuration, 0)` — 0
  collapses `displayTargetHours` to 0 in default mode if the
  `default_on_duration` entity is unknown, safely disabling drag.

Pool defaults are tighter than water-boiler's (12 / 0 / 6) because
pool has no `current_duration` / override / finish-time fallbacks; a
"0 means we don't know" pattern is safer here than a fictitious
fallback. Documented inline.

> **Superseded by review-fix #01 (N5) and review-fix #02 (N1).** See
> `docs/stories/QS-237.story_review_fix_#01.md` — the
> `defaultDuration` default was bumped from `0` to `1` to match the
> Python-side default in `bistate_duration.py:99` (smoother HA-boot
> visual: handle briefly shows `1h` instead of `0h` while
> `default_on_duration` is still booting). The "0 means we don't
> know" safety argument is recovered by review-fix #02 N1, which
> introduces a separate `defaultDurationKnown` flag that gates the
> water-fill `progressRatio` on the raw sensor state — so the `1`
> fallback is now COSMETIC ONLY (handle positioning), and the
> water-fill stays at 0 during the boot window. See
> `docs/stories/QS-237.story_review_fix_#02.md` item N1.
