## Findings to fix

### [must-fix] M1 — infinite loop on non-finite `max_default_hours`
- File: `custom_components/quiet_solar/ui/resources/shared/qs-ring-duration-base.js:115-120` (`_allowedHalfHours`)
- Severity: must-fix
- Source: qs-review-edge-case-hunter
- Scope: NEW regression introduced by round-2 S8 (configurable drag range)
- Description: `cap = Number(maxHours) > 0 ? Number(maxHours) : 12` evaluates to
  `Infinity` for a non-finite config (`max_default_hours: Infinity` or
  `1e999`, since `Number("Infinity") > 0 === true` and
  `Number("1e999") === Infinity`). The loop `for (let i = 0; i <= cap;
  i += 0.5)` then never terminates and hangs the browser tab. It fires
  synchronously inside `_render()` whenever `canDragHandle` is true (default
  mode, enabled, target > 0) — so the freeze triggers on the next render, no
  drag required. Reachable at all four duration-card call sites:
  `qs-on-off-duration-card.js:323`, `qs-radiator-card.js:509`,
  `qs-climate-card.js:1451`, `qs-water-boiler-card.js:1318`. The old code
  hard-looped `i <= 12` and was bounded.
- Proposed fix: Guard with `Number.isFinite` AND clamp to a sane upper bound
  (this also resolves N1):
  `const n = Number(maxHours); const cap = Number.isFinite(n) && n > 0 ? Math.min(n, 168) : 12;`
  Add a unit/structural test passing `Infinity` / `1e999` / `0` / negative /
  non-numeric `maxHours` and asserting `_allowedHalfHours` returns a bounded
  array (and terminates).

### [should-fix] S1 — pool drag-release reverses the service-call order
- Files: `custom_components/quiet_solar/ui/resources/qs-pool-card.js`
  (drag-commit / `onCommit`) ↔
  `custom_components/quiet_solar/ui/resources/shared/qs-card-base.js`
  (`_wireTargetHandle`, ~line 7661)
- Severity: should-fix
- Source: qs-review-blind-hunter
- Scope: NEW behavioral regression introduced by the migration
- Description: The old pool `onUp` handler did
  `_select(pool_mode, 'bistate_mode_default')` FIRST, then
  `_setNumber(default_on_duration, dragValue)`. The shared `_wireTargetHandle`
  inverts this — it calls `_setNumber(entityId, dragValue)` first and runs
  the mode-select afterward via pool's `onCommit`. If writing the duration
  while NOT yet in default mode is rejected/clamped backend-side, the dragged
  value won't stick.
- Proposed fix: Restore the original ordering for the pool (mode-select →
  duration write). Either (a) have pool's `onCommit` run before the number
  write, or (b) move the mode-select into a pre-commit hook on
  `_wireTargetHandle`. Verify against the pool's backend service contract
  that the duration write is valid only after default mode is active.

### [should-fix] S2 — `_wireTargetHandle` doesn't await async `onCommit`
- File: `custom_components/quiet_solar/ui/resources/shared/qs-card-base.js:7661`
- Severity: should-fix
- Source: qs-review-blind-hunter
- Scope: NEW regression introduced by the migration
- Description: `if (onCommit) onCommit(dragValue);` is not awaited, but pool
  passes an `async` `onCommit`. The mode-set `_select` now fires
  fire-and-forget (old code awaited it). Combined with the S1 order swap, the
  mode change may land after the local-target timeout logic and races the
  next render.
- Proposed fix: `await onCommit(dragValue)` (make the enclosing handler
  async), or fold the select into the awaited body so the old ordering +
  await guarantee is preserved. Resolve together with S1. Add/extend a test
  asserting the pool's drag-commit awaits the mode-select and preserves
  ordering.

### [nice-to-have] N1 — clamp very large finite `maxHours`
- File: `custom_components/quiet_solar/ui/resources/shared/qs-ring-duration-base.js:115-120`
- Severity: nice-to-have
- Source: qs-review-edge-case-hunter
- Description: A very large finite `maxHours` (e.g. `max_default_hours:
  100000`) builds a ~200,001-element array on every `_render()` and makes the
  snap `reduce` in `_wireTargetHandle.onMove` O(n) per pointermove. Not a
  hang, but wasteful.
- Proposed fix: Clamp `cap` to a sane upper bound (e.g. `Math.min(cap, 168)`).
  This is folded into the M1 fix — implement once.

### [nice-to-have] N2 — orphan `.qstmp` files leak on hard kill
- File: `custom_components/quiet_solar/ui/dashboard.py:444-471` (`_async_atomic_write_bytes`)
- Severity: nice-to-have
- Source: qs-review-edge-case-hunter
- Description: The N3 (round-2) orphan cleanup only fires on a caught
  `OSError` within the same process. With the new unique temp name
  (`{dst}.{pid}.{counter}.qstmp`), a hard kill (SIGKILL/OOM/power loss)
  between `f_out.write` and `os.replace` leaves a uniquely-named orphan that
  is never reclaimed and accumulates across crashes. The prior shared-name
  scheme self-healed (next run reused the same name). Low impact (orphans are
  never registered/served), but a slow disk leak.
- Proposed fix: Add a one-time glob sweep of `{namespace}/*.qstmp` at the top
  of `async_update_resources` to reclaim orphans.

### [nice-to-have] N3 — radiator flame `baseY` gate should use `Number.isFinite`
- File: `custom_components/quiet_solar/ui/resources/qs-radiator-card.js:222`
- Severity: nice-to-have
- Source: qs-review-edge-case-hunter
- Description: The round-2 S11 fix hardened the engine's regen gate to
  `Number.isFinite(baseY)`, but the card still sets
  `this._flameBaseY = Number.isNaN(flameBaseY) ? null : flameBaseY`, which
  would let `Infinity` through to the initial-paint path
  (`getInitialPaths(...)` → `baseY.toFixed(2)` → `"Infinity"` in the SVG `d`).
  Currently unreachable (all `flameBaseY` inputs are finite), so latent only.
- Proposed fix: Use `Number.isFinite(flameBaseY)` at line 222 for consistency
  with the S11 reasoning.

### [nice-to-have] N5 — dead flame mirror state in radiator card

### [nice-to-have] N6 — `_wireResetButton` doesn't wire keyboard activation
- File: `custom_components/quiet_solar/ui/resources/shared/qs-ring-duration-base.js` (`_wireResetButton`)
- Severity: nice-to-have
- Source: qs-review-blind-hunter
- Description: Unlike `_wirePowerButton`/`_wireGreenButton`/`_wireTimePicker`/
  `_wireOverrideButton`, the reset wire-helper does NOT call
  `_registerKeyActivation`. Matches prior per-card behavior (not a
