## Findings to fix

### [should-fix] S1 — Self-lockout via drag-to-zero
- File: `custom_components/quiet_solar/ui/resources/qs-pool-card.js:333`
- Severity: should-fix
- Source: qs-review-edge-case-hunter
- Description: With AC-3 the gate became
  `canDragHandle = isEnabled && isDefaultMode && displayTargetHours > 0`.
  The new snap list `_allowedHalfHours(24)` includes `0` as a valid
  commit value, so a user in `bistate_mode_default` can drag the handle
  to the bottom and release; `onUp` commits
  `_setNumber(default_on_duration, 0)`; after the 5s `_localTargetPct`
  hold expires, the next render computes `displayTargetHours = 0`,
  flips `canDragHandle` to `false`, and the handle is removed from the
  SVG. The user is locked out of drag-recovery and must use the HA
  `default_on_duration` number entity directly. Pre-QS-237 the gate was
  `hasValidTarget = isEnabled` (independent of target value), so this
  is a new regression introduced by the PR.
- Proposed fix:
  1. In `qs-pool-card.js`, change
     `const canDragHandle = isEnabled && isDefaultMode && displayTargetHours > 0;`
     to
     `const canDragHandle = isEnabled && isDefaultMode;`.
     Add a one-line comment explaining why we deliberately drop the
     `displayTargetHours > 0` term: in default mode the drag commits
     `default_on_duration` (a user-editable number), so we must keep
     drag-recovery reachable even when the current value is 0.
  2. In `tests/test_dashboard_rendering.py`, add a regression test
     `test_pool_card_drag_remains_reachable_when_default_duration_is_zero`
     that asserts the source contains
     `canDragHandle = isEnabled && isDefaultMode` and does NOT contain
     `displayTargetHours > 0` in the `canDragHandle` expression. Use a
     whitespace-tolerant regex (see N3).
  3. Update the existing AC-3 test
     `test_pool_card_drag_gated_on_default_mode` if it pinned the old
     `displayTargetHours > 0` term — replace that pin with the new
     two-term gate. (The story's AC-3 anchor in the source file must
     also be reconciled — see below in S2/S3 for the matrix.)

### [should-fix] S2 — AC-5 has no positive structural test for `const maxHours = 24;`
- File: `tests/test_dashboard_rendering.py` (missing test)
- Severity: should-fix
- Source: qs-review-acceptance-auditor
- Description: Story AC-5 (lines 183-185 of
  `docs/stories/QS-237.story.md`) prescribes:
  "structural test asserts the source contains `const maxHours = 24;`
  (with the explanatory comment immediately above)." The PR implements
  the literal at `qs-pool-card.js:231-238` but no test pins it. The
  negative-only shield (pool's exclusion from
  `test_card_derives_max_hours_via_clamp_helper` parametrize at
  `tests/test_dashboard_rendering.py:7865-7873`) only fires if someone
  adds pool back to the parametrize list, not if someone swaps `24`
  for `this._clampMaxHours(...)` — exactly the QS-235 regression we
  were guarding against.
- Proposed fix: add a new structural test in
  `tests/test_dashboard_rendering.py`, alongside the other AC-5
  guard:
  ```python
  def test_pool_card_pins_max_hours_literal_with_comment():
      """AC-5: pool keeps `const maxHours = 24;` with the explanatory
      comment immediately above (and does NOT use _clampMaxHours)."""
      pool = (POOL_CARD_PATH).read_text(encoding="utf-8")
      stripped = _strip_js_comments(pool)
      assert re.search(r"const\s+maxHours\s*=\s*24\s*;", stripped), (
          "AC-5: pool must keep `const maxHours = 24;` literal."
      )
      # The comment must remain in the *unstripped* source.
      assert "Pool deliberately keeps a 24h literal" in pool, (
          "AC-5: explanatory comment above `const maxHours = 24;` "
          "must remain (anchor for future maintainers)."
      )
  ```
  Use the exact wording from the comment already present in
  `qs-pool-card.js:231-238` for the second assertion (verify the
  current wording before writing the test; adjust the substring to a
  unique anchor inside the comment).

### [should-fix] S3 — AC-6 has no dedicated test for `_safeNumber` adoption
- File: `tests/test_dashboard_rendering.py` (missing test)
- Severity: should-fix
- Source: qs-review-acceptance-auditor
- Description: Story AC-6 (lines 187-208) prescribes the three
  `_safeNumber(sensor, default)` reads (`targetHours`, `hoursRun`,
  `defaultDuration`). PR implements all three at
  `qs-pool-card.js:250-252`. No test pins them; a future revert to raw
  `Number(sensor?.state || N)` would not be caught by any of the five
  new tests.
- Proposed fix: add a new structural test in
  `tests/test_dashboard_rendering.py`:
  ```python
  def test_pool_card_uses_safenumber_for_sensor_reads():
      """AC-6: pool reads sensor states via _safeNumber(..., default)
      for targetHours, hoursRun, and defaultDuration."""
      pool = _strip_js_comments(
          (POOL_CARD_PATH).read_text(encoding="utf-8")
      )
      for var_name in ("targetHours", "hoursRun", "defaultDuration"):
          assert re.search(
              rf"const\s+{var_name}\s*=\s*this\._safeNumber\(",
              pool,
          ), (
              f"AC-6: {var_name} must be read via this._safeNumber(...) "
              "(family pattern guards against unknown/unavailable states)."
          )
  ```
  Note: this aligns with N5 below — if we change `defaultDuration`'s
  default from `0` to `1` (family-consistent), this test still passes
  (it only asserts the `_safeNumber(` prefix, not the default value).
  S3 and N5 are therefore composable.

### [nice-to-have] N1 — poolMode `||` fallback semantics
- File: `custom_components/quiet_solar/ui/resources/qs-pool-card.js:223-224`
- Severity: nice-to-have
- Source: qs-review-blind-hunter + qs-review-edge-case-hunter (merged)
- Description: `poolMode = selPoolMode?.state || 'bistate_mode_default'`
  only triggers the fallback when `state` is `undefined`/`''`/`null`;
  literal `'unknown'`/`'unavailable'` strings (typical HA boot states)
  are truthy and flow down the non-default branch, disabling drag.
  This matches the family pattern (radiator/water-boiler) and is
  presumably intentional — but a one-line comment makes the intent
  explicit for future maintainers.
- Proposed fix: add a comment immediately above the `poolMode`
  assignment in `qs-pool-card.js`:
  ```js
  // Fallback to default mode only when the select entity is missing
  // (state undefined/null/''); transient 'unknown'/'unavailable'
  // strings flow to the non-default branch and disable drag, which is
  // the family-consistent behaviour (radiator/water-boiler match).
  const poolMode = selPoolMode?.state || 'bistate_mode_default';
  ```

### [nice-to-have] N3 — Substring assertions are too loose
- File: `tests/test_dashboard_rendering.py:7975-7986, 8005-8025, 8040-8044, 8058-8063`
- Severity: nice-to-have
- Source: qs-review-blind-hunter + qs-review-coderabbit (merged)
- Description: Several new assertions pin exact source substrings
  (e.g. `"displayTargetHours = isDefaultMode ? defaultDuration : targetHours"`,
  `"${colors.primary};">${this._fmt(displayTargetHours)}h"`,
  `"canDragHandle = isEnabled && isDefaultMode && displayTargetHours > 0"`,
  `"this._allowedHalfHours(maxHours)"`). These are fragile to harmless
  formatting/reflow in `qs-pool-card.js` and produce noisy CI failures
  without behavioural regressions.
- Proposed fix: refactor each of these substring assertions to use
  `re.search(...)` with `\s*` between tokens. Examples:
  ```python
  assert re.search(
      r"displayTargetHours\s*=\s*isDefaultMode\s*\?\s*"
      r"defaultDuration\s*:\s*targetHours",
      pool,
  ), "AC-1: displayTargetHours computation must be pinned."

  assert re.search(
      r"canDragHandle\s*=\s*isEnabled\s*&&\s*isDefaultMode\s*;",
      pool,
  ), "AC-3 (post-S1): drag gate is two-term in default mode."

  assert re.search(
      r"this\._allowedHalfHours\(\s*maxHours\s*\)",
      pool,
  ), "AC-4: snap list must use _allowedHalfHours(maxHours)."
  ```
  Reconcile this refactor with S1 (the gate becomes a two-term
  expression) and S2/S3 (the new tests should already be regex-based).
  Do NOT loosen the negative assertions (e.g. the
  `"displayTargetHours > 0"` should-NOT-appear assertion for S1 must
  remain strict — that's the whole point).

### [nice-to-have] N5 — Handle flicker during HA boot in default mode
- File: `custom_components/quiet_solar/ui/resources/qs-pool-card.js:252`
- Severity: nice-to-have
- Source: qs-review-edge-case-hunter
- Description:
  `defaultDuration = this._safeNumber(sDefaultOnDuration, 0)` returns
  `0` whenever the `default_on_duration` number entity has not yet
  registered (e.g. HA boot, integration reload) or its state is
  transiently `unknown`/`unavailable`. In default mode this makes
  `displayTargetHours = 0` → `canDragHandle = false` → handle hidden
  until the next HA push delivers the Python-side default (`1.0` per
  `bistate_duration.py:99`). Water-boiler and on_off_duration default
  to `defaultDuration = 6` to mask this gap. Family-consistent
  fallback for pool would be the Python default `1.0`.
  After S1, the handle would no longer be hidden (drag gate no longer
  depends on `displayTargetHours > 0`), so the flicker becomes a
  visual jump from `1h` to the real value instead of a handle pop-in.
  Still cleaner than `0h`.
- Proposed fix: change the default in `qs-pool-card.js:252`:
  ```js
  const defaultDuration = this._safeNumber(sDefaultOnDuration, 1);
  ```
  (matches the `bistate_duration.py:99` Python-side default of `1.0`).
  This composes with S3's regex test (which only checks the
  `_safeNumber(` prefix).

### [nice-to-have] N6 — Half-hour snap rounding mismatch in big text
- File: `custom_components/quiet_solar/ui/resources/qs-pool-card.js:422`
- Severity: nice-to-have
- Source: qs-review-edge-case-hunter
- Description: The new `_allowedHalfHours(24)` snap list emits
  half-hour values; on commit `displayTargetHours = 8.5` is rendered
  by `_fmt(displayTargetHours)` (default `round=true`) as `"9h"`, even
  though the stored entity value is `8.5`. The `dragMove` live-update
  span shows `_fmt(snapValue, false) = "8.5"` (un-rounded), so on
  release the target snaps visually from `"8.5"` → `"9"`. The drag
  preview and the final committed display disagree on rendering.
  Family-wide (radiator/water-boiler/on_off_duration share this
  rounding); newly observable on pool now that pool joined the
  half-hour grid.
- Proposed fix: change the big-text render in `qs-pool-card.js:422`
  from `${this._fmt(displayTargetHours)}h` to
  `${this._fmt(displayTargetHours, false)}h` (un-rounded), matching
  the drag-preview span. The same change should be considered for the
  rest of the duration-card family in a follow-up issue, but this
  plan only touches pool to keep PR scope tight. Add a one-line
  comment explaining why pool diverges from the family `round=true`
  default (drag preview vs. committed display consistency on the
  half-hour grid).
  Reconcile the existing AC-2 test
  `test_pool_card_target_value_uses_palette_color_not_theme_variable`
  if it pins `_fmt(displayTargetHours)` — update the pin to
  `_fmt(displayTargetHours, false)` (or use a whitespace-tolerant
  regex per N3 that allows either form, then keep the strict pin in
  a separate dedicated test).
