## Story

As a developer,
I want the quality gate to be smarter about what it runs and to show progress during execution,
so that dev-only changes don't waste time on full linting/coverage and long test runs aren't silent.

## Acceptance Criteria

1. Given only dev-infrastructure files are modified (tests/, _qsprocess*/, docs/, scripts/, *.md, .claude/, .cursor/, .opencode/),
   When the quality gate runs,
   Then it skips ruff, mypy, translations, and full pytest+coverage, running only the changed test files.

2. Given production source files (custom_components/) are modified,
   When the quality gate runs,
   Then it runs the full suite (ruff format, ruff lint, mypy, translations, pytest+coverage) as before.

3. Given `--full` flag is passed,
   When only dev-infrastructure files are modified,
   Then the full suite runs regardless of scope detection.

4. Given the full pytest suite is running (~4800 tests),
   When tests execute,
   Then progress is reported to stderr every ~10% (passed/failed/error counts).

5. Given any gate step (ruff format, ruff lint, mypy, translations) is running,
   When it starts and completes,
   Then a brief status line is emitted to stderr (running... then PASS/FAIL).

6. Given the setup-task agent completes,
   When it presents next steps to the user,
   Then it prints text directions to open a workspace on the worktree and start a new session — it does NOT attempt to Task-spawn or auto-hand-off.

## Dev Notes

### Affected Files

- `scripts/qs/quality_gate.py` — scope detection, progress reporting, `--full` flag
- `.opencode/agents/qs-setup-task.md` — remove auto-handoff, add text directions

### Architecture Constraints

- Progress output goes to stderr (not stdout) to preserve `--json` mode
- Scope detection compares against `origin/main` using git diff
- Cache is only written for full-scope passes (dev-only passes are not cached)
- No changes to `_qsprocess/` or `.claude/` (hands-off areas)

### Key Design Decisions

- Test count is collected first via `--collect-only` to calculate 10% milestones
- Progress uses threading to read stderr from pytest subprocess concurrently
- Dev-only scope includes top-level config files (pyproject.toml, .gitignore, etc.)
