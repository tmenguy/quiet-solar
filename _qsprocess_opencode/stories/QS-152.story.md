# Story bug-Github-#152: Fix home controlled/non-controlled consumption computation

issue: 152
branch: "QS_152"
Status: ready-for-dev

## Story
As a Quiet Solar user,
I want the home controlled consumption to exclude power from user-overridden loads (manually started/stopped bistate loads and manually forced car charges),
so that the non-controlled consumption accurately reflects only truly uncontrolled energy usage.

## Acceptance Criteria

1. Given a bistate load with an active user override (detected via `get_override_state() != "NO OVERRIDE"`, including "ASKED FOR RESET" which is still physically overridden)
   When the home computes controlled consumption in `home_non_controlled_consumption_sensor_state_getter`
   Then the load's power is NOT included in `controlled_consumption` (it stays in non-controlled)

2. Given a car/charger with a user-initiated force charge (set via the UI `user_force_charge_now()` button)
   When the home computes controlled consumption
   Then the charger's power for that car is NOT included in `controlled_consumption`
   And the charger's `get_override_state()` returns `"Override: {car_name}"` (via the constraint's `originator: "user_override"`)

3. Given the historical consumption reset computation (`QSHomeSolarAndConsumptionHistoryAndForecast`)
   When a load's `load_override_state` sensor history shows it was user-overridden during part of the period
   Then that load's historical power during the overridden intervals is excluded from controlled consumption history
   And when no override sensor history exists (cold-start after sensor creation), the load is assumed "not overridden" (conservative default)

4. Given a `QSDynamicGroup` containing loads with mixed override states
   When `is_user_overridden()` is called on the group
   Then it returns `True` if ALL children are overridden, `False` if NONE are overridden, and `None` if it's a mix
    And `get_device_power_latest_possible_valid_value` with `ignore_auto_and_user_overridden_load=True` excludes fully-overridden groups, and for mixed-override groups falls back to per-child aggregation summing only non-overridden children's power
   And when `ignore_auto_and_user_overridden_load=False`, the method behaves identically to today (no override filtering)

    ```python
    def _apply_override_mask(
        self, load_sensor: QSSolarHistoryVals, override_states: list[LazyState]
    ) -> None:
        """Zero out power values in load_sensor for intervals where override was active.

        Walks override_states chronologically. For each state change:
        - State starting with "Override:" → mark as overridden
        - Otherwise ("NO OVERRIDE") → mark as not overridden
        - "ASKED FOR RESET OVERRIDE" → still physically overridden, mark as overridden

        For each overridden interval, sets load_sensor.values[0][idx] = 0
        (row 0 = power values, row 1 = day tracking — preserved).
        """
        if load_sensor.values is None:
            return

        is_overridden = False
        prev_time = None

        for state in override_states:
            if state.state is None or state.state in [STATE_UNKNOWN, STATE_UNAVAILABLE]:
                continue

            current_time = state.last_changed
            new_is_overridden = state.state.startswith("Override:")

            # If previous state was overridden, zero out all intervals from prev_time to current_time
            if is_overridden and prev_time is not None:
                start_idx, _ = load_sensor.get_index_from_time(prev_time)
                end_idx, _ = load_sensor.get_index_from_time(current_time)
                self._zero_range(load_sensor, start_idx, end_idx)

            is_overridden = new_is_overridden
            prev_time = current_time

        # Handle trailing override (overridden until end of history)
        if is_overridden and prev_time is not None:
            end_idx, _ = load_sensor.get_index_from_time(
                datetime.now(tz=prev_time.tzinfo)
            )
            self._zero_range(load_sensor, load_sensor.get_index_from_time(prev_time)[0], end_idx)
    ```

    def _zero_range(load_sensor: QSSolarHistoryVals, start_idx: int, end_idx: int) -> None:

        if start_idx <= end_idx:

## Dev Notes

### Architecture constraints
- `home_model/` NEVER imports `homeassistant.*` — `is_user_overridden()` base goes on `AbstractDevice` in `home_model/load.py`, override on `AbstractLoad` in same file (domain layer only)
- The key insight: adding `load_info={"originator": "user_override"}` to the charger force constraint makes the existing `get_override_state()` work for chargers with zero new code in the domain layer
- `is_user_overridden()` return type is `bool | None` — `True` = all overridden, `False` = none overridden, `None` = mixed (used by dynamic groups). `AbstractDevice` base returns `False`, `AbstractLoad` returns `True`/`False`, `QSDynamicGroup` returns `True`/`False`/`None`
- All override filtering in `get_device_power_latest_possible_valid_value` is gated on the `ignore_auto_and_user_overridden_load` parameter — when `False`, behavior is **identical to today**. This eliminates any risk of side effects on non-consumption callers.
- `QSDynamicGroup` inherits from `HADeviceMixin` and `AbstractDevice` (NOT `AbstractLoad`), so `is_user_overridden()` is a separate override of the `AbstractDevice` base, not the `AbstractLoad` override
