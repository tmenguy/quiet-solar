# Story QS-162: Remove all BMad references from OpenCode development workflow

issue: 162
branch: "QS_162"
Status: ready-for-dev

## Story

As a developer using the OpenCode workflow,
I want all BMad references removed from OpenCode configuration files and templates,
so that the OpenCode pipeline is self-contained and does not depend on `_bmad-output/` or `_qsprocess/` paths.

## Acceptance Criteria

### AC1: Templates are self-contained

Given any of the 13 agent templates in `_qsprocess_opencode/agent_templates/`
When the template is rendered into a per-task agent
Then the rendered agent contains zero references to `_bmad-output`, `_qsprocess/` (excluding `_qsprocess_opencode/`), or BMad skill files

**Note:** 7 templates have refs to update; 6 templates are already clean
(qs-plan-critic, qs-plan-concrete-planner, qs-plan-scope-guardian,
qs-review-blind-hunter, qs-review-edge-case-hunter, qs-review-acceptance-auditor).
All 13 must be verified by grep.

### AC2: Static agents are self-contained

Given the 2 static agents in `.opencode/agents/` (`qs-setup-task.md`, `qs-release.md`)
When an agent is loaded by OpenCode
Then it contains zero references to `_qsprocess/` (excluding `_qsprocess_opencode/`) or `_bmad-output` as **source paths**

**Note:** Protection guards ("Do NOT modify `_qsprocess/`") are kept — the
original tree is still hands-off. Only refs that **read from** those paths
as data sources are removed.

### AC3: AGENTS.md is OpenCode-only

Given `AGENTS.md` is loaded as the OpenCode entry point
When any reference check is run
Then it contains zero references to `_bmad-output` or `_qsprocess/` (excluding `_qsprocess_opencode/`)
And all required content (project rules, code style rules) is redirected to `_qsprocess_opencode/` paths

### AC4: Product documents are migrated

Given the product documents in `_bmad-output/` (project-context.md, architecture.md, product-brief.md)
When they are needed by OpenCode agents
Then copies exist under `_qsprocess_opencode/` with BMad references removed
And any agent or template referencing them uses the `_qsprocess_opencode/` path

### AC5: Implementation artifacts are migrated

Given the implementation artifacts in `_bmad-output/implementation-artifacts/` (~62 files)
When migration is complete
Then each artifact has a corresponding file in `_qsprocess_opencode/stories/` following the QS naming convention
And files that already have a QS-N.story.md counterpart are skipped (not duplicated)
And a migration script exists at `scripts/qs_opencode/migrate_artifacts.py` for reproducibility
And naming collisions are handled with a `-duplicate` suffix and logged as warnings
And the script prints a summary: `source_count = migrated + skipped` (count assertion)

**Note:** Migrated artifact **content** is NOT scrubbed of prohibited refs.
These are historical records and get the same exemption as existing stories (AC7).

### AC6: OpenCode documentation is updated

Given the documentation files in `_qsprocess_opencode/` (README.md, PLAN.md, SMOKE_TEST.md, SESSION_LOG.md)
When any reference check is run
Then they contain zero prohibited references (applying the classification scheme in Task 8)

### AC7: Existing and migrated stories are not broken

Given existing story files in `_qsprocess_opencode/stories/` (QS-137 through QS-159)
And newly migrated artifact files in `_qsprocess_opencode/stories/`
When they reference `_qsprocess/` or `_bmad-output/` paths
Then those references are historical context and are left unchanged
And these files are excluded from the verification grep in Task 10

### AC8: CLAUDE.md is updated (user-manual step)

Given `CLAUDE.md` is hands-off for the implement agent
When the story implementation is complete
Then the exact diff for CLAUDE.md is printed for the user to apply manually
And the diff adds `_qsprocess_opencode/` to the hands-off list

## Dev Notes

### Architecture constraints

- **Two-tree coexistence**: `_bmad-output/` and `_qsprocess/` remain untouched — they serve Claude Code/Cursor. OpenCode gets its own copies.
- **Copy not move**: All migrations are copies. Original files are never deleted or modified.
- **No content loss**: When removing a `Read _qsprocess/skills/*.md` directive from a template, the template already contains the equivalent protocol inline — nothing is lost.
- **Hands-off enforcement**: `CLAUDE.md` is hands-off per project-rules.md. The implement agent prints a diff but does NOT edit it.
- **Protection guards stay**: Lines like "Do NOT modify `_qsprocess/`" in agent files are protection guards and remain even though the agent no longer reads from `_qsprocess/`.

### Task ordering

Tasks 2–4 must complete before Task 1 (templates need the new paths to exist).
Task 5 (migration script) is independent of Tasks 1–4.
Tasks 6–8 can run after Task 1.
Task 9 runs last (needs all changes visible for diff generation).
Task 10 runs after everything else.

Recommended order: 2 → 3 → 4 → 1 → 5 → 6 → 7 → 8 → 9 → 10
